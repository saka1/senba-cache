## 0. TL;DR

c17s = c16s から **同期通知の所在を tag → entry に逃がす** variant。`Entry` に
`AtomicU32 version` を追加し、Path A は tag を一切触らずに entry version の
偶奇 flip だけで reader を seqlock させる。tag の VERSION bit が不要になり、
HASH 領域が 1 bit 拡張 (8→9 bit、c11s と同等)。reader の SIMD scan からは
EMPTY-lane 検出を削除可能 (Path A は tag を EMPTY 化しないため)。

```rust
#[repr(C)]
struct Entry<K, V> {
    version: AtomicU32,   // 偶数 = stable, 奇数 = in-flight
    key:     K,
    value:   V,
}
```

合格 gate は (a) bench_concurrent **adv-hot read-heavy 16T** で c16s 比 +5% Mops、
(b) Mops AB skew=1.0 T=4 で **regression なし** (≤ −2%、c17s 主目的は read-only
adv-hot の回収であって T=4 mixed の優位ではない)、の同時成立。両方未達なら reject。

## 1. 動機 — c14s/c16s に残った 2 つの構造的退行

c14s/c16s で tag は **3 役を兼任**している:

| 役 | 用途 | 該当 bit |
|---|---|---|
| 1. lookup index | reader SIMD masked compare | LIVE + HASH (8 bit) |
| 2. arena pointer | entries[id] への引き | ID (6 bit) |
| 3. **同期通知** | reader seqlock fire | VERSION + EMPTY transient |

役 3 のために以下が固定コストとして残る:

- **(i) `find_get` の EMPTY-lane SIMD overhead** — Path A の EMPTY transient を検出するため、
  reader は scan 中に `cmpeq(masked, zero)` + movemask + range mask を毎 chunk 1 回追加実行する。
  Path A 並行が稀でも overhead は固定。read-only adv-hot 16T で c14s/c13s = 0.78 (`c14s-sweep` §4.2)。
- **(ii) adv-hot HR drop** — reader が Path A の EMPTY transient を観測したまま racing を
  消化しきれず None を返す race。bounded retry (MAX_READER_RETRY=4) は対症療法、構造的問題は残る。
- **(iii) HASH bit 数 −1** — VERSION bit を tag から切り出した分、c11s 9 bit → c8/c14s/c16s 8 bit に
  collision rate が 2 倍化。

これらは tag 兼任構造の中では同時に解けない (VERSION bit を消すと Path A の seqlock fire が
出来ず、EMPTY-lane 検出を削ると Path A false-miss が増える)。**役 3 を tag から外す**のが α の核。

## 2. アーキテクチャ — 役 3 を entry version に移管

### 2.1 tag layout (16 bit、VERSION 削除)

```text
bit 15  14         8 7              0
┌─────┬───────────┬─────────────────┐
│LIVE │  ID (6b)  │   HASH (9b)     │   ← c11s と同 layout、c14s/c16s から +1 bit
└─────┴───────────┴─────────────────┘
SCAN_MASK = LIVE | HASH (= 0xFFFF & !ID_MASK)
```

VERSION bit (旧 0x4000) を削除して HASH に編入。reader needle 比較は LIVE + HASH の
9 bit 一致で false-positive 率が 1/512 (c14s/c16s の 1/256 比 −50%)。

### 2.2 Entry に version 追加

```rust
#[repr(C)]
struct Entry<K, V> {
    version: AtomicU32,   // 偶数 = stable, 奇数 = in-flight
    key:     K,
    value:   V,
}
```

`Entry<u64, u64>` の sizeof は c16s 16B → c17s 24B (round up to 32B align、ID_SHIFT=5)。
per-shard cap=64 で entries 配列が 1024B → 2048B に膨らむ。**Slot32 ブラケット相当の cost**
(library surface の `SlotSize` は触らない、c17s は research artifact)。

ID_SHIFT は `id_shift_from_entry_size(sizeof)` で const-eval 自動算出、5 になる。
ID_MASK = `0x3F << 5 = 0x07E0`、HASH_MASK = `0xFFFF & !LIVE & !ID_MASK = 0x781F`、
SCAN_MASK = `LIVE | HASH_MASK = 0xF81F`。HASH bit 数 = 9 (c11s と同等)。

### 2.3 Path A — tag 不変、entry version flip で reader 通知

```text
1. find_lockfree_for_path_a(key, needle) → (pos, tag, id)
   ※ tag は CAS expected として使うのではなく、id 抽出のためだけに使う
2. entries[id].version.fetch_add(1, AcqRel)         ← odd 化、in-flight 通告
3. ptr::read(entries[id])  + ptr::write new K, V
4. entries[id].version.fetch_add(1, Release)        ← even 化、stable 復帰
5. visited.fetch_or(1<<pos, Relaxed)
```

**tag 操作はゼロ**。Path A は entries 配列内で完結し、VERSION flip / EMPTY transient
は発生しない。CAS が消えて fetch_add 2 回に置換される (atomic op 数は 1 → 2 だが、同一
cache line 内で連続するため L1 hot)。

**Path A 同時実行の排他**: tag CAS が無いため、複数 writer が同 key に Path A を
かけた場合 entry version が衝突しうる。これを以下で解決:

```rust
// 2'. CAS で odd 化 (= 別 writer が既に in-flight でないことを確認)
let v = entries[id].version.load(Acquire);
if v & 1 != 0 { return Err(value); }  // 別 writer 進行中 → Path B/C escalate
match entries[id].version.compare_exchange(v, v + 1, AcqRel, Acquire) {
    Ok(_) => {}
    Err(_) => return Err(value),       // race lost → escalate
}
// 3. ptr::read + ptr::write (= 排他確保済み)
// 4. version.store(v + 2, Release)    ← unconditional、CAS 成功した側のみが進行
```

CAS-based version claim で Path A の atomic op 数は CAS 1 + store 1 = 2 (c14s/c16s と同等)。
失敗した writer は Path B/C に escalate (= writer Mutex 取得後に再 find して
`writer_update_in_place` 経路、CAS 不要で確実に書ける)。

### 2.4 Reader — 2-tier seqlock

```text
[find_get] (EMPTY-lane SIMD overhead 削除済み)
  for chunk in tags.chunks(LANE):
    SIMD compare: chunk & SCAN_MASK == needle?  → cand mask
    for each cand bit:
      try_candidate(pos)

[try_candidate]
  t1 = tags[pos].load(Acquire)              ← tier 2 用 snapshot
  if (t1 & SCAN_MASK) != needle: return Miss
  id = id_of(t1)
  v1 = entries[id].version.load(Acquire)    ← tier 1 (Path A 検出)
  if v1 & 1 != 0: return Racing
  buf = ManuallyDrop::new(ptr::read(entries[id]))
  v2 = entries[id].version.load(Acquire)
  if v1 != v2: return Racing                ← Path A cycle 観測
  t2 = tags[pos].load(Acquire)
  if t1 != t2 || (t2 & LIVE) == 0: return Racing  ← tier 2 (Path C shift 観測)
  if buf.key == *key:
      visited.fetch_or(1<<pos)
      return Found(buf.value.clone())
  Miss
```

**2-tier の責任分担**:

- **tier 1 (entry version)** — Path A (= 同 id への上書き) を捕える。adv-hot read-heavy で頻発、
  steady-state でも 1 op に 1 atomic load の cost で済む。
- **tier 2 (tag re-load)** — Path C (= shift で tag が別 id を指すように変わる) を捕える。
  Path C は eviction 時のみなので発火頻度は低い。Path A は tag を触らないのでこのチェックを
  通り抜ける。

Path C 中の `tags[i] = EMPTY → tags[i] = next_tag` 経路は据え置き。reader は tier 2 で
これを観測する (t1 != t2 で fire)。**find_get での EMPTY-lane SIMD 検出は削除**
(Path A の EMPTY transient が無くなったので不要)。

### 2.5 Path B/C — c16s と同型 + entry version 操作追加

Path B (warmup install): `entries[len].version = 0` で初期化、新 K/V を `ptr::write`、
tag を Release store。version は奇遇 cycle ではなく初期値 0 のまま (新 entry なので reader
が衝突する race 自体存在しない、tag が LIVE 化された瞬間に reader が拾える)。

Path C (evict + shift + install): evict 対象 `entries[evict_id]` を上書きする際に
**version を fetch_add 2 回**で囲う (Path A と同じ idiom)。reader の tier 1 で Path C 経路の
entry overwrite を捕える。tag shift 自体は c16s と同型 (EMPTY transient 経由)。

`writer_update_in_place` も同様に version 偶奇 flip で reader 通知 (CAS 不要、Mutex 配下)。

### 2.6 SHARDS=64 固定

c14s/c16s と同方針 (`feedback_c_variant_shard_count.md`)。`cap=4096 / SHARDS=64
→ per-shard cap=64`、6-bit ID 上限ぴったり。

### 2.7 Slot32 化と footprint

per-shard cap=64 / Entry sizeof=32B で entries 配列 = 2048B = 32 cache line。
c16s (1024B = 16 line) 比 **2×**。tags 列 (128B) と visited (8B / ShardHot 内) は不変、
ShardHot layout も c16s と同型 (`#[repr(C, align(64))] ShardHot { Mutex<WriterState>,
AtomicU64 visited, AtomicUsize len }`) で 64B 維持。

cap-fits 帯 (= cache 全体が L3 内に収まる帯) で entries 倍化が L3 transfer 量を増やす
リスクはある。adv-hot 16T (= read-heavy / 同 entry 集中) では entries[id] が L1 hot な
1 line に集中するので影響軽微の見込み。

## 3. コード変更スコープ

### 3.1 新規ファイル

`research/src/experimental/sieve_c17s.rs` — c16s からのコピー派生。

### 3.2 機械的置換

| c16s | c17s |
|---|---|
| `struct Entry<K, V> { key, value }` | `struct Entry<K, V> { version: AtomicU32, key, value }` |
| tag layout: LIVE + VERSION + ID + HASH | tag layout: LIVE + ID + HASH (HASH +1 bit) |
| Path A: tag CAS to EMPTY → ptr::r/w → tag CAS to flipped VERSION | Path A: version CAS to odd → ptr::r/w → version store to even |
| reader: t1, ptr::read, t2, t1==t2 check | reader: t1, v1, ptr::read, v2, t2, v1==v2 + t1==t2 check |
| `find_get`: EMPTY-lane cmpeq + range mask + racing flag | EMPTY-lane 検出を削除 |
| `find_lockfree_for_path_a`: 共有 SIMD スキャン (visited 触らない) | 同様、tag CAS 用 expected_tag は version snapshot に置換 |

### 3.3 ロジック変更 (機械的でない箇所)

1. **Entry の Drop 経路** — `Entry::version` は `AtomicU32` で `Drop` 不要、`Shard::drop`
   内で LIVE entry を `assume_init_drop` する経路は変わらず。
2. **Path B (warmup install)** — `entries[len]` を `MaybeUninit::write` で初期化する際、
   `Entry { version: AtomicU32::new(0), key, value }` で組み立てる。
3. **Path C (evict + install)** — `writer_evict_and_install` 内で evict 対象の
   `entries[evict_id]` 上書き前後に `version.fetch_add` 2 回。shift loop は c16s と同型
   (tag を EMPTY 経由で動かす)、shift される `entries[id]` 自体は触らないので version 不要。
4. **`writer_update_in_place`** — Mutex 配下の既存 key 更新。`tags[pos] = EMPTY → write
   entry → tags[pos] = new_tag` の経路を、`version fetch_add → write entry → version
   fetch_add → tag は不変 (LIVE のまま)` に置換。tag を触らない方向に統一。
5. **`Shard::live_count` / `live_ids`** (test only) — tag の LIVE bit だけ見る既存実装で OK。

### 3.4 変わらない部分

- ShardHot layout (`Mutex<WriterState>`, `AtomicU64 visited`, `AtomicUsize len`)
- SHARDS=64、6-bit ID schema
- AVX2 needle 比較 (SCAN_MASK 値だけが変わる)
- visited bit fetch_or の conditional load (c11s 由来)
- Path B/C escalate の trigger (Path A CAS 失敗時)
- `unsafe impl Send/Sync` の SAFETY 議論

## 5. 検証 step

### Step 1: 正しさ — oracle 一致

```bash
cargo test -p senba-research sieve_c17s
cargo test -p senba-research --test oracle c17s
cargo test --workspace
```

合格条件: oracle 4 構成すべて diff=0、`concurrent_invariants_under_zipf` PASS、
全テスト pass。c17s が SIEVE 等価でない場合 (Path A の version 操作が
sieve_orig と eviction stream で divergent) はここで止まる。

### Step 2: Mops AB (gate (a) — adv-hot read-heavy 16T)

```bash
./target/release/bench_concurrent --variant c16s,c17s --threads 16 --shards 64 \
    --cap 4096 --keys 100000 --skew 1.4 --ops 80000000 --warmup 1000000 --trials 5 \
    --op-mix read_heavy
```

c14s-sweep §4.2 の T1 シナリオに最も近い形 (adv-hot ≈ skew 1.4、read-heavy =
read 95% / write 5% 想定)。**Gate (a)**: `c17s_mops / c16s_mops >= 1.05` (5 trial
平均、CV<0.05)。

実装: `bench_concurrent` の `--op-mix read_heavy` 既存 arm を流用。read 95/write 5 か
類似比率を選ぶ (既存 op_mix の選択肢に合わせる)。

### Step 3: Regression check (gate (b) — skew=1.0 mixed T=4)

```bash
./target/release/bench_concurrent --variant c16s,c17s --threads 4 --shards 64 \
    --cap 4096 --keys 100000 --skew 1.0 --ops 80000000 --warmup 1000000 --trials 5
```

**Gate (b)**: `c17s_mops / c16s_mops >= 0.98` (= regression ≤ 2%)。c17s の主目的は
adv-hot read-heavy の回収であって T=4 mixed の優位ではない。c14s/c16s sweet spot で
大きな regression が出るなら設計再考。

### Step 4: thread sweep T={1,2,4,8,16}

```bash
for t in 1 2 4 8 16; do
  ./target/release/bench_concurrent --variant c16s,c17s --threads $t --shards 64 \
      --cap 4096 --keys 100000 --skew 1.0 --ops 80000000 --warmup 1000000 --trials 3
done
```

異常パターン (T=2 で大幅 regression / T=16 で逆転 / scaling 崩壊) を検出。

### 合否判定

| Gate (a) | Gate (b) | 判断 |
|---|---|---|
| ✓ | ✓ | **採用**。adv-hot read-heavy の構造的退行が解消、後続 design space 探索の起点 |
| ✓ | ✗ | adv-hot は伸びたが skew=1.0 で退行 → 2-tier seqlock cost が出ている、tier 2 を skip 可能か検討 |
| ✗ | ✓ | adv-hot で伸びない → 仮説が破れている、EMPTY-lane SIMD overhead 以外の主因がある |
| ✗ | ✗ | reject。`docs/reports/2026-05-11-c17s-results.md` に reject 理由を残す |

## 7. スコープ外 (今回はやらない)

- **VTune memory-access (gate (b) 相当)** — 今回は WSL2 で Mops AB だけ取得、Windows VTune は
  c16s 採用時に別 task 化 (`c16s-results` Step 4 と同じ pattern)。
- **adv-hot read-heavy で workload generator を本物の adversarial-hot に置き換え** — c14s-sweep
  §4.2 の元 T1 は `bench_single_shard` 経由。今回は `bench_concurrent` 既存 op_mix で代替。
  本格 reproduce は follow-up。
- **G2-α-2 (version を別配列 versions[] に逃がす版)** — α-1 の Slot32 化が footprint で問題に
  なった場合の代替案、今回は α-1 で先行検証。
- **G2-β SoA state** — α の long-term 上位互換、別 variant 扱い。
- **senba lib への取り込み** — c17s は experimental。lib 化は採用が確定し perf-gate /
  oracle が通った後。
- **Loom 等の formal concurrency 検証** — Path A の version CAS / 2-tier seqlock は新規構造
  なので将来的には要、今回 scope 外。

## 8. 不採択時の next move 候補

c17s reject 時:

1. **G2-α-2 (versions 別配列)** — α-1 の Slot32 化が L3 圧迫主因なら、Entry 不変で version
   だけ別配列に置く。reader は cache line 1 個追加 touch。
2. **EMPTY-lane SIMD 検出のみ条件付き削除** — `find_get` 内で writer Mutex held 中の shard
   だけ EMPTY-lane 検出を有効化、Path A 並行が無い shard ではスキップ。flag が 1 line 追加。
3. **G2-β (SoA state)** — `slots: Box<[AtomicU32]>` で version + lock + epoch を分離、
   Entry 不変。AVX-512 V5 / B1 SoA tag split との整合性が高い。
