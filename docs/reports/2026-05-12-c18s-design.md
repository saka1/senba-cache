# 2026-05-12 — c18s 設計: versions 別配列化 (G2-α-2)

- 種別: **設計ノート (実装前)**。c17s からの差分仕様 + AB protocol。
- 直接の動機: `2026-05-11-c17s-results.md` §7.1 の (b) gate fail (skew=1.0 gim T=4 で −3.7%
  退行) を、Slot32 化 (Entry 16B → 32B) 由来の footprint cost を構造除去することで解消する。
- 関連:
  - `2026-05-11-c17s-design.md` / `2026-05-11-c17s-results.md` — c18s が継承する layout と
    継承しない部分 (Entry / version 配置)

## 0. TL;DR

c18s = c17s から **`Entry::version` を別配列 `versions: [AtomicU32; 64]` に逃がす** variant
(G2-α-2)。Entry を Slot16 に戻し (`{ key, value }` の 16B、c16s と同型)、entries arena を
2KB → 1KB に半減する。reader / Path A の version load/CAS 先が `entries[id].version` から
`versions[id]` に変わるだけで、SIEVE 等価性 / tier 1 seqlock / EMPTY-lane SIMD 削除済 /
path_c_epoch coarse retry — 全部 c17s から継承する。

合わせて、**path_c_epoch を ShardHot から取り出して versions と同じ独立 cluster に
co-locate** する ([C] 配置)。c17s では path_c_epoch は ShardHot 内で writer-hot な
visited fetch_or / len store と同居しており、Path A 頻発帯では reader が読む path_c_epoch
line が visited 経由で頻繁に invalidate される構造。c18s では reader が触る同期通知系
(versions + path_c_epoch) を 1 block にまとめ、writer-hot cluster (Mutex/visited/len) と
分離する。

```rust
#[repr(C, align(64))]
struct ReaderState {
    path_c_epoch: AtomicU64,    // line 0, offset 0..8
    _pad0: [u8; 56],            // line 0, offset 8..64 (writer 干渉ゼロ保証)
    versions: [AtomicU32; 64],   // line 1..4 (256B = 4 line)
}
```

合格 gate は (a) bench_concurrent **adv-hot read-heavy 16T** で c16s 比 +5% Mops 維持
(c17s の +27.4% を ≥ +20% で保持できれば良い)、(b) Mops skew=1.0 gim T=4 で **regression
≤ 2%** (c17s −3.7% を ≥ −2% に解消)、の同時成立。両方未達なら reject、片方のみなら
条件付きで採否を re-judge。

## 1. 動機 — c17s の (b) gate fail を構造的に解く

c17s-results §4 / §5.1 で確認した skew=1.0 gim T=4〜16 の −3.7%〜−4.7% regression に
ついて、以下 3 つの仮因が並んだ:

1. **Slot32 化 (Entry 16B → 32B)** — entries arena が per-shard 1024B → 2048B (16 → 32 line)。
2. **2-tier seqlock の固定 cost** — try_candidate で entry version load が 2 回追加。
3. **path_c_epoch の reader 側 load** — get_by_hash の miss 経路で 2 回 atomic load。

(2) は §11.1 で tier 2 削除して 0.8pp 縮めた。(3) は §11.2 で hit 経路 skip を試みて
逆悪化、revert (epoch_before は ShardHot の len.load で L1-hot 化済み、削除効果ほぼゼロ)。
**残るのは (1) Slot32 footprint** で、これが skew=1.0 gim (entries 全域 touch) で支配的。

c18s は (1) を構造除去する。Entry sizeof を 16B に戻すには version を Entry から外すしか
ない。**外した version をどこに置くか**が α-2 の設計論点で、自然な解は:

- **versions: [AtomicU32; 64]** を独立配列として持つ (c11s の visited[] と同型の SoA 化)
- per-shard 256B = 4 cache line、Path A の CAS / store 先がここに変わる
- reader は `versions[id]` を load する (cache line touch +1 だが entries[id] line は 1KB 化)

副次的に、c17s-results §8.1 #2 で示唆された **path_c_epoch の reader-only 系への移動**を
同時実施する。c17s では path_c_epoch は ShardHot に同居しており、Path A の visited
fetch_or でその line が常時 invalidate される。adv-hot 高 T 帯で path_c_epoch load が
L1-hot を外れている疑い (§11.2 では gim でしか測っていない)。c18s では versions と同 block に
配置することで writer-hot cluster (Mutex/visited/len) から完全分離する。

## 2. アーキテクチャ — versions 別配列化 + path_c_epoch 移動

### 2.1 tag layout (16 bit、c17s と同 bit 数 / 別 bit 位置)

```text
bit 15  14    10 9              0
┌─────┬───────┬─────────────────┐
│LIVE │ID(6b) │   HASH (9b)     │   ← ID_SHIFT = 4 (Entry 16B)
└─────┴───────┴─────────────────┘
SCAN_MASK = LIVE | HASH (= 0xFFFF & !ID_MASK)
```

ID_SHIFT は `id_shift_from_entry_size(sizeof)` で const-eval 自動算出、Entry sizeof = 16B
で **4** (c17s は 5)。`ID_MASK = 0x3F << 4 = 0x03F0`、`HASH_MASK = 0x7FFF & !0x03F0 =
0x7C0F`、SCAN_MASK = `LIVE | HASH_MASK = 0xFC0F`。HASH bit 数 = 9 (c17s と同等、bit 位置
だけ移動)。

`needle_from_hash` は ID_SHIFT=4 で再導出: 高位 9 bit を取り出し、low_mask = `0x0F`
(下位 4 bit) と high_mask で分配して spread する。

### 2.2 Entry を Slot16 に戻す

```rust
#[repr(C, align(16))]
struct Entry<K, V> { key: K, value: V }   // 16B for u64/u64
```

c11s/c14s/c16s と同じ Slot16 layout に復帰。Entry 自体は version を持たない。
entries arena per-shard cap=64 = 1024B = 16 cache line (c17s 32 line の半分)。

### 2.3 ReaderState block (versions + path_c_epoch、reader-only cluster)

```rust
#[repr(C, align(64))]
struct ReaderState {
    /// Path C 完了時に bump。reader は scan 前後で snapshot して shift-in-progress を
    /// 検出する coarse-grained seqlock。Path A や `writer_update_in_place` は触らない。
    /// line 0 専用化することで versions[*] の Path A W から完全分離 (干渉ゼロ)。
    path_c_epoch: AtomicU64,
    _pad0: [u8; 56],
    /// 各 entry の seqlock version。偶数 = stable、奇数 = in-flight。Path A / Path C
    /// entries 上書きで CAS even→odd → write entries[id] → store even+2 で囲う。
    /// 4 cache line に散る (16 versions/line)、Path A W は id 単位で散布。
    versions: [AtomicU32; 64],
}
```

- 計 5 cache line (1 epoch + 4 versions)
- reader は `path_c_epoch.load → versions[id].load` で同 prefetch 区間を touch
- versions[id] への Path A W は line 1〜4 (16 versions ずつ) に分散、line 0 (epoch) は不汚染
- writer-hot cluster (ShardHot) からは完全独立 → MESI 干渉が path_c_epoch から消える

`_pad0: [u8; 56]` は明示的に **path_c_epoch line を専有** する (versions と同居させない)
ためで、Path C 以外この line を W しないことを保証する。

### 2.4 ShardHot 縮減 (path_c_epoch を取り除く)

```rust
#[repr(C, align(64))]
struct ShardHot {
    writer: Mutex<WriterState>,
    visited: AtomicU64,
    len: AtomicUsize,
    // path_c_epoch は ReaderState に移動。ShardHot は writer-hot cluster 専用化。
    _pad: [u8; ?],   // 64B 維持の sanity assert
}
```

writer 触る hot 集合のみ。reader/Path A は visited (fetch_or) と len (load) でここを
touch するが、path_c_epoch は同 line に居ないので reader-side transfer から外れる。
sizeof=64 / align=64 維持の `const _: ()` assert を残す。

### 2.5 Shard 構造体

```rust
pub struct Shard<K, V> {
    capacity: usize,
    /// tag 列。AtomicU16 で原子操作。R + Path C のみ W (eviction 時の shift loop)。
    tags: Box<[AtomicU16]>,                                    // 2 line
    /// reader-only cluster: path_c_epoch (Path C のみ W) + versions (Path A / Path C W)。
    reader_state: Box<ReaderState>,                            // 5 line
    /// entries arena。R+W workload 依存。Slot16 化で c17s 比半減。
    entries: UnsafeCell<Box<[MaybeUninit<Entry<K, V>>]>>,      // 16 line
    /// writer-hot cluster (Mutex / visited / len)。
    hot: ShardHot,                                              // 1 line
}
```

`reader_state` は `Box<ReaderState>` にすることで Shard struct 自体の sizeof を不変に
保ちつつ heap 上に 5 line aligned で配置する (Box は align 64 を request されたら
allocator 側で satisfy する想定、Rust 1.50+)。

### 2.6 Path A — versions[id] の偶奇 flip で reader 通知

```text
1. find_lockfree_for_path_a(key, needle) → (pos, id, v_snap)
   ※ tag は CAS expected ではなく id 抽出用。version snapshot は versions[id] から取る
2. versions[id].compare_exchange(v_snap, v_snap+1, Acquire, Acquire)   ← odd 化
3. ptr::read entries[id].value (旧 V) + ptr::write entries[id].value (新 V)
4. versions[id].store(v_snap + 2, Release)                              ← even 化
5. visited.fetch_or(1<<pos, Relaxed)
```

c17s の `entries[id].version` 操作と完全に同型、対象が `versions[id]` に変わるだけ。
atomic op 数 (CAS 1 + store 1) は不変。tag は完全不変。

### 2.7 Reader try_candidate

```text
[try_candidate]
  t1 = tags[pos].load(Acquire)              ← ID 抽出用
  if (t1 & SCAN_MASK) != needle: return Miss
  id = id_of(t1)
  v1 = reader_state.versions[id].load(Acquire)   ← tier 1 (Path A 検出)
  if v1 & 1 != 0: return Racing
  buf = ManuallyDrop::new(ptr::read(entries[id]))
  v2 = reader_state.versions[id].load(Acquire)
  if v1 != v2: return Racing
  if buf.key == *key:
      visited.fetch_or(1<<pos)
      return Found(buf.value.clone())
  Miss
```

c17s v2 (tier 2 削除済) と同型、version の load 先が `entries[id].version` から
`reader_state.versions[id]` に変わるだけ。

cache line touch は c17s 比 **+1** (versions[id] line は entries[id] line と別物)。
ただし entries[id] line は 32B → 16B 化で密度向上、entries arena 全体 footprint は半減。
adv-hot で hot key 集中する場合は versions[hot_id] と entries[hot_id] が両方 L1-hot 化
するので影響軽微の見込み。

### 2.8 SHARDS=64 固定

c14s/c16s/c17s と同方針。

### 2.9 Slot16 復帰と footprint

| field | c17s | c18s | 差分 |
|---|---:|---:|---:|
| tags | 128B (2 line) | 128B (2 line) | 0 |
| entries | 2048B (32 line) | 1024B (16 line) | **−1024B** |
| versions | 0 (entries 内同居) | 256B (4 line) | +256B |
| path_c_epoch | (ShardHot 内 8B) | 64B (1 line, _pad 込み) | +56B |
| ShardHot | 64B (1 line) | 64B (1 line) | 0 |
| **計** | **2240B (35 line)** | **1472B (23 line)** | **−768B (−12 line)** |

per-shard footprint は **34% 削減**。skew=1.0 gim (entries 全域 touch) での L1d 圧迫が
構造的に緩和される見込み。adv-hot (1 hot key 集中) では entries[hot_id] 1 line + 
versions[hot_id] 1 line でいずれも L1-hot に張り付くので影響軽微。

## 3. コード変更スコープ

### 3.1 新規ファイル

`research/src/experimental/sieve_c18s.rs` — c17s からのコピー派生。

### 3.2 機械的置換

| c17s | c18s |
|---|---|
| `struct Entry<K, V> { version: AtomicU32, key, value }` (32B, align(32)) | `struct Entry<K, V> { key, value }` (16B, align(16)) |
| ID_SHIFT = 5, ID_MASK = 0x07E0, HASH_MASK = 0x781F | ID_SHIFT = 4, ID_MASK = 0x03F0, HASH_MASK = 0x7C0F |
| ShardHot に `path_c_epoch: AtomicU64` | `path_c_epoch` 削除 (ReaderState へ移動) |
| Shard に `entries`, `hot` のみ | Shard に `reader_state: Box<ReaderState>` 追加 |
| `entries[id].version.compare_exchange` | `reader_state.versions[id].compare_exchange` |
| `entries[id].version.load` | `reader_state.versions[id].load` |
| `entries[id].version.store(v+2)` | `reader_state.versions[id].store(v+2)` |
| `hot.path_c_epoch.fetch_add` | `reader_state.path_c_epoch.fetch_add` |
| `hot.path_c_epoch.load` | `reader_state.path_c_epoch.load` |
| `Entry { version: AtomicU32::new(0), key, value }` | warmup install 時 `versions[len].store(0)` 後に `Entry { key, value }` |

### 3.3 ロジック変更 (機械的でない箇所)

1. **Path B (warmup install)** — `entries[len]` を `MaybeUninit::write` で初期化する前に
   `reader_state.versions[len].store(0, Release)` で version を 0 にリセット。Entry
   自体は `Entry { key, value }` のみ (version field 不在)。
2. **Path C (evict + install)** — evict 対象 `entries[evict_id]` を上書きする際の version
   操作は `reader_state.versions[evict_id]` で行う。idiom は c17s と同型。
3. **`writer_update_in_place`** — Mutex 配下の既存 key 更新。version flip は
   `reader_state.versions[id]` で行う。
4. **`Shard::live_count` / `live_ids`** (test only) — tag の LIVE bit だけ見る既存実装で OK。
5. **`needle_from_hash`** — ID_SHIFT=4 で spread 関数を再導出。

### 3.4 変わらない部分

- SHARDS=64、6-bit ID schema
- AVX2 needle 比較 (SCAN_MASK 値だけが変わる)
- visited bit fetch_or の conditional load (c11s 由来)
- Path B/C escalate の trigger (Path A CAS 失敗時)
- 2-tier seqlock の tier 2 削除済 (c17s v2 を継承)
- `unsafe impl Send/Sync` の SAFETY 議論 (versions[id] への CAS 排他で同型)

## 5. 検証 step

### Step 1: 正しさ — oracle 一致

```bash
cargo test -p senba-research sieve_c18s
cargo test -p senba-research --test oracle c18s
cargo test --workspace
```

合格条件: oracle 4 構成すべて diff=0、`concurrent_invariants_under_zipf` PASS、
全テスト pass。

### Step 2: Mops AB (gate (a) — adv-hot read-heavy 16T)

```bash
./target/release/bench_concurrent --variant c16s,c17s,c18s --threads 16 --shards 64 \
    --cap 4096 --keys 100000 --skew 1.4 --ops 40000000 --warmup 1000000 --trials 5 \
    --op-mix read-heavy
```

**Gate (a)**: `c18s_mops / c16s_mops >= 1.05`。c17s が +27.4% なので、c18s は versions
+1 line touch のコストを引いて +20〜25% で着地予想。

### Step 3: Regression check (gate (b) — skew=1.0 gim T=4)

```bash
./target/release/bench_concurrent --variant c16s,c17s,c18s --threads 4 --shards 64 \
    --cap 4096 --keys 100000 --skew 1.0 --ops 40000000 --warmup 1000000 --trials 5
```

**Gate (b)**: `c18s_mops / c16s_mops >= 0.98` (= regression ≤ 2%)。c17s の −3.7% を
−2% 以内に解消するのが主目的。entries half 化 + path_c_epoch 分離の合算効果で
gate を超えれば採用。

### Step 4: thread sweep T={1,2,4,8,16}

```bash
for t in 1 2 4 8 16; do
  ./target/release/bench_concurrent --variant c16s,c17s,c18s --threads $t --shards 64 \
      --cap 4096 --keys 100000 --skew 1.0 --ops 40000000 --warmup 1000000 --trials 3
done
# 同じく adv-hot read-heavy 軸も
for t in 1 2 4 8 16; do
  ./target/release/bench_concurrent --variant c16s,c17s,c18s --threads $t --shards 64 \
      --cap 4096 --keys 100000 --skew 1.4 --ops 40000000 --warmup 1000000 --trials 3 \
      --op-mix read-heavy
done
```

c16s/c17s/c18s の 3 way sweep で `(workload, T)` 軸 use-case map を更新。

### 合否判定

| Gate (a) | Gate (b) | 判断 |
|---|---|---|
| ✓ | ✓ | **採用**。adv-hot 構造勝ちと skew=1.0 gim 退行解消の両立達成 |
| ✓ | ✗ | adv-hot は維持したが gim 退行解消できず → entries half 化単独では足りない、別案検討 |
| ✗ | ✓ | adv-hot 退行 → versions +1 line touch のコストが想定より大きい、reader cache 設計再考 |
| ✗ | ✗ | reject。c17s 据え置き |

## 7. スコープ外 (今回はやらない)

- **VTune memory-access 検証** — Step 4 で c18s 採用が見えたら follow-up task。Step 2/3 の
  gate と thread sweep だけで一次採否判定。
- **G2-β SoA state** — versions 単独 SoA 化に成功したら、tags/visited も SoA 化する long-term
  上位互換 (improvement-ideas §D.2)。c18s の結果次第で next step。
- **senba lib への取り込み** — c18s は experimental。lib 化は採用が確定し perf-gate /
  oracle が通った後。
- **path_c_epoch 分離単独 AB** — c18s = 「entries half」 + 「epoch 分離」の同時実施で 1 回測定。
  因果分離は Step 4 thread sweep で c17s vs c18s の差分形状を読む (epoch 分離の効きが
  adv-hot 高 T 帯で見える、というのが仮説 3)。
- **String 系 value での検証** — `cseries-string-baseline` で確認した V::Clone soundness
  は c17s から継承 (entry-level seqlock の早期 escape は versions[id] でも同型に効く)。
  別軸として大 value での性能検証は別 task。

## 8. 不採択時の next move 候補

c18s reject 時:

1. **path_c_epoch 単独移動** — c17s から versions 同居をやめて epoch だけ独立 line に。
   adv-hot 高 T で効くなら c17s に back-port。
2. **versions inline + tag 縮約** — Entry を Slot24 (24B = align(8)? padding) で version
   並べ替え、id_shift を 3 にして HASH bit 数を増やす。Slot32 vs Slot16 の中間。
3. **G2-β SoA state** — versions / lock / epoch を単一 word `AtomicU64` にまとめる。実装大、
   AVX-512 V5 統合の前提。
