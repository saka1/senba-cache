### 2.1 前 sweep からの変更

- **Twitter trace を OSDI Yang 形式 5 cluster に変更**: 前 sweep は libCacheSim 同梱の
  cluster52.csv 単独 (libCacheSim 形式) で評価していた。cluster52 は shared-hot 強で
  r1 の不採用領域 trace。Yang 形式 5 cluster (`external/twitter-cache-trace/cluster006`
  〜 `cluster034`) は read-heavy 中位 skew が多く、cluster019 / cluster034 で r1 は
  pareto-dominant。bench harness に `--source twitter-yang` を追加 (`file::twitter_csv_from_path`
  が String key を `DefaultHasher` で u64 化)。
- **ARC trace を 8 種に拡張**: 前 sweep は OLTP のみ。本 sweep で DS1 / S1 / S3 / P1 /
  P8 / ConCat / MergeP を追加し、HR 維持型 (DS1, MergeP) と shared-hot 型 (OLTP) を
  対比できるようにした。
- **T 軸を T=1 から埋めた**: 前 sweep は T=16 cell のみ表示で scalability knee が
  見えなかった。本 sweep で T={1,2,4,8,16} を全部出して x=T, y=Mops, line per WAYS
  曲線を生成 (`docs/benchmark/r1-sweep/figures/fig_mops_vs_threads_*`)。

## 3. 結果

trace replay は **sliced**: thread t は `trace[t·L/T .. (t+1)·L/T)` を循環再生。c17s
baseline も同じ sliced replay で apples-to-apples 比較。

### 3.1 Twitter clusters (OSDI Yang 形式)

T=16 / value=u64 の WAYS sweep:

| cluster | c17s HR | c17s Mops | r1 w=2 Mops/HR drop | r1 w=4 Mops/HR drop | r1 w=8 Mops/HR drop |
|---|---:|---:|---|---|---|
| cluster006 | 0.349 | 31.19 | 27.58 / 12.26pp | 31.25 / 21.56pp | 32.99 / 27.55pp |
| cluster016 | 0.364 | 17.61 | 16.22 / 5.17pp | 19.39 / 7.86pp | **23.78 / 9.14pp** |
| cluster018 | 0.564 | 27.80 | 25.96 / 5.61pp | 28.21 / 12.79pp | **37.33 / 15.89pp** |
| cluster019 | 0.240 | 11.41 | 12.59 / 0.67pp | **14.26 / 1.46pp** | **20.27 / 2.31pp** |
| cluster034 | 0.347 | 16.45 | 17.37 / 2.17pp | **21.50 / 4.65pp** | 28.25 / 7.68pp |

**観察**:
- **cluster019 が r1 の sweet spot**: low baseline HR (0.24) で hot key 集合が小さく、
  WAYS 倍の duplication をしても hot working set 全体が capacity に収まる。w=8 で
  Mops +77.7% / HR drop 2.31pp は本 sweep 最強。
- **cluster034 も採用**: baseline HR 0.35 だが long-tail が短く、w=4 で HR drop 4.65pp 内、
  Mops +30.7%。
- **cluster006 / cluster018 は不採用**: cluster006 は shared-hot 過大 (top key が全
  thread に共有)、cluster018 は cap-fits 強 (HR 0.56) で WAYS 倍 split が HR を直接削る。
- **cluster016 は中間**: w=8 で +35% / 9.14pp。Mops は出るが採用基準 (HR drop ≤5pp) を
  わずかに外れる。

### 3.2 Zipf (合成 workload)

T=16 sweep:

| (skew, op-mix, value) | c17s Mops | r1 w=2 gain/drop | r1 w=4 gain/drop | r1 w=8 gain/drop |
|---|---:|---|---|---|
| s=1.4, read-heavy, u64 | 143.72 | +5.4% / 0.94pp | +13.6% / 2.19pp | +12.8% / 3.79pp |
| s=1.4, read-heavy, string | 92.49 | **+27.3% / 0.93pp** | **+30.5% / 2.16pp** | **+31.8% / 3.75pp** |
| s=1.4, gim, u64 | 159.81 | +6.5% / 0.91pp | +8.9% / 2.21pp | +6.6% / 3.86pp |
| s=1.0, gim, u64 | 54.82 | +2.6% / 5.83pp | +10.0% / 11.64pp | +15.2% / 17.38pp |
| s=0.8, gim, u64 | 35.33 | +4.8% / 7.41pp | +12.5% / 13.91pp | +27.4% / 19.39pp |
| s=1.0, gim, string | 20.09 | +55.7% / 5.86pp | +103.5% / 11.65pp | +127.0% / 17.41pp |
| s=0.8, gim, string | 11.36 | +71.6% / 7.41pp | +139.9% / 13.83pp | +171.1% / 19.42pp |

**観察**:
- **String で gain が劇的に増幅**: s=1.4 read-heavy で u64 +13.6% → string +30.5% (~2.2x)。
  s=1.0/0.8 では +10% → +100% 超に増幅 (low baseline Mops のためゲイン比率が大きく見えるが、
  Mops 絶対値で見ても string で +14 Mops 改善)。eviction path の `Drop<String>` が c17s
  では bouncing と co-located で実行されており、r1 で per-shard local になって overhead
  が直接消える。
- **gim より read-heavy の方が利得安定**: gim (90% get, 10% insert) は writer load が
  低く bouncing 起因の利得が出にくい。
- **s=1.0/0.8 は HR drop が cap で律速**: shared key の long-tail で WAYS 倍 split が
  effective cap を圧縮し、HR が線形に下がる。

T=16 / u64 の WAYS sweep:

|---|---:|---:|---|---|---|
| OLTP | 0.442 | 25.95 | -2.3% / 7.70pp | +17.3% / 14.36pp | +56.6% / 21.53pp |
| DS1 | 0.001 | 7.27 | +3.7% / 0.01pp | +3.8% / 0.03pp | **+36.9% / 0.03pp** |
| MergeP | 0.012 | 17.62 | -5.2% / 0.38pp | +12.2% / 0.74pp | +10.2% / 0.89pp |
| ConCat | 0.013 | 17.45 | -5.6% / 0.48pp | +9.0% / 0.80pp | +19.6% / 0.88pp |
| P8 | 0.046 | 17.28 | +1.1% / 2.47pp | +19.0% / 3.80pp | +3.3% / 4.28pp |
| P1 | 0.048 | 20.33 | -2.4% / 2.52pp | +8.7% / 3.89pp | +2.8% / 4.33pp |
| S1 | 0.004 | 14.90 | -16.6% / 0.21pp | -33.5% / 0.31pp | -33.6% / 0.37pp |
| S3 | 0.004 | 17.27 | -36.8% / 0.27pp | -40.9% / 0.38pp | -44.1% / 0.44pp |

**観察**:
- **DS1 / MergeP / ConCat は HR-preserving zone**: baseline HR が極小 (< 2%) で
  cap-fits 帯から外れているため、WAYS 倍 split しても HR ほとんど変わらず。MergeP は
  T=8 w=4 で +31.2% / 0.46pp、ConCat は T=2 w=8 で +27.5% / 2.54pp が採用領域。
- **DS1 T=16 w=8 +36.9% / 0.03pp** は本 sweep の HR drop 最小採用 cell。ただし T=16
  Mops vs T=8 で baseline 自体が 11.42 → 7.27 と dropout しており (図
  `fig_mops_vs_threads_arc_DS1_cap4000_u64.png` 参照)、WSL2 noise の影響が大きい cell。
  trials=5 への増量が必要。
- **OLTP は cap-fits 強で不採用**: HR 0.44 から WAYS 倍 split で 0.23 (-21.5pp) に
  落ちる。Mops +56.6% は劇的だが採用基準外。
- **S1 / S3 は全域劣後**: HR drop 微小だが Mops も c17s より低い (-33〜-44%)。これは
  trace 長と warmup の関係で、sliced replay 後の T=16 で thread 当たりの slice が
  warmup に届かない可能性が高い (要 follow-up)。

### 3.4 Scalability knee (Mops vs T 曲線)

設計 §3.3 の幾何 "T ≤ WAYS で hot ShardHot line が物理的に別 core の L1 に固定" の
直接観測。cluster019 u64 が最も clean な knee を示す
(`fig_mops_vs_threads_twitter_cluster019_u64.png`):

| T | c17s | r1 w=1 | r1 w=2 | r1 w=4 | r1 w=8 |
|---:|---:|---:|---:|---:|---:|
| 1 | 2.31 | 2.25 | 2.33 | 2.37 | 2.53 |
| 2 | 4.05 | 4.09 | 4.52 | 4.63 | 4.89 |
| 4 | 7.33 | 7.21 | 7.78 | 8.45 | 9.40 |
| 8 | 9.61 | 9.90 | 10.93 | 13.25 | 13.53 |
| 16 | 11.41 | 11.75 | 12.59 | 14.26 | **20.27** |

- **T=1,2 では r1 w=1〜8 が baseline と同等**: contention が無い regime で routing
  affinity は無効、TLS-id load の僅かな overhead で w=1 と c17s が ±3% 一致。
- **T=4 から w=4/w=8 が分離**: bouncing が顕在化、affinity が効き始める。
- **T=16 で w=8 が explosive**: c17s 11.41 → r1 w=8 20.27 で +77.7%。設計仮説
  「T ≤ WAYS で bouncing-free」とは異なり、**T > WAYS でも affinity の partial benefit が
  累積して T が増えるほど絶対値で離れる**。これは sliced replay と SIEVE writer state
  の組合せで T が増えると bouncing 部分が累積し、c17s 側の degradation が r1 側より
  早く始まるため。

## 4. Discussion

### 4.1 cluster019 / cluster034 が r1 の本質的 sweet spot

cluster019 (Twitter ETC キャッシング) は本 sweep で r1 が最強の cell を提供:
- **HR drop ≤2.5pp の中で Mops +77.7%**
- これは前 sweep の cluster52 (shared-hot 過大) で見えなかった r1 の真価。
- 特徴: baseline HR 0.24 (cap=4096 では over-capacity の long-tail 系)、read/write 比
  はやや write 寄り、key universe が広いが top hot key 集合が小さい

設計 §3.1 #2 「shared hot key のうち hot 集合が極端に小さい部分集合」は
adv-hot read-heavy Zipf を念頭に置いていたが、**実 trace では cluster019 のような
medium HR / long-tail 系で同じ条件が自然に成立する**ことが分かった。

### 4.2 String value で routing benefit が増幅される構造

s=1.4 read-heavy で u64 +13.6% → string +30.5% (T=16 w=4)。c17s では `Drop<String>` が
eviction で writer state を持つ thread (= Mutex 取得 thread) 上で実行され、Drop の
allocator call (`free()`) が hot line bouncing と co-located の cache line を touch する。
r1 で shard を thread に固定すると Drop も per-shard local になり、cross-core allocator
contention が消える。

→ **value type が Drop cost を持つほど r1 の routing benefit が増幅される**。これは
本 sweep の発見で、設計時の予測 (string は eviction 直撃で benefit 一律) を
精細化する: routing affinity の利得は Drop コストと positive correlation を持つ。

### 4.3 Twitter cluster 内の r1 適性分布

5 cluster で r1 適性が大きく違う:

| cluster | baseline HR | r1 適性 | 推定 access pattern |
|---|---:|:---:|---|
| cluster019 | 0.24 | ★★★ | medium-skew read-heavy、long-tail |
| cluster034 | 0.35 | ★★ | medium-skew、middle hot set |
| cluster016 | 0.36 | ★ | balanced、Mops 出るが HR drop 限界外 |
| cluster018 | 0.56 | △ | cap-fits 強、Mops 出るが HR drop 過大 |
| cluster006 | 0.35 | ✗ | shared-hot 過大、全 thread が同じ key 集合に concentration |

cluster006 の HR drop 27pp (w=8) は「Twitter 全体での Yang 報告 (Zipf 0.7-0.9) と整合」
ではなく、**cluster006 個別の極端 shared-hot 性** に由来。Twitter ETC cluster の中でも
"商業バナーキャッシュ" (cluster006) のような shared 集中型と "ユーザ recommendation
shard" (cluster019) のような disjoint 寄り型では r1 適性が劇的に違う。

→ r1 の library 化を検討する際には **per-trace adaptive WAYS** が要る (= adaptive WAYS は
follow-up §6 #2)。

### 4.4 ARC traces の HR-preserving zone

DS1 / MergeP / ConCat は baseline HR が極小 (< 2%) で、WAYS 倍 split しても HR が
そもそも cap で律速されているため drop しない。MergeP T=8 w=4 +31.2% / 0.46pp は
**HR を完全に保ったまま** routing affinity を取れる cell の好例。

これは設計時に想定していなかった採用領域: **HR が既に低い trace では r1 の HR drop は
構造的にゼロ**で、Mops gain だけが残る。ARC paper の write-heavy / capacity-bound trace は
全体的にこの zone に入る (= ARC OLTP は例外で cap-fits 帯)。

### 4.5 設計仮説 #3 の訂正

設計 §7 #3 「adv-hot read-heavy で WAYS=4 / T={4,8,16} のいずれかで Mops +20% 以上」は
**value 軸を見落としていた**。u64 では +13.6% (未達)、string では +30.5% (達成)。
r-series 設計の hypothesis space を 1 軸広げる: value type が Drop コストを持つかが
判定に効く。

### 4.6 WSL2 noise (Risk #3) の残存影響

trials=3 で std を error bar に取ったが、ARC DS1 / S1 / P1 で T=4/T=16 の cell が
baseline c17s 側で +/- 30% の dropout を示している (`fig_mops_vs_threads_arc_DS1_*.png`
参照)。これは WSL2 oversubscription regime での kernel scheduler ノイズが主因と推定。
trials=5 / ops=8M / 別バックグラウンド負荷 0 での再計測が必要 (follow-up §6 #1)。

## 5. Verdict

**部分採用 (partial accept)** — 設計 §6.3 criteria に従い:

- ✓ **cluster019 / cluster034** で Mops +30〜78% × HR drop <5pp の採用領域 cell が
  多数 (cluster019 で 6 cell、cluster034 で 2 cell)
- ✓ **Zipf 1.4 read-heavy × string** で Mops +27〜32% × HR drop <5pp (3 cell)
- ✓ **ARC DS1 / MergeP / ConCat / P8** の HR-preserving zone で採用 cell (6 cell)
- ✗ **cluster006 / cluster018 / Zipf 1.0/0.8 / ARC OLTP / ARC S1/S3 / ARC P1** は
  全 cell で採用領域外
- → 採用領域 31 cell / 評価 520 cell ≈ 6%。workload class で見ると 8 class 中 3 class
  (cluster019/034、Zipf 1.4 read-heavy string、ARC 低 HR 系) が pareto-dominant 領域を持つ

実装は `research/src/experimental/sieve_r1.rs` に保持、bench harness (bench_concurrent /
bench_vtune_concurrent) に r1 dispatch を統合済み。**senba lib 化は引き続き保留**: 採用
workload class が明確だが、cluster006 のような不採用 trace で大幅劣後 (-25%) するため
default WAYS=1 + opt-in API でしか出せない。adaptive WAYS (follow-up §6 #2) を
含めた r2s/r3 の設計空間を見てから lib 化判断。

### 旧 verdict との差分

前 report (cluster52 単独 / T=16 偏り) は「条件付き採用 2/132 cell」と書いたが、
**評価軸が狭すぎた**。本 sweep で:
- Twitter cluster set を 5 種に拡張 → 採用領域 trace 2 個発見
- value=string を加える → adv-hot read-heavy の判定が反転
- ARC trace を 8 種に拡張 → HR-preserving zone を発見
- T 軸を T=1 から埋める → scalability knee の位置を確認

旧 report は cluster52 (shared-hot 強) + u64 + T=16 という r1 の不採用領域に偏った
スナップショットだった。本 sweep でその偏りを矯正し、r1 の本質的採用領域を可視化した。

## 6. Follow-ups

優先度順:

1. **trials=5 再計測 (Risk #3)**: WSL2 noise が大きい cell (ARC DS1/S1/P1 T=4 と T=16) を
   trials=5 / ops=8M で再計測。core cell {cluster019, cluster034} × {T=4,8,16} × {w=2,4,8}
   × {u64, string} の 36 cell を ±2σ <2% まで絞る。
2. **adaptive WAYS の prototype**: per-trace に WAYS が大きく違う事実 (cluster006 は w=1、
   cluster019 は w=8) を見て、runtime に hot key entropy / collision rate を観測して
   WAYS を切り替える scheme を r2 として検討。
3. **VTune memory-access** (`bench_vtune_concurrent --variant r1@ways=4 --threads 4`):
   c14s-vtune §3 の 3 hot line abs mem stall が r1 で 0 に落ちることを実測。設計 §6.4
   仮説 #2 の構造除去を確証。
4. **String value の Drop cost prof**: `perf record -e cycles -g` で cluster019 string
   T=16 を c17s vs r1 w=8 で比較し、`__rust_dealloc` (= String Drop) の cross-core
   contention 差分を直接観測。§4.2 の Drop 増幅仮説の直接検証。
5. **per-thread keyspace 系 trace 追加**: server connection-id hash 模擬の disjoint
   trace を 1 個追加して、設計 §7 #5 「HR drop ≈ 0」領域を 1 cell でも観測。
6. **bare-Linux 再計測** (memory `wsl2_measurement_confound` 運用): 本 sweep の核 cell
   (cluster019 全 cell + Zipf 1.4 read-heavy string) を bare metal で 1 回回す。
7. **r2s (soft hybrid) 設計再評価**: cluster019 で r1 が HR drop <2.5pp の cell を出した
   ので、soft hybrid の "HR 完全保証" の motivation は弱まった。逆に cluster006 / OLTP
   のような shared-hot 系で r1 が大幅劣後する事実は r2s motivation を維持する。
   r2s の優先度は前 report からやや低下、adaptive WAYS (#2) を先行検討。
8. **T=32 / WAYS=16 cell**: 本 sweep で除外したが、bare-Linux で 16 phys core 環境を
   作れたら T=32 で oversubscription regime の挙動を見たい。
