## 0. TL;DR

r1 は **shard 間 routing に thread affinity を導入する**新シリーズ r-series の 1st
variant。c17s の単シャード構造 (ShardHot / Slot16 / AVX2 / Mutex co-location) を **無改変
で継承** し、`Cache::shard_of_hash(hash) → shard_idx` という **1 引数の routing 関数を
`shard_of(hash, tls_id) → shard_idx` という 2 引数化** する。SHARDS=64 は固定、6 bit の
shard index を `(set_bits, way_bits)` に分解し、set は hash で、way は TLS thread-id で
選ぶ。

形式的には:

```text
WAYS        : usize        — constructor 引数、power-of-2、1 ≤ WAYS ≤ SHARDS
SETS        = SHARDS / WAYS
set_mask    = SETS - 1
way_mask    = WAYS - 1
shard_idx   = (hash as usize & set_mask) * WAYS + (tls_id & way_mask)
```

`WAYS=1` は c17s と完全に等価 (sanity check)、`WAYS=4` で T=4 まで hot-line bouncing 完全
構造除去、`WAYS=8` で T=8 まで、というスケーリング。**主成果物は (T × WAYS) sweep の
HR vs Mops 曲線**であり、c-series 流の単一 acceptance gate (`Mops +X% で採用`) には収めない
方針 (§6 参照)。

## 1. 動機 — c-series の天井と r-series の解空間

### 1.1 単シャード並行制御は概ね枯れた

c-series で試したこと:

| variant | 主構造 | 結果 |
|---|---|---|
| c8 / c9 | Mutex<HashMap> baseline | 構造 baseline |
| c10s | per-shard Mutex + tag scan | 並行化の出発点 |
| c11s | lock-free find + conditional visited fetch_or | reader path 大幅改善 |
| c13s | Path A CAS + Path B/C escalate | writer 軽量化、SIEVE 意味退行で reject |
| c14s | Path A MAX_RETRY=1 + AVX2 reader | adv-hot HR 回復、Path A 確立 |
| c15s | sloppy visited (TLS-RNG gate) | reject、reader visited 既に 1ns |
| c16s | 3 hot line を ShardHot に co-locate | 採用、VTune で 3 line abs mem stall −30〜−65% |
| c17s | Slot32 + entry-level seqlock | 条件付き採用、adv-hot read-heavy +27.4%、現行 best |
| c18s | versions 別配列 + path_c_epoch 分離 | reject、reader cache line +2 のコストが entries 半減利得を上回り |

c14s-vtune §3 で **L3 内 c2c bouncing が wall-clock の主因**、3 hot line (writer state /
visited / Mutex word) が同格 cluster と断定。c16s でこれら 3 line を **同 cache line に
co-locate** して abs mem stall を 30〜65% 削った。c17s で更に reader の同期通知系を tag
から entry version に逃して adv-hot read-heavy で **+27.4%** の structural win を得た。

しかし c18s reject の含意は重い: field split (versions を別配列に出す) や cache line
boundary の細工は、c17s が既に達した「writer-hot cluster 1 line / reader-hot cluster 1
prefetch 区間」という最適配置の **これ以上の細分化は逆効果**ということ。reader cache line
touch を +2 した代償が entries 半減利得を上回る現象は、**単一 hot line を複数 thread が
触る** という根本構造を温存している限り回避できない。

### 1.2 hot-line bouncing の構造的解消には routing しか手がない

c14s-vtune §3 の hot-line bouncing は **「同じ shard を複数 thread が書く」ことに由来**
する。c16s/c17s はその cache line を 1 つに集約してコストを最小化したが、bouncing 自体は
構造として残っている。これを消すには:

- (A) 同じ shard を複数 thread が書く頻度を下げる
- (B) または、同じ shard でも writer 状態を thread ごとに分離する

(B) は単シャード内の subdivision で、c17s のレイアウトを再分割することになる
(私の Option A 案だった)。SHARDS=64 / cap=1024+ 規模では shard 当たり entries が少なく
(cap=1024 で 16 ent/shard)、内部分割は SIMD scan や Slot layout を直撃する。

(A) こそが r-series の解空間。**「同一 key + 異 thread の write を物理的に別 shard に
振り分ける」** ことで、3 hot line bouncing を構造的に 0 にできる (同一 shard を触る writer
が 1 thread に限定されれば、その shard の hot lines は当該 thread の L1/L2 に固定される)。

### 1.3 r-series と c-series の関係

- r-series = shard 間 routing の解空間 (Cache → shard の写像の設計)
- c-series = shard 内 concurrency primitives の解空間 (1 shard 内の reader/writer protocol)

両者は **直交**。r1 は c17s の単シャード concurrency 機構を 100% そのまま使う。将来 r1
の結果次第で「r 系統 × c 系統の cross product」(例: r1 × c19s) も自然に取れる。

## 2. 解空間の整理 — hard vs soft trilemma

### 2.1 routing 設計の 3 つの dimension

routing 関数 `shard_of: (K, t) → shard_idx` の設計には独立な 3 つの目標があり、3 つ全部
同時に満たす routing は構造的に存在しない:

- **(i) writer hot-line bouncing 構造除去**: 同一 shard を 1 thread しか書かない
- **(ii) HR は c17s と等価**: 同一 key は cache 内に 1 copy だけ存在
- **(iii) read/insert の probe amplification なし**: 1 op = 1 shard touch

これは routing 設計の **trilemma**。各 scheme の trade-off:

| Scheme | (i) bouncing-free | (ii) HR 等価 | (iii) probe O(1) |
|---|:---:|:---:|:---:|
| c17s (hash-only routing) | ✗ | ✓ | ✓ |
| **Hard (per-thread routing)** | **✓** | **✗ (≤ S 倍 drop)** | **✓** |
| Soft (set-assoc + probe) | ✓ | ✓ | ✗ (≤ S 倍 probe) |

r1 は **Hard 側**を採る。理由は §3 で詳述。

### 2.2 Hard scheme の構造

- routing: `shard_of(K, t) → ((hash(K) & set_mask) * WAYS) + (tls_id & way_mask)` で
  deterministic に 1 shard を選ぶ
- 同一 key を thread t1 と t2 が触ると、別 shard に行く (= duplicate 化されうる)
- 同一 (set, way) は 1 thread しか書かない (TLS id が unique なら) → bouncing 構造除去
- HR drop: 1 thread 視点の effective cap = total_cap / WAYS。shared hot key は最大 WAYS 倍
  duplicate 化

### 2.3 Soft scheme の構造

- routing: K → `S = WAYS` 個の shard 集合 (set)
- writer は set 内 1 つを affinity で選んで書く
- reader は set 全 S shard を probe してから miss 判定
- insert は set 全 S shard を probe して existing copy を update、なければ自分の affinity
  位置に新規挿入
- HR は c17s 等価 (1 key 1 copy 維持)
- probe cost: read miss / insert 共に O(S)

### 2.4 prior art map

この hard/soft 区別は分散・並行データ構造文脈で繰り返し再発見されている:

- **Hard 側 (per-X local + 必要時 fallback)**:
  - **LongAdder** (Java util.concurrent): per-thread cell に hard write、`sum()` が soft
    read。write の bouncing 完全除去、read は consolidation cost を負う。
  - **jemalloc tcache / Linux per-CPU page allocator / mimalloc heap-per-thread**: hard
    ownership + 枯渇時 inter-CPU steal の hybrid。
  - **Caffeine read buffer**: read event の per-thread hard write + 後段 single-consumer drain。
  - 本リポでは `write-contention-design-space.md` §7 Phase 2 (packed LongAdder for visited)
    が同 family の解空間として既出。

- **Soft 側 (multi-probe)**:
  - **CPU L1/L2 set-associative cache**: hardware が set 内全 way を並列 tag compare。
    way-prediction で 1st probe を当てる試みあり (= 我々の affinity に相当)。
  - **Cuckoo hashing** (Pagh & Rodler 2001, libcuckoo): k 個の hash 関数で set 全 probe。
  - **Two-choice / power-of-d hashing** (Mitzenmacher): K の置き場候補を d 個用意、最低
    load の slot に挿入、read は両方 probe。
  - **Concurrent optimistic cuckoo** (Fan / Andersen / Kaminsky NSDI '13): 並行 hash table
    で probe amp と consistency を両立する隣接領域。

実は CPU L1 そのものがこの trilemma の hardware 解で、set-associative (soft) で HR を保ち
つつ way-prediction や victim cache で probe amp を緩和、その複雑度を hardware が負担して
いる。

## 3. 設計選択の根拠

### 3.1 なぜ hard scheme を 1st variant にするか

複数の理由:

1. **実装 blast radius が最小**: routing 関数 + TLS id allocator の 2 点修正で c17s から
   派生できる。Shard 内部構造 (`ShardHot` / `Slot16` / AVX2 scan / Path A/B/C) は完全保存。
2. **HR drop が定量化可能・workload 依存性が研究的に最も興味深い**:
   - per-thread keyspace (server connection-id hash 等) → **HR drop ゼロ** (むしろ各
     thread が独立 cache を持つ形になり、c2c traffic 完全除去で正の効果)
   - shared hot key (caching popular content) → **HR drop は WAYS 倍 duplication 分**
   - この差は実 trace でしか測れない (ARC OLTP / Twitter cluster / Zipf で挙動が全部
     違う)。1st variant ではこの distribution を可視化することが主成果。
3. **soft scheme の probe amp は SIEVE の write-heavy miss path と相性が悪い**: SIEVE は
   miss path で insert + (Path C shift) + evict が走る。soft scheme は insert で必ず set
   全 S probe して existing copy 有無を確認しないと duplicate 化する。tag scan を S 倍化
   する直接コストは小さくない (c14s-vtune では `find_get_avx2` は write-contention 相対で
   低かったが、それは bouncing が主因だったからで、tag scan 自体は relevant cost)。
4. **soft hybrid は 2nd variant に温存**: hard で HR drop が想定以上に大きければ、それ
   自体が「shared-hot workload では soft hybrid を入れる動機が定量化された」という有用な
   reject 結果になる。逆に hard で十分なら soft の実装複雑度を負わずに済む。

### 3.2 affinity の同定子に TLS thread-id を採る理由

選択肢:

- (A) TLS thread-local counter (atomic 払い出し)
- (B) Linux `__rseq_abi.cpu_id` (or rseq fallback)
- (C) RDTSCP の TSC_AUX (APIC ID)

途中で (B) を一旦推したが、議論を経て **(A) TLS-id を strict に強い解として採択**。
根拠:

1. **「OS が thread を core に固定する」前提が成立する regime と成立しない regime がある**:
   - N_threads ≤ N_cores の steady-state CPU-bound なら CFS は migrate しない
     (`sched_migration_cost_ns` ~500us)。この regime では CPU-id ≈ TLS-id。
   - N_threads > N_cores (oversubscription) で CFS が time-slice すると、CPU-id 派では
     **同一 (set, CPU way) を複数 thread が共有** → bouncing が部分的に戻る。TLS-id 派は
     thread が migrate しても自身の way を保持するので **bouncing 構造除去は無条件で成立**。
2. **migration event 単発**: CPU-id だと thread の working set は旧 core の sub-shard
   に取り残されて HR drop、TLS-id だと working set は thread に follow する (line を新
   core に pull するだけ)。
3. **CPU-id の優位 (duplicate factor の HW 上限) は benchmark 文脈では効かない**: harness
   で T を {4, 8, 16} と振るので duplicate 数は実験者の制御変数。HW 上限は library 化
   フェーズで初めて意味が出る。
4. **read cost ほぼ同等**: TLS load は `mov [fs:0x...]` 1 命令、rseq cpu_id も同等。
5. **Linux 限定の制約不要**: benchmark の WSL2 / bare Linux / Windows VTune cross-check
   を future option として残せる。

c-series benchmark の文脈では TLS-id が **無条件に strict に強い**ので採択。

### 3.3 hard scheme の幾何学的説明

c17s baseline: hot key K → 単一 shard P。T threads が全員 P に concentration、ShardHot 1
cache line に T-way bouncing。これが c14s-vtune §3 の症状。

r1 (hard): hot key K → set `S_K = {P*WAYS, P*WAYS+1, ..., P*WAYS+WAYS-1}`。thread t →
`P*WAYS + (tls_id mod WAYS)`。**T ≤ WAYS なら T 個の hot ShardHot line が物理的に別 core
の L1 に固定**、bouncing ゼロ。

SHARDS=64 / cap=1024+ という典型 regime で `WAYS=4` を取れば、SETS=16 で hash 分散性は
維持されつつ (Zipf 1.0 で 16 bin 均等性 σ/μ ~0.35 程度で十分均等)、bouncing 構造除去の
freedom を way 軸として明示的に確保する仕掛けになる。

### 3.4 set-associative framing の利点

`shard_of(hash, tls_id) → shard_idx` を以下の 2 通りの mental model で記述できる:

- **View 1 (set-associative)**: SHARDS=64 を `SETS × WAYS` 個の set に組み、K → set、TLS
  → way。set 内 way の選択が thread-deterministic な点で、厳密な CPU L1 set-associative
  (eviction policy が way を選ぶ) とは異なるが、構造としては parallel。
- **View 2 (S parallel sub-caches)**: cache を **WAYS 個の独立 c17s sub-cache を glue**
  したものと見る。thread t は `t mod WAYS` の sub-cache を使う。sub-cache 内は完全に
  c17s として動く (SHARDS=SETS、cap=total/WAYS)。

View 2 が implementation 上はっきりしていて、**「c17s を WAYS 個並べる + TLS dispatch を
front-end に入れる」** という最小 diff の理解が成立する。View 1 が perf 解析 (set 内の
way 競合の構造) に向く。両 view を併用する。

## 4. アーキテクチャ

### 4.1 routing 関数

```rust
#[inline]
fn shard_of(&self, hash: u64, tls_id: u32) -> usize {
    let set = (hash as usize) & self.set_mask;
    let way = (tls_id as usize) & self.way_mask;
    set * self.ways + way
}
```

- `self.ways` = WAYS (constructor 設定)、power-of-2
- `self.set_mask` = SETS - 1 = (SHARDS / WAYS) - 1
- `self.way_mask` = WAYS - 1
- `hash` の低位 bit を set 選択に、TLS-id の低位 bit を way 選択に使う。両者は独立な
  bit field を消費するので statistical bias なし
- 計算は AND × 2 + IMUL × 1 (WAYS が compile-time const なら shift)

実装最適化として `WAYS` を const generic 化することも検討するが、研究 variant としては
runtime 値で十分 (sweep の harness が楽)。

### 4.2 TLS id allocator

```rust
static NEXT_TLS_ID: AtomicU32 = AtomicU32::new(0);

thread_local! {
    static TLS_ID: Cell<Option<u32>> = const { Cell::new(None) };
}

#[inline]
fn current_tls_id() -> u32 {
    TLS_ID.with(|cell| {
        match cell.get() {
            Some(id) => id,
            None => {
                let id = NEXT_TLS_ID.fetch_add(1, Ordering::Relaxed);
                cell.set(Some(id));
                id
            }
        }
    })
}
```

- 各 thread 初回 cache 操作時に atomic counter で uniq id を払い出し
- thread 終了時の id 回収は **やらない** (research 範囲外、短命 thread が多い workload
  は別問題)
- id 空間は u32 で 40 億超まで耐性。harness benchmark では T ≤ 32 で十分
- `with` の cost は `mov [fs:tls_off]` + match で ~1 ns、library load 1 段増分は許容
- module: `research/src/experimental/tls_id.rs` (新規) または `sieve_r1.rs` 内 private

### 4.3 Cache::Inner 拡張

```rust
struct Inner<K, V, S, H> {
    shards: Box<[Shard<K, V, S>]>,
    set_mask: usize,
    way_mask: usize,
    ways: usize,
    hasher: H,
    has_avx2_bmi1: bool,
}
```

`shard_mask` が `set_mask` + `way_mask` の 2 field に置き換わる。`ways` field は IMUL の
入力。`ways=1` のとき `set_mask = shard_mask`, `way_mask = 0` で c17s と等価動作。

### 4.4 公開 API の変更

`r1` は研究 variant なので独立 module (`sieve_r1.rs`) として持つ。`senba::Cache` 自体は
無改変。`Cache::with_ways(capacity, shards, ways, hasher)` のような新 constructor を
研究 variant 側 wrapper として提供する。

`CacheImpl` trait (research/src/experimental/mod.rs) は無改変。`get/insert/remove/contains`
の signature は同じで、内部 routing が TLS-id を参照するだけ。

### 4.5 read path / write path

両 path とも c17s と同型、`shard_of_hash(h)` → `shard_of(h, current_tls_id())` の 1 行
置換のみ。

```rust
fn get(&self, key: &K) -> Option<V> {
    let h = self.hasher.hash_one(key);
    let tls = current_tls_id();
    self.shards[self.shard_of(h, tls)].get(key, h, self.has_avx2_bmi1)
}
```

c17s の Shard 内部 (`ShardHot`, `Slot16`, AVX2 scan, Path A/B/C, 2-tier seqlock, Path C
epoch) は 100% 不変。

### 4.6 capacity 配分

総 capacity を `WAYS` で割って各 sub-cache に等分 → さらに `SETS` で割って各 shard に分配。
これにより 1 thread 視点の effective cap = `total_cap / WAYS` (View 2 の sub-cache cap と
一致)。

c-series 既存と同じ `base = capacity / SHARDS; extra = capacity % SHARDS` の分配ロジックを
そのまま流用。`WAYS` 軸は capacity 計算には影響しない (SHARDS=64 固定なので個別 shard
cap は WAYS によらず一定)。

### 4.7 oracle test の扱い

r1 は単一 thread でも routing は (hash, tls_id) → shard_idx で動く。tls_id が固定 (test
内で 1 thread) なら結果は (hash, tls_id_constant) で決まり、`sieve_orig` (hash → shard) と
の等価性は **shard 集合が違うので一致しない**。

→ oracle test は **WAYS=1 (= c17s 等価) で sieve_orig との一致を確認** することにする。
WAYS≥2 では evicted-key sequence が原理的に違うので、oracle test は走らせない。代わりに
`concurrent_invariants_under_zipf` (shard 内一意性 / LIVE 一貫性) を WAYS sweep で全 PASS
することを必要条件とする。

- `concurrent_test_suite.rs` の既存 test を r1 でも走らせる (WAYS sweep)

## 6. 計測計画 (主成果物: HR vs Mops 曲線)

### 6.1 perf-gate (c-series 単シャード性能の維持確認)

r1 は WAYS=1 で c17s と bit-for-bit 等価動作するはずなので、**perf-gate 8 シナリオを
WAYS=1 で走らせて regression なし**を確認する。

```bash
cargo bench -p senba-research --bench sieve_cache_perf -- --save-baseline c17s-pre
# r1 実装後
cargo bench -p senba-research --bench sieve_cache_perf -- --baseline c17s-pre
```

ただしこの perf-gate は単シャード並行 cache の gate であり、**r1 採否の基準ではない**。
r1 採否は §6.2 で行う。perf-gate はあくまで「c17s と等価動作する WAYS=1 path で構造的
退行を起こしていない」ことのサニティチェック。

> **gate 設定の整理**: r-series は単シャードとは別の解空間。c-series 流の単一閾値 acceptance
> (例: `gim T=4 Mops +X% で採用`) は r-series の研究目的 (HR drop と Mops gain の trade-off
> 構造の可視化) と整合しない。`+X%` を gate に置くと、その条件で勝つ WAYS 値だけが採用に
> なり、HR drop の大きい workload で reject される構造的弱点が見えなくなる。

### 6.2 主成果物: (T × WAYS) sweep の HR vs Mops 曲線

これが r1 の研究 deliverable の中心。

**軸**:
- T ∈ {1, 2, 4, 8, 16, 32}
- WAYS ∈ {1, 2, 4, 8, 16, 32}
- ただし T=1 / WAYS=1 は c17s 等価 sanity 用、`T ≥ WAYS` の組合せは bouncing 部分残存域で
  解析的に興味深い (T=2*WAYS / T=4*WAYS 等)

**workload (各 cell で測る)**:

| workload | 想定挙動 |
|---|---|
| Zipf 1.0 gim | shared-hot 中庸、HR drop が WAYS に比例予想 |
| Zipf 1.4 adv-hot read-heavy | c14s-vtune scenario、Mops gain 最大予想 |
| Zipf 0.8 (近 uniform) gim | shared-hot 薄い、HR drop は per-thread keyspace 漸近 |
| Twitter cluster (5 種) | 実 trace 混合、bench_concurrent の既存 source |
| ARC OLTP cap=4000 | senba 既往優位 trace、HR 維持できるか |

**測る指標**:
- HR (hit rate)
- Mops (throughput)
- p99 latency (thread sweep の thermal/contention indicator として)

**plot**:
- 各 workload で 2D 図: x = WAYS、y = Mops、color = T。HR の値を annotation。
- 各 workload で 2D 図: x = WAYS、y = HR、color = T。Mops を annotation。
- per-workload pareto: x = HR drop (vs c17s)、y = Mops gain (vs c17s)。各 (T, WAYS) cell
  を点として打つ。**この pareto frontier が成果物の核**。

### 6.3 判定基準 (採用 / reject)

単一 gate は置かない。代わりに **pareto frontier の構造** で判定する:

- **採用条件**: 1 つ以上の (T, WAYS) cell で **(HR drop < 5pp) かつ (Mops gain > +20% vs c17s)**
  を達成し、その workload が実用的に意味のあるもの (= per-thread keyspace 系または
  adv-hot read-heavy 系)。
- **強く採用**: 複数の workload (例えば adv-hot + Twitter clu1 + ARC OLTP) で上記条件を
  pareto frontier に乗せられる。
- **reject**: 全 workload / 全 (T, WAYS) で c17s を pareto dominate できない。
- **条件付き**: 特定 workload (例: shared-hot uniform) で HR drop が大きすぎて pareto
  劣後だが、別 workload で勝つ → soft hybrid (2nd variant = r2s) の動機が定量化された
  という artifact 価値で部分採用。

### 6.4 補助計測 (採用が見えたら follow-up)

- **VTune memory-access** (`bench_vtune_concurrent.rs` 経由、`Variant::R1 { ways: 4 }`):
  c16s/c17s で測った 3 hot line (writer state / visited / Mutex word) の abs mem stall が
  WAYS=4 / T=4 で **0 に落ちる**ことの確認。落ちなければ TLS-id 払い出しの実装に bug。
- **HW counter**: L3 miss / cross-socket transfer (`mem_load_l3_miss` 等) の T sweep。
  WAYS=4 / T=4 で flat、WAYS=4 / T=16 で再上昇するはず (4 thread / way の oversubscription)。
- **TLS overhead**: WAYS=1 / T=1 で c17s と Mops 差分。`current_tls_id()` の TLS load
  cost が見える。≥ 2% なら inline 化を再考。

## 8. スコープ外 (今回はやらない)

- **soft hybrid (r2s)** — r1 結果次第で 2nd variant 候補。set 全 probe で 1 copy 保証する
  scheme。実装大、SIEVE write-heavy miss path と probe amp の相性が未知。r1 で HR drop
  が小さい workload が存在しなければ r2s を起こす意義が薄い (= r1 の結果がそのまま
  r2s gate を決める)。
- **CPU-id (rseq) 派生** — TLS-id 採択の §3.2 で議論済。Linux 限定 + N_threads ≤ N_cores
  限定で意味のある最適化なので、TLS-id 派生の結果次第で次の variant 候補。
- **WAYS=64 (= 1 set / 全 way)** — set 数が 1 になり hash 分散性が完全に潰れる極端構成。
  研究的興味はあるが workload sweep の主帯ではない。WAYS={1,2,4,8} を主帯とし WAYS=16/32 を
  edge case として計測。
- **senba lib への取り込み** — r1 は experimental。lib 化は採否確定 + perf-gate /
  oracle が通った後。
- **`Cache::Clone` の意味** — r1 の Clone は per-thread state を共有しない (TLS-id は
  thread-local)。Clone 後の cache が同じ key を別 thread から触ると別 shard に行く挙動が
  c17s と異なるが、これは r1 の仕様として受容する。
- **graceful TLS-id 回収** — thread 終了時の id 再利用は実装しない。短命 thread が大量に
  生まれる workload は別問題として記録のみ。

## 9. 不採択時の next move 候補

r1 全面 reject 時:

1. **soft hybrid (r2s)** — set 内 probe で 1 copy 保証。HR 維持と引き換えに probe amp。
   r1 reject の原因が「全 workload で HR drop が大きすぎ」なら r2s の動機がある。
2. **adaptive WAYS** — hot key 検出時のみ duplicate を許す 2 段構成。実装大、c-series
   から離れる方向。
3. **per-thread mini-shard insertion** — hot K だけ TLS スタックの mini-cache に置き、
   背後は c17s。Caffeine ring buffer に近い構造。

r1 部分採用 (特定 workload のみ勝つ) 時:

1. **WAYS を default 1 + opt-in 構造化** — library 用 API として "with_routing_affinity"
   を expose、適用 workload は user 判断。
2. **r2s に進む** — soft hybrid が部分採用 workload を埋められるか測る。

## 11. 議論サマリ (discussion log)

この設計に至るまでの 1on1 議論で確定した事項を、後続実装者が文脈を見失わないよう記録:

1. **c-series 単シャード saturation 認識**: c18s reject 後、単シャード layout / SoA 化の
   field split で得られる利得は概ね枯れたと判断。次の天井は routing 側 (本企画)。
2. **「sub-sharding」命名は誤導的**: 当初 "primary × sub" 案を出したが、これは「shard を
   内部分割する」誤読を産み、SHARDS=64 を変えない set-associative routing と等価であり
   ながら framing が悪い。r-series / set-associative routing の語彙が clean。
3. **TLS-id vs CPU-id 議論**: CPU-id (rseq) を一旦推薦したが、「OS が固定する前提が
   oversubscription regime で壊れる」事を整理して TLS-id が strict に強いと結論。
   N_threads ≤ N_cores では両者 equivalent、N_threads > N_cores で TLS-id が単独で勝つ。
4. **hard vs soft 区別の prior art**: CPU L1 set-associative の direct-mapped vs
   set-associative、LongAdder、jemalloc tcache、cuckoo hashing 等で繰り返し再発見されている
   構造区別。trilemma (i)(ii)(iii) のうち c17s = (ii)(iii)、hard = (i)(iii)、soft = (i)(ii)。
5. **perf-gate の意味付け再整理**: 単シャード並行 cache のための gate であり、r-series
   の解空間とは別。r1 では perf-gate を WAYS=1 sanity に格下げし、採否判定は (T × WAYS)
   sweep の HR vs Mops pareto で行う。
6. **HR drop は softでも完全には避けられない**: soft scheme でも insert race で transient
   duplicate が発生しうる事に注意。hard と soft の差は「duplicate を構造として受け入れるか
   probe で除去するか」の選択。
7. **r-series 命名**: routing が解空間の本質、c-series (concurrency primitives) と直交。
   1st variant 名 = r1。
