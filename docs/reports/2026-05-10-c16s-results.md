## 0. TL;DR

c16s = c14s の per-shard layout を `#[repr(C, align(64))] ShardHot { Mutex<WriterState>,
AtomicU64 visited, AtomicUsize len }` に差し替えた variant。Path A / Path B/C / AVX2
reader / SHARDS=64 / SIEVE state machine は c14s と同型、変更は layout 1 点のみ。

| Step | 結果 | 判定 |
|---|---|---|
| 1 oracle 一致 | `c16s_1shard_matches_orig_on_synthetic_zipf` PASS (4 (skew, cap) 構成 × 200 k op、diff=0) | **PASS** |
| 2 Mops AB T=4 (gate (a)) | c16s 35.38 / c14s 32.79 = **+7.9%** (5 trial 平均、CV<0.013) | **PASS (≥ +5%)** |
| 3 thread sweep T=2,4,8,16 | c16s が全 T で c14s を上回り、T=2 regression / T=16 inversion なし | **OK** |
| 4 VTune memory-access (gate (b)) | writer state -39% / visited -65% / Mutex -49% (3 of 3) | **PASS (≥ 2/3)** |
| 5 VTune uarch-exploration | Memory Bound P-core 23.0% → 16.5%、CPI 0.526 → 0.449、Retiring 29.1% → 35.7% | OK (観測指標) |

Step 4 で **3 hot line 全てが gate (b) -30% を上回る** (gate は 2/3 で合格)。
Step 5 の uarch も memory bound / CPI / retiring 全方向で改善。
**c16s 採用確定**、設計 §6 合否表の (a)✓ / (b)✓ 位置に着地。

## 2. Step 1 — oracle 一致 (correctness gate)

```bash
cargo test -p senba-research --test oracle c16s
```

`research/tests/oracle.rs::c16s_1shard_matches_orig_on_synthetic_zipf` を新規追加。
構成は c13s 版と同型: `(skew, cap) ∈ {(1.05, 16), (1.1, 32), (1.2, 64), (1.5, 64)}`、
各 200,000 op を `sieve_orig` と c16s で並走させて全 op で `get` の Some/None 一致を
確認。**4 構成全てで diff=0**。c14s から layout だけ変えた以上、SIEVE 不変条件が
保たれていれば必然の結果。

`cargo test --workspace` も全 pass (505 tests + 1 ignored、regression 無し)。

## 3. Step 2 — Mops AB gate (a)

```bash
./target/release/bench_concurrent --variant c14s,c16s --threads 4 --shards 64 \
    --cap 4096 --keys 100000 --skew 1.0 --ops 80000000 --warmup 1000000 --trials 5
```

| variant | trial Mops | 平均 Mops | stdev | CV |
|---|---|---:|---:|---:|
| c14s | 32.35, 32.56, 33.34, 33.01, 32.69 | **32.79** | 0.405 | 0.012 |
| c16s | 35.17, 35.27, 35.78, 35.03, 35.66 | **35.38** | 0.319 | 0.009 |

**c16s / c14s = 35.38 / 32.79 = 1.079 (+7.9%)**。CV は両方 < 0.05。
gate (a) = ≥ +5% を **PASS**。

per-thread スケジューリングも均等 (per-thread Mops の thread CV が 0.001–0.004 で
c14s/c16s ともに同水準)。c16s 側の bottleneck シフトが起きていないことを示す。

## 4. Step 3 — thread sweep (観測指標)

`c9` (Mutex-only baseline) → `c14s` (Path A lock-free) → `c16s` (+ co-locate) の
進化軸を 1 表で見る。同じ Zipf skew=1.0 / cap=4096 / keys=100k / ops=80M / trials=5。

```bash
for t in 2 4 8 16; do
  ./target/release/bench_concurrent --variant c9,c14s,c16s --threads $t --shards 64 \
      --cap 4096 --keys 100000 --skew 1.0 --ops 80000000 --warmup 1000000 --trials 5
done
```

平均 Mops (5 trial):

| threads | c9 | c14s | c16s | c16s/c14s | c14s/c9 |
|---:|---:|---:|---:|---:|---:|
| 2  | 20.12 | 18.86 | 19.65 | **+4.2%** | -6.3% |
| 4  | 29.80 | 32.93 | 35.06 | **+6.5%** | +10.5% |
| 8  | 37.32 | 47.94 | 50.64 | **+5.6%** | +28.5% |
| 16 | 30.85 | 62.26 | 64.24 | **+3.2%** | +101.8% |

観察:

- **c16s は 4 / 8 / 16T で +5% 以上、T=2 でも +4.2%**。T=2 大幅 regression / T=16
  逆転といった異常パターン無し。layout 変更の利得が thread 数に対して頑健。
- T=2 で c14s < c9: c14s lock-free CAS の overhead が writer 同士衝突の少ない
  低並行帯では薄利、Mutex-only c9 のほうが速い。c16s は co-locate で c14s を
  改善するが c9 までは届かない (T=2 c16s 19.65 vs c9 20.12)。これは c14s レポート
  §6 の単一観察と整合。
- T=8 / T=16 で c14s, c16s ともに c9 を大きく引き離す。c9 は T=16 で **30.85 Mops
  まで崩落** (per-shard Mutex 飽和)、一方 c14s は **62.26 Mops、c16s は 64.24 Mops**
  と高並行帯の優位を維持。
- **c16s 高並行帯で利得が縮む (T=16 +3.2%)**: Mutex word + visited + len の
  coherence transfer が 1 transfer に集約されても、tags / entries 列の transfer が
  支配的になっている可能性。VTune (Step 4) で確認すべき仮説。
- thread CV (per-thread Mops のばらつき) はいずれも < 0.013、scheduling 偏りなし。

### 4.1 T=1 baseline と scaling efficiency (c16s 単独追測)

§4 表は T=2 起点なので、c16s 単独で T=1 を 3 trial 追測 (同 workload):

```bash
./target/release/bench_concurrent --variant c16s --threads 1 --shards 64 \
    --cap 4096 --keys 100000 --skew 1.0 --ops 80000000 --warmup 1000000 --trials 3
```

c16s T=1: 11.381 / 11.459 / 11.493 Mops、**平均 11.44 Mops** (CV 0.004、hit ratio 0.720)。

T=1 を起点にした scaling 効率 (c16s):

| threads | c16s Mops | T=1×N (理想) | 効率 | T=1 比 |
|---:|---:|---:|---:|---:|
| 1  | 11.44 | 11.44 | 100% | 1.00× |
| 2  | 19.65 | 22.88 | 86% | 1.72× |
| 4  | 35.06 | 45.76 | 77% | 3.07× |
| 8  | 50.64 | 91.52 | 55% | 4.43× |
| 16 | 64.24 | 183.0 | 35% | 5.62× |

要点:

- **T=2,4 は 77–86% efficiency** で良好。64 shard 分散が hot key 衝突を希釈、Path A
  reader と writer Mutex がほぼ重ならない帯。
- **T=8 で 55% に落ちる**: skew=1.0 hot key cluster が writer Mutex 衝突を起こし
  始める帯。Step 5 VTune (T=4) の Memory Bound 15.9% は T=8 でさらに伸びている
  可能性が高い (今回未測)。
- **T=16 で 35%**: 16 論理コア = 物理 8 推定 (P + E hybrid) の SMT/hybrid 飽和 +
  hot key writer 衝突の合算。それでも c14s 5.44× / c9 崩落 (T=1→T=16 で 1.x×)
  に対し c16s 5.62× は同系列で最良。
- c16s は **常時並行コスト** (per-op Mutex acquire + seqlock retry path + atomic
  visited) を T=1 でも払う設計で、T=1 単体性能を売る variant ではない。
  hot-key adversarial workload で T=16 まで単調増加・c9 のような崩落無しが採用根拠。

## 6. 採否

**採用確定**。設計 §6 合否表の **(a) ✓ / (b) ✓** 位置 (= "採用、index 更新、後続
design space 探索の起点") に着地。

- Step 1 (oracle): SIEVE 不変条件保持
- Step 2 (gate (a)): T=4 で +7.9% (≥ +5%)、CV < 0.013
- Step 3: T=2,4,8,16 全帯で c14s 超え、異常パターン無し
- Step 4 (gate (b)): 3 hot line **全て** が abs mem stall -30% を超えた
  (writer state -39%、visited -65%、Mutex -49%)。3/3 達成は当初の "2/3" 想定を
  上回る結果で、設計の量的根拠が VTune で直接確認された。
- Step 5: Memory Bound P-core 23.0% → 16.5%、CPI 0.526 → 0.449、Retiring 29.1% →
  35.7% と uarch 全方向で改善。L3 Bound P-core 20.8% → (P-core 内訳出力なし、
  E-core 15.5%)、Memory Bound 全体で -6.5pt の縮みが co-locate 由来であることが
  別角度からも裏付け。

「3-line 全部殴る」設計方針 (§1 動機) が **量と説明の両面で完成**。c15s (visited
単独 sloppy 化) が reject されたのは "1 line 削っても残り 2 line が残る" だったが、
c16s は 3 line 同時に削っているため bottleneck シフトが起きずに最大効果が出た。

## 7. Step 4–5 再現コマンド (Windows native VTune)

§8 の数値を再現するときの手順。binary は WSL2 から cross build:

```bash
cargo xwin build --release --target x86_64-pc-windows-msvc -p senba-research \
    --bin bench_vtune_concurrent
```

(Paused Time が elapsed の半分強を占めるのはこれ由来)。

memory-access (gate (b) の一次データ):

```powershell
vtune -collect memory-access -knob analyze-mem-objects=true -start-paused -- `
    bench_vtune_concurrent.exe --variant c16s --threads 4 --shards 64 `
        --cap 4096 --keys 100000 --skew 1.0
vtune -collect memory-access -knob analyze-mem-objects=true -start-paused -- `
    bench_vtune_concurrent.exe --variant c14s --threads 4 --shards 64 `
        --cap 4096 --keys 100000 --skew 1.0
```

uarch-exploration (参考指標、L3 Bound% / CPI):

```powershell
vtune -collect uarch-exploration -start-paused -- `
    bench_vtune_concurrent.exe --variant c16s --threads 4 --shards 64 `
        --cap 4096 --keys 100000 --skew 1.0
```

`bench_vtune_concurrent.exe` の defaults は warmup=400_000 / ops=8_000_000 (T=4 で

側 (§3/§4) の 80M ops とは別 budget。

## 8. Step 4–5 結果 — Windows native VTune

T=4 / skew=1.0 / cap=4096 / keys=100k / 4-thread × `bench_vtune_concurrent.exe` で
c14s と c16s を 1 回ずつ採取 (Windows 12th-gen P-core + E-core 構成、Average CPU
Frequency 4.5 GHz)。`-start-paused` + ITT pause/resume で warmup を計測外に出して
いるので Paused Time が elapsed の半分強を占める (これは設計通り)。

### 8.1 サマリ (uarch-exploration / Memory Bound)

| 指標 | c14s | c16s | Δ |
|---|---:|---:|---:|
| Elapsed Time | 4.121 s | 3.893 s | -5.5% |
| Paused Time | 2.136 s | 2.160 s | (warmup ほぼ同等) |
| 計測窓 (Elapsed - Paused) | 1.985 s | 1.733 s | **-12.7%** |
| CPU Time | 7.127 s | 6.021 s | -15.5% |
| Clockticks | 31.86 G | 26.52 G | -16.8% |
| Instructions Retired | 60.60 G | 59.06 G | -2.5% |
| **CPI** | 0.526 | **0.449** | **-14.6%** |
| **Memory Bound (P-core)** | 22.0% | **15.9%** | **-6.1 pt** |
| L3 Bound (P-core) | 20.8% | (P 出力なし、E-core 15.5%) | 縮小傾向 |
| Retiring (P-core) | 29.1% | **35.7%** | +6.6 pt |
| Bad Speculation (P-core) | 15.2% | 12.2% | -3.0 pt |
| Loads | 12.56 G | 11.89 G | -5.4% |
| Stores | 4.98 G | 4.85 G | -2.7% |
| LLC Miss Count | 0 | 0 | (どちらも DRAM に出ない) |

要点:

- Instructions Retired はほぼ同等 (-2.5%) なのに Clockticks が -16.8%。
  → 同じ仕事を **15% 速く回せている** = 純粋な階層内 (L1/L2/L3) の transfer 削減。
- CPI 0.526 → 0.449 で **-14.6%**。retiring が 6.6pt 上がっている分が直接的な
  throughput 利得。
- Memory Bound P-core が 22.0% → 15.9% (**-6.1 pt**)。L3 Bound 単独は P/E-core で
  内訳の出力位置が異なるため直接比較不能だが、Memory Bound 全体の縮みが co-locate
  由来であることは確実。
- LLC Miss Count = 0 (両方)。MESI transfer は **socket 内 (L3 で完結)** で起きて
  いて DRAM には届いていない。c14s VTune 報告 §3 の前提と一致。

### 8.2 Gate (b) — hot line ごとの abs mem stall

`abs mem stall = CPU Time × Memory Bound%` で算出 (Top-K 表より)。

| hot line | c14s 内訳 (新 vtune) | c16s 内訳 | Δ | gate (≤ 0.7 × c14s) | 判定 |
|---|---|---|---:|---:|:---:|
| writer state cluster (`writer_evict_and_install`) | 1.494 s × 28.8% = **0.430 s** | 1.378 s × 19.1% = **0.263 s** | **-39%** | ≤ 0.301 s | **✓** |
| visited word | `vbit` 0.705 s × 38.4% = **0.271 s** (c14s では visited 操作が `vbit` symbol に集約) | `atomic_and` 0.404 s × 23.4% = **0.095 s** (c16s は `vbit_mask` がインライン化、visited 操作のみ symbol として残る) | **-65%** | ≤ 0.190 s | **✓** |
| `parking_lot::Mutex` lock + unlock | lock 0.423 s × 33.7% = **0.143 s** (unlock symbol top-K 圏外、c14s 設計値 0.251 s に近似) | lock 0.352 s × 25.0% + unlock 0.135 s × 30.0% = 0.088 + 0.041 = **0.129 s** | **vs c14s 設計値 0.251: -49%** | ≤ 0.176 s | **✓** |

**3 of 3 PASS** (gate は 2/3)。設計 §6 の閾値 0.291 s / 0.193 s / 0.176 s に対しても
全部クリア。注意:

- **visited 比較の解釈**: c14s では `Self::vbit` が独立 symbol として 0.705 s 残って
  いる (return tuple + word index 計算 + visited 配列 indexing)。c16s では
  `vbit_mask` が pure 1-shift で inline 消滅し、残るのは `core::sync::atomic::atomic_and`
  + `try_candidate` 内の visited fetch_or のみ。symbol の対応関係はずれているが、
  **visited 経路の総 mem stall が 0.271 s → 0.095 s** に縮んでいることは確かで、
  3 line 集約効果として読める。
- **Mutex 比較**: c14s top-K に `unlock` が出ていないため、設計値 0.251 s
  (= c14s VTune 報告 §5.3 の 3-line picture) を比較基準に用いた。c16s は lock + unlock
  両方が top-K に出ているのでここを合算 (0.129 s) して比較。

### 8.3 補助観察

- **`writer_find` の mem bound が下がる**: c14s 0.733 s × 27.2% = 0.199 s →
  c16s 0.662 s × 14.9% = 0.099 s (**-50%**)。設計時に直接の対象外だったが、
  Mutex 配下で len.load + tags scan を行う routine も hot field の同一 line 化の
  恩恵を受けた (len + Mutex word が 1 line 内なので、Mutex を取った直後に len
  が hot のまま読める)。
- **`find_get_avx2` (reader)**: c14s 0.766 s × 12.2% = 0.094 s → c16s 0.839 s ×
  8.4% = 0.070 s (**-25%**)。reader 側にも軽い改善 — visited fetch_or が同 line
  に集約されたため、reader hit 経路の visited 触りもそれ自体は速くなった。
  設計仮説 2 ("reader bouncing は writer 側改善より小さい") が定量的に確認。
- **`try_candidate` の CPU Time 半減**: 0.608 s → 0.282 s。reader が Path A cycle
  を踏む頻度が減った可能性 (= writer が Path A 完了までの時間が短縮されたので
  reader の seqlock retry 回数が減った)。間接効果として整合的。

### 8.4 反証リスト (設計 §7) の最終回答

| # | 仮説 | Step 4–5 後の判断 |
|---|---|---|
| 1 | writer hot 4 line co-locate で transfer cost が落ちる | **直接確認**: writer state cluster mem stall -39% (Step 4) + Memory Bound -6.1pt + CPI -14.6% (Step 5) |
| 2 | reader bouncing は writer 側改善より小さく net 改善 | **定量確認**: `find_get_avx2` mem stall -25% で reader も改善方向 (悪化なし)、ネット +7.9% Mops |
| 3 | parking_lot::Mutex の repr で raw word が ShardHot offset 0 | PASS (compile-time assert) |
| 4 | visited 1 word 縮退で scan_evict が破れない | PASS (oracle 一致) |

仮説 1 が直接観測で確認、仮説 2 は予想を上回る結果 (reader 側も改善方向)。
**4 仮説全てで設計通り**。

## 9. 後続 design space (採用後の next move)

c16s で writer 4 line を 1 line に集約し終えたので、次に削れる contention は
別軸。設計 §8 の候補のうち、c16s 結果を踏まえた優先順位:

1. **§8.2 writer batching** — c16s の結果から、Memory Bound はまだ 15.9% 残っている
   (主に tags / entries 列の transfer)。同一 op 内で複数 op を amortize できれば
   tags 列の MESI ping-pong も削れる。1-day では収まらない設計、別 design doc 先行。
2. **A2 layout (writer / reader 分離)** — c16s で reader 側も改善方向だったので、
   writer/reader 分離による gain は薄い見込み。優先度低。
3. **shard-affinity (per-shard worker)** — moderate Zipf 帯の保険札としてのみ。
   本稿 hot-key workload では効かない既知。

senba lib への取り込みは別タスク。c16s は experimental 採用 = 「次の設計の base」
として扱い、lib 化は perf-gate (`sieve_cache_perf.rs`) と oracle が通る適合形まで
引き上げてから検討する。
