## TL;DR

c13s は **c12s 不採用 (SIEVE 等価性破壊) を踏まえた再設計**。継承面:

- **base**: senba::Cache lineage (shift-on-evict + 単一 `len` field) — SIEVE 等価が構造的に保証される。c12s の install-at-evicted-pos は捨てる。
- **reader**: c11s 由来 (visited 分離 + conditional `fetch_or` + AVX2 find)。
- **writer Path A** (= 既存 key の value 更新): c12s 由来の **CAS slot claim** + **VERSION bit** で reader cycle 検知。Path A は eviction を起こさないので SIEVE state machine に何も触らず、lock-free でも順序保持。
- **writer Path B/C** (= warmup install / evict + shift): writer Mutex 配下、senba::Cache 流の shift-on-evict。

採否判定の主軸 (read-heavy 95/5 zipf 16T):

| workload | c8 | c11s | c12s | **c13s** | c13s/c8 | c13s/c11s | HR (c13s vs orig) |
|---|---:|---:|---:|---:|---:|---:|---:|
| zipf-0.7 | 26.2 | 18.2 | 74.4† | **19.7** | 0.75x | 1.08x | ±0.002 ✓ |
| zipf-1.0 | 23.4 | 18.9 | 71.5† | **27.1** | **1.16x** | **1.43x** | ±0.005 ✓ |
| zipf-1.2 | 21.8 | 20.4 | 66.7† | **43.1** | **1.98x** | **2.11x** | ±0.004 ✓ |
| adversarial-hot | 19.9 | 27.2 | 85.8† | **70.3** | 3.54x | 2.59x | **-0.26 ✗ (※)** |
| uniform | 21.2 | 19.2 | 107.8† | **6.4** | **0.30x** | **0.34x** | ±0 ✓ |

† c12s は HR 破壊 (-3〜-44%) を伴うので throughput 比較対象から除外、参考値。

- **mixed wins on skewed read-heavy**: zipf-1.0/1.2 で c11s 比 +43〜111%、c8 比 +16〜98%。HR も sieve_orig と一致。**c11s plateau (writer Mutex 律速) を SIEVE 等価性を保ったまま部分的に解消した**。
- **uniform read-heavy で大規模 regression**: c11s 19.2 → c13s 6.4 (-67%)。原因は §3 で詳述するが、要約すると **uniform は write の 95% が "新規 key" → Path A は必ず CAS 失敗で MAX_RETRY 消費 → Path B/C escalate** という worst-case path を踏み続けるため。Path A の retry 自体がほぼ pure overhead として乗る。
- **read-heavy adversarial-hot で HR -0.26**: hot key に対する Path A cycle (CAS + VERSION flip) が高頻度で発生 → reader の seqlock validate が `t1 != t2` で false-miss → bench harness が miss として cnt。actual cache content には key=0 が常時居るので "本物の miss" ではないが、API 表面 (`get(&K) -> Option<V>`) としては Some を返さないので production 視点でも実害。
- **read-only / gim はほぼ c11s と等価**: ±5% 内。reader path に diff が無いので予想通り。
- **HR 同等性**: read-heavy adversarial-hot 以外の全セルで sieve_orig と ±0.005 一致。SIEVE 等価性は構造的に守られている。

**判定**: **条件付き採用候補**。「writer Mutex 律速を SIEVE 等価性を保ったまま緩和できる」という c12s で取り逃した目標は達成。ただし採用の前提として:

1. **uniform read-heavy regression は要修正**: Path A が新規 key で必ず失敗することを find_lockfree が早期検知して MAX_RETRY を回さずに即 Path B/C へ落ちる必要がある (§3.1 で具体策)。
2. **read-heavy adversarial-hot HR drop は API 仕様の再確認が必要**: 「seqlock false-miss を `Option<V>` で Some 化保証する」のか、「false-miss を許容して reader 側 retry を caller 責務にする」のか、senba::concurrent::Cache の上位仕様で決める案件 (§3.2)。

両方とも c13s 内で対応可能 (構造的問題ではない)。修正版を c13s' / c14s として再評価する筋。本変種は research artifact として残す価値あり (= "Path A だけ lock-free で SIEVE 等価が両立する" の最初の正例)。

## §1 背景 — c11s/c12s からの位置取り

| variant | writer | install pos | SIEVE 等価 | read-heavy zipf-1.0 16T |
|---|---|---|:---:|---:|
| c8 | Mutex | tail (shift) | ✓ | 21.1 Mops |
| c11s | Mutex (+ visited 分離 + conditional fetch_or) | tail (shift) | ✓ | 18.4 |
| **c12s** | lock-free CAS | **evicted pos** | **✗ (HR -33%)** | 68.0 |
| **c13s** | Path A: lock-free CAS / Path B,C: Mutex | tail (shift) | **✓** | 27.1 |

c12s 不採用理由 (前報告 §1) は「install-at-evicted-pos が SIEVE state machine を別物に変質させる」(= 新 entry が次の sweep で即 evict されやすい)。c13s はこの問題を **「lock-free 化を Path A (= 既存 key value update、eviction を起こさない経路) に限定」** することで回避する。Path A は SIEVE state machine を一切触らない (tag の HASH/ID/LIVE は元値、visited を SET するだけ) ので、SIEVE 等価性が構造的に保証される。

VERSION bit (0x4000) は c13s が新導入。Path A の CAS sentinel = EMPTY → write → 同 HASH/ID/LIVE で復帰、という cycle で reader の `t1 == t2` seqlock 検査が cycle を見逃すのを防ぐため、Path A ごとに VERSION を flip して全 16 bit が変化することを保証する (`SCAN_MASK` は VERSION を除外、find は VERSION 値に関わらず一致)。

#### read-heavy zipf-1.0 16T scaling

| T | c8 | c11s | c12s | **c13s** |
|---:|---:|---:|---:|---:|
| 1 | 14.2 | 16.1 | 18.6 | **16.5** |

c11s は 4T で plateau し、8T/16T で writer Mutex contention に呑まれて減速 (36.4 → 26.3 → 18.9)。c13s は同じ plateau を **8T/16T で再上昇に転じる** (39.6 → 36.6 → 27.1)。Path A (= zipf hot key への value 更新) を lock-free 化したことで hot key write contention が消え、scaling shape が改善。ただし c12s の linear scaling (42 → 60 → 71) には届かない: c13s は Path B/C (= 新 key install) が依然 Mutex 配下で、5% write の中の "新 key" 比率分が plateau する。

zipf-1.2 (より高 skew) では Path A 比率が更に高まり、c13s 43.1 Mops まで伸びて c11s 比 +111%、c8 比 +98%。

### 2.3 uniform read-heavy regression (要対処)

| T | c8 | c11s | c12s | **c13s** |
|---:|---:|---:|---:|---:|
| 1 | 48.8 | 54.3 | 66.4 | **19.4** |
| 2 | 42.6 | 50.9 | 74.2 | **13.6** |
| 4 | 44.3 | 37.2 | 85.7 | **8.2** |
| 8 | 29.0 | 23.8 | 97.7 | **6.8** |
| 16 | 21.2 | 19.2 | 107.8 | **6.4** |

uniform は keys=100k から 95% read / 5% write、write 側の key は random (= ほぼ全部新規 key)。

**root cause**: c13s の Path A は find_lockfree で「該当 key が cache 内に居るか」を探すが、uniform の write key は cache 内に居ない (cap=64、key space=100k)。`find_lockfree_for_path_a` は

1. `tags[0..len]` を scan して needle 一致 tag を探す
2. 一致したら entries[id] を ManuallyDrop で snapshot
3. seqlock validate (t1 == t2)
4. K::eq で key 一致確認

を全 64 slot に対して行う。100k key space + 8 bit hash → false positive (= needle 一致するが key 不一致) が

```
P(false positive per slot) ≈ 1/256 = 0.4%
expected scans per write ≈ 64 × 0.4% = 0.26 (期待) だが zipf でない uniform 分布で
peak は遥かに高く、64 slot 全 scan が常時走る。
```

加えて Path A は MAX_RETRY=4 を消費してから escalate するので、新 key write 1 回ごとに **4 回 find_lockfree が空振り → Path B/C** が確定動作。これが pure overhead。

更に Path B/C 内の `writer_find` も Path A 並行を考慮した spin-loop (EMPTY 検出時に hint::spin_loop) を持ち、tag を **per-slot ManuallyDrop::read + spin** という形で c11s より重い処理になっている。

#### §3.1 修正案 (c13s' / c14s)

write 操作 (insert) が `key not present` で Path A 経由を回避する道:

- (a) **Path A を 1 回だけ試行** (MAX_RETRY=1): find_lockfree が見つからなければ即 escalate。retry コストが消える。CAS 失敗 (= 並行 writer の Path A or Path B/C 進行中) は escalate して Mutex 順番待ちでも fairness で吸収される。`uniform read-heavy 16T` だけで考えると `c11s + Path A 1 回トライの overhead` ≒ c11s ± 数%。
- (b) **find_lockfree を AVX2 化**: c11s reader と同形にする。現状 scalar (Path A 用 find は AVX2 化していない、c12s が scalar だったのを踏襲)。1 retry でもスキャン本体が AVX2 で 16 lane / chunk になれば 64 slot scan は 4 ループ → 数十 ns。
- (c) **bloom filter / writer-side Cuckoo prefilter**: API として Cache に "presence hint" を持たせる、senba 仕様越境のため後手。

(a) + (b) で uniform read-heavy regression は c11s ± 5% 圏に収まる見込み。Path A 採用条件付きの本領 (skewed zipf read-heavy) は維持される (zipf hot key は CAS が初回成功する蓋然性が高い)。

### 2.4 read-heavy adversarial-hot HR drop

| T | c11s HR | c13s HR | delta |
|---:|---:|---:|---:|
| 1 | 0.950 | 0.950 | 0.000 |
| 2 | 0.933 | 0.937 | +0.004 |
| 4 | 0.897 | 0.851 | -0.046 |
| 8 | 0.863 | **0.649** | **-0.214** |
| 16 | 0.865 | **0.606** | **-0.259** |

(`docs/reports/data/2026-05-08-c13s-hr-read-heavy.png`)

adversarial-hot は key=1、95% read / 5% write すべて同じ key=0 に集中。

**root cause**: 16T で全 thread が key=0 への Path A を高頻度で叩く → CAS-EMPTY → write → store-back のサイクルが per-pos で高頻度に発生。reader の `try_candidate` は

```rust
let t1 = self.tags[pos].load(Acquire);
// ↓ 並行 Path A が CAS で EMPTY を入れる
let buf = ManuallyDrop::new(ptr::read(entries[id]));
let t2 = self.tags[pos].load(Acquire);
if t1 != t2 || (t2 & LIVE) == 0 { return None; }  // ← false-miss
```

の seqlock dance で、cycle 中の任意点で `t1` (= LIVE) と `t2` (= EMPTY) を観測すると None を返す。bench harness はこれを miss として count。

**実際には key=0 はずっと cache 内に居る**ので "cache content miss" ではない (`§2.5 invariants` で確認)。しかし `Cache::get` API が `Some(V)` を返さなかったので caller 視点では取りこぼし。

これは **c12s では起きなかった** (c12s は writer の VERSION cycle が無く、tag 復帰時の seqlock 検出は別経路だった)。c13s 固有の API observability 問題。

#### §3.2 対処方針

3 案:

- (A) **reader 側 retry**: try_candidate の `t1 != t2` 検出時に caller スコープで bounded retry (例: 4 回)。read-heavy 系で latency が +O(retry) するが false-miss は減る。ただし true miss (= 別 key 撃ち中) と区別できないので最大数を絞る必要。
- (B) **API 仕様で false-miss を許容**: `get` が稀に Some(V) を返さなくても caller (= `senba::concurrent::Cache` の上位) で `get_or_insert_with` 系で吸収。production 用途では recompute コストとのトレードオフ次第、adversarial-hot のように同 key を毎 op 撃つ workload は実環境では稀。
- (C) **VERSION cycle を無くす**: Path A を「seqlock cycle 不要」な形に再設計。例えば entries[id] の更新を CAS で行い tag を一切 touch しない (entries arena 側に version word を持たせる) → tag は LIVE 維持、reader の seqlock は trigger しない。実装複雑度大、別 variant (c14a) として要実験。

senba 仕様を再確認した上で (A) bounded retry が最低コスト。本変種は (A) 未実装で評価。

### 2.5 並行不変条件 (concurrent_invariants_under_zipf test)

`research/src/experimental/sieve_c13s.rs::tests::concurrent_invariants_under_zipf` で 4T / 50k op / cap=256 / 8 shards / zipf-1.0 で:

- `len <= cap` 保持
- shard ごとの live_ids に重複なし (id reuse 衝突なし)
- 各 shard の live_count == len
- 全 key の get 結果が破壊されていない (k で insert したら k に近い値が返る)

を確認。c13s implementation は Send + Sync soundness と SIEVE structural invariants を守る。

## §3 修正案まとめ — c14s の輪郭

c13s の不採用要因 2 点 (uniform read-heavy regression / adversarial-hot HR drop) は構造的問題ではなく実装 tunable で解消可能:

| 修正 | 想定効果 | 実装規模 |
|---|---|---|
| Path A MAX_RETRY=1 | uniform read-heavy regression を c11s ± 5% 圏に | trivial |
| find_lockfree AVX2 化 | uniform read-heavy regression 更に縮小 | medium (c11s reader を流用) |
| reader bounded retry | adversarial-hot HR drop を 0.85+ 復元 | small |

3 点まとめて c14s として実装、**c11s ± 5% を全軸で維持しつつ skewed zipf read-heavy で c13s 同等のゲイン** を狙うのが次手。並行で c11s vs c11sw / c11w (writer 経路改善 with Mutex) という方向もあり、どちらも senba::concurrent::Cache 昇格候補として正面比較する価値がある。

## §4 結論

c13s は **「SIEVE 等価性を保ったまま hot key write contention を緩和する」** という c12s で取り逃した目標を、Path A だけを lock-free 化することで初めて成立させた変種。

- 採否判定の主軸 read-heavy zipf-1.0 16T で c11s 比 +43%、c8 比 +16%、HR 完全一致。zipf-1.2 16T で更に c11s 比 +111%。
- 一方 uniform read-heavy で c11s 比 -67% という serious regression を残す (Path A retry 空振り問題)。adversarial-hot read-heavy で HR -0.26 (reader seqlock false-miss API observability 問題)。
- 両方とも c13s 内 tunable で対処可能 (§3)、構造的不採用ではない。

**senba::concurrent::Cache 昇格候補としては保留**。c14s (= c13s + Path A MAX_RETRY=1 + find_lockfree AVX2 + reader bounded retry) で再評価し、c11s + Path A の効果を切り分けて評価するのが筋。

本変種は `senba_research::experimental::sieve_c13s` に保持、後続の c14s / c11sw 比較における **「Path A lock-free 単独の効果」** の baseline として活用する。
