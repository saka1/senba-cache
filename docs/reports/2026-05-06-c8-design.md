# 2026-05-06 sieve_c8 — j8 並行版 設計 + 第一手 smoke

## 目的

`sieve_j8` (set-associative SIEVE, inline 20 B/cap, Twitter trace で orig 比 −9〜−20 ns/op)
を **マルチスレッド対応**させる初版 `sieve_c8` を起こした。SIEVE は hit path が
visited bit のセットだけで構造書き換えゼロ、という NSDI'24 paper §6 の
「lock-friendly」性質を直接活かす並行設計を作る。

## 方針 (事前合意)

- **read path lock-free + write path per-shard Mutex (parking_lot)**
- **seqlock-via-tag**: j8 が tag (u16) 内に id (6bit) + hash (8bit) + LIVE + VISITED を
  詰めて「slot に何が起きても tag が変わる」性質を持っているのを、そのまま
  seqlock の sequence number 兼 locator として再利用する。reader は tag 1 回読み →
  `entries[id]` raw read → tag 再 load (re-validate)、t1 == t2 && LIVE のみ採用。
- API: `&self` get/insert、`get` は `Option<V>` (V を Copy で copy out)。
- 値型制約: **`K, V: Copy`**。一般化は roadmap (c8a: Arc<V> wrap、c8e: epoch GC)。
- ハーネス: `std::thread::scope` + `std::sync::Barrier` 自作、`bench_concurrent` 独立 bin。

## 実装上の主な差分 (j8 → c8)

| j8 | c8 |
|---|---|
| `Vec<u16>` | `Box<[AtomicU16]>` |
| `Vec<MaybeUninit<Entry>>` | `UnsafeCell<Box<[MaybeUninit<Entry>]>>` |
| `tail: usize`, `len: usize` | `AtomicUsize` (reader 視認) |
| `hand: usize` | `Mutex<WriterState>` 配下 (writer のみ) |
| `&mut self` API | `&self` API、writer は `Mutex` 取得で直列化 |
| `find_scalar` | tag access を `load(Acquire)` + 候補ごと seqlock dance |
| `find_avx2` | SIMD scan は best-effort filter、候補ごと scalar 用 seqlock dance を流用 |
| existing-key update: 直接 `e.value = v` | EMPTY 経由で tag を確実に bump し、reader の torn read を seqlock 検出させる |

writer プロトコル (Mutex 配下):

- **evict at pos**: `tags[pos].store(EMPTY, Release)` → `fence(Release)` →
  `entries[id].assume_init_read()` → drop。tag 無効化を entries 操作に先行。
- **install at pos**: `entries[id].write(new)` → `tags[pos].store(LIVE | id | hash, Release)`。
  entries 書き込みを tag 公開に先行。
- **existing key update**: `tags[pos].store(EMPTY, Release)` → `entries[id].write(new)` →
  `tags[pos].store(old | VISITED, Release)`。一度 EMPTY を経由することで reader が
  「t1 = old, read entries, t2 = old」で torn value を採用する経路を断つ。

## 形式的な健全性ギャップ (明示)

reader の `entries[id]` raw read は writer の同 slot mutate と race するので
Rust 抽象機械上は **data race UB**。本実装は次の組み合わせで実機 x86_64 上
well-defined を主張する:

1. `K: Copy + V: Copy` で torn read が後続 Drop / Clone に伝播しない
2. `AtomicU16` の Acquire load / Release store で seqlock fence、候補 1 件ごと
   tag 一致を確認
3. writer は tag 無効化 → entries 書き換え → 新 tag 公開の順序を厳守

miri はこの racy read を当然にUB として拾うので、並行 test (`concurrent_invariants_under_zipf`)
は `#[cfg(not(miri))]` で抑制。1 shard 同期テスト群と j8/orig oracle 一致テストは
miri でも回せる。

将来形 (c8a / c8e) ではこのギャップを構造的に埋める。

## SIMD scan の扱い (c8 内)

ユーザー判断で **AVX2 path 搭載**。`AtomicU16` 配列を `*const u16` cast で
`_mm256_loadu_si256` する path は Rust 抽象機械上は灰色 (atomic と non-atomic 混在
access) だが、x86_64 で aligned u16 load は HW アトミック、SIMD load は 16 個の
HW atomic load が同一 cache line から取られたのと等価な ASM になる。SIMD scan は
**best-effort filter** として位置付け、候補 1 件ごとに scalar 用 seqlock dance を
流用して memory ordering を担保する設計にした。

### 生成 ASM の確認 (合法性検証)

`cargo rustc --release --bin bench_concurrent -- --emit=asm` でダンプし
`Shard::find_get_avx2` を目視:

- SIMD load: `vpand ymm2, ymm1, ymmword ptr [r11 + 2*r14]` (memory-fold した
  256-bit aligned load)
- 比較 / 抽出: `vpcmpeqw / vpmovmskb` 標準
- BLSR ×2: `blsr ecx, ebp; blsr ebp, ecx` (BMI1 ×2 で 1 候補ペア消費)
- tzcnt: `tzcnt ecx, ebp` (最下位 set bit を bit 番号として取得)
- 候補 tag t1 再 load: `movzx esi, word ptr [rsi + 2*rcx]` (Acquire load = 通常 mov)
- entries[id] read: `mov r15, qword ptr [rsi + r8]; mov r8, qword ptr [rsi + r8 + 8]`
  (Entry<u64,u64> = 16B を 2 mov で copy)
- 候補 tag t2 再 load: `movzx esi, word ptr [rsi + 2*rcx]`
- validate: `cmp si, r12w; jne; test si, si; jns` (t1==t2 と LIVE bit 確認)
- key compare: `cmp r15, rbx; jne`
- visited bit set: `lock or word ptr [rax + 2*rcx], 16384` (atomic OR with VISITED)

すべて合法 x86_64 (AVX2 + BMI1)、suspicious な split load なし。seqlock dance も
asm レベルで明示されている。

## 実装中に発見・修正したバグ

### Phantom non-empty tag (= LIVE 落ち + VISITED 立ち)

初版で `concurrent_invariants_under_zipf` が flakily に compact assertion で死んだ
(`debug_assert_eq!(len, write)` に left=31, right=32)。

**原因**: reader の `tags[i].fetch_or(VISITED, Relaxed)` が、writer の evict 直後
(= `tags[i] = EMPTY`) に発火するレース。reader は seqlock の re-validate (t2) を
しているが、re-validate と `fetch_or` の間に writer が EMPTY を書き込むと、
reader の fetch_or が `0 | VISITED = 0x4000` (LIVE 無し・VISITED のみ) という
phantom 値を残してしまう。writer 側の `t != EMPTY` 判定が `0x4000` を「非空」と
誤って数え、compact の `len == write` が崩れる。

**修正**: writer の判定を `t != EMPTY` から **`t & LIVE != 0`** に統一。
phantom 値は通りすがりに `tags[i].store(EMPTY, Release)` で正規化する。
変更点: `writer_scan_evict`, `writer_first_live`, `writer_compact`。

修正後 `concurrent_invariants_under_zipf` を 20/20 連続 pass 確認。

## 第一手 smoke

```
$ ./target/release/bench_concurrent --cap 512 --threads 4 --skew 1.0 \
    --keys 100000 --ops 4000000 --warmup 200000 --trials 3
```

| trial | aggregate Mops/s | hit ratio | p50 chunk ns | p99 chunk ns | thread CV |
|---|---|---|---|---|---|
| 0 | 17.47 | 0.546 | 209 | 459 | 0.006 |
| 1 | 17.83 | 0.546 | 208 | 431 | 0.004 |
| 2 | 18.35 | 0.546 | 207 | 343 | 0.003 |

### スレッドスケーリング (cap=512, ops=4M, warmup=200k)

| skew | 1T Mops | 2T Mops | 4T Mops | 4T/1T | 1T p50 | 4T p50 |
|---|---|---|---|---|---|---|
| 1.0 | 10.29 | 14.00 | 19.19 | 1.86× | 92 ns | 194 ns |
| 1.2 | 15.62 | 20.57 | 30.81 | 1.97× | 61 ns | 122 ns |

観察:

- **CV ほぼゼロ (≤ 0.04)** — 全 thread が同じ rate で進んでおり、Mutex 競合は
  実用的に観測されない。SIEVE の hit path 構造的優位 (lock-free read) が
  そのまま現れている。
- **1T overhead vs j8** : 1T で c8 ≈ 92 ns/op (skew=1.0)、j8 single-thread の
  zipf=1.0 / cap=4096 / per_shard=64 は同条件で 50 ns/op 帯。差 +42 ns は
  事前見積もり (+10〜20 ns) より大きい。内訳推定: parking_lot lock+unlock
  uncontended ~12 ns + AtomicU16 fetch_or ~10 ns + seqlock dance (extra Acquire
  load ×2 + read_volatile entries) ~15 ns + Box<[AtomicU16]> 経由のインデクス
  オーバーヘッド ~5 ns。**memfair の見直しが必要**。
- **skew=1.2 で per-op 速い**: hot key 集中で hit ratio が 0.81 に上がり、
  miss path (Mutex 必要) の頻度が減るため。SIEVE 並行化の "得意ワークロード"
  傾向。
- **scaling efficiency**: 4T で 1.86〜1.97× (理想 4× の半分以下)。これは
  threads=4 にしても **shard 数が 8 のままで insert は per-shard 直列**、
  さらに Zipf hot key (k=0) が常に同じ shard に当たるため。**SHARDS=32〜64
  に上げ、per_shard を MAX_PER_SHARD=64 内に収める** sweep を次の本ベンチで
  取りたい。

## 次手候補

1. **SHARDS=32 / 64 sweep**: 現状 `bench_concurrent` の SHARDS は 8 ハードコード。
   bench.rs と同様に `c8_n32` `c8_n64` 変種を追加して per-shard contention の
   解像度を上げる。
2. **threads × cap × skew Pareto sweep**: skew ∈ {0.8, 1.0, 1.2}, threads ∈
   {1, 2, 4, 8, 16}, cap ∈ {512, 1024, 2048} (SHARDS=64 で per_shard=8〜32) で
   aggregate throughput と HR を取り、Mutex 飽和点を地図化。
3. **ベースライン比較**: 現状の `mini_moka` / `moka` を bench_concurrent でも
   叩けるよう adapter を追加し、`<u64, u64>` 並列環境で W-TinyLFU 系との
   throughput / HR / p99 を直接比較する。
4. **memfair 観点**: c8 inline footprint は j8 (20 B/cap) + AtomicU16 wrap +
   parking_lot Mutex (per shard ~32 B 程度) + UnsafeCell オーバーヘッド。
   実測でどの程度膨らむか別レポートで定量化。
5. **c8a (Arc<V> wrap) 着手判断**: 「Copy 制約のため `<u64, String>` 等の
   workload を測れない」がボトルネックになる時点で着手。
