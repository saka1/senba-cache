## 0. TL;DR

r1 の 2 弱点を別 routing rule で攻める 3 variant + 等価仮説のコントロールを並べて
1 sweep で測る:

| variant | routing rule | 攻める弱点 | 主仮説 |
|---|---|---|---|
| **r2h** (hash control) | `way = mix(tls_id, hash(K)) mod ways` | (cluster019 で r2h << r1 の予測検証) | thread affinity 寄与の切り分け |
| **r2s** (soft spillover) | writer's way が soft cap `cap_per_way × α` を超えたら `(t+1) mod ways` に spill。α ∈ {1.25, 1.5, 2.0} を const generic | (B) per-way cliff | α=1.5 で cluster006 HR drop 半減、cluster019 Mops 80%+ 維持 |
| **r2p** (power-of-2) | insert 時に writer's way と random way を比較、`len` 小の方に挿入 | (B) per-way cliff + (A) duplicate 軽減 | r2s より affinity 減、HR drop も均等化 |
| (control) **senba::Cache(shards=512)** | hash-only routing | — | r2h ≈ senba(shards=512) を測り「ways は shard 細分化と等価か」を判定 |

WAYS=1 / α=1.0 で全 variant が r1 / sieve_orig と bit-equivalent に動くことを oracle で
担保。主成果物は `figures/summary.md` 一本 (仮説 H1–H7 判定 + 不採用領域 cell の variant
横並び表 + pareto 1 枚) で、`r1-vs-moka-cap-sweep` のような per-cell 図量産はしない
(raw CSV を git commit、必要時 plot.py 再走で再現)。

## 1. r1 弱点の構造分解

r1 で HR drop が起きる機序は 2 系統あり、どちらに焦点を当てるかが routing rule の
選択軸になる。

### 1.1 (A) 重複蓄積

- 機序: thread t が K を way(t) に insert → 後で evict → thread u が再 miss → way(u)
  に insert。時間積分で hot K が複数 way に同居
- 典型 cell: `cluster006` (全 thread が同じ hot set を踏む、HR drop 27pp@w=8)
- 解法カテゴリ: cross-way visibility (canonical fallback、frequency tier)、または
  ways を増やさずに spillover で済ます

### 1.2 (B) per-way 実効 cap cliff

- 機序: thread t の最近 insert で way(t) が圧迫、hot K が capacity 由来で早期 evict。
  `cap/ways` が hot working set を下回ると HR が崩壊
- 典型 cell: ARC OLTP cap=4000 (HR 0.44 → 0.23 / w=8、cap-fits 強)、DS1 cap=8M
  (per-way 1M で working set に届かず HR 3%)
- 解法カテゴリ: capacity-aware spillover (soft cap で隣接 way に逃がす)、または
  per-way 内で load balance (power-of-2)

### 1.3 r1 vs r2 vs senba::Cache の対立軸

```
                        ┌── HR floor 低 (DS1/MergeP)      → 全 variant 同等
                        │
            ┌─ HR-tolerant ─┤
            │           │
            │           └── 中位 skew long-tail (cluster019) → r1 が勝つ
   trace ─ ┤
            │           ┌── shared-hot 極端 (cluster006)    → r1 落ちる、senba(細分) 同等
            │           │
            └─ HR-cliff ─┤
                        └── cap-fits (ARC OLTP)            → r1 落ちる、cap-tune が要る
```

r2s は HR-cliff 帯で r1 から HR を取り戻すことが目的。r2p は r2s の load balance 観点
での比較対象。r2h は「routing rule で thread を捨てると senba(細分) に縮退するか」を
直接観測。

## 2. 採用しない他候補

`2026-05-13` 議論で挙がった候補のうち r2 には乗せない:

- **S3 read-side migration** (read 時に K を reader's way に移動): trigger 閾値 N の
  チューニング軸が増え、cluster019 のような既に r1 で勝てる workload に対して trigger
  が走らない / 走り過ぎ どちらも noise 源になる。先に r2s/r2p で cliff 緩和を取り、
  必要なら r3 系で再検討
- **S4 canonical-way fallback** (`way = hash(K) mod ways` と writer's way の二重管理):
  miss path で全 way probe が要り、r1 の本質利得 (probe 省略) を一部失う。r2s で
  (A) 重複蓄積を緩和できない場合に re-open
- **S5 frequency tier** (Caffeine 風の hot tier + cold tier 2 段構成): 実装重く、
  promotion threshold / tier 比率のチューニング軸が爆発。r2 系で取れない HR drop が
  残った時の最終手段
- **adaptive WAYS**: 切替判断に entropy 推定が要るが、ユーザ討議で **明確な切替
  ルールが無い** ことを確認 (cluster006 vs cluster019 が skew 単独で判別不能 / sketch
  オーバーヘッドが r1 利得を打ち消す / shadow probe 設計が膨らむ)。当面棚上げ

## 3. 設計詳細

### 3.1 r2h (hash-routed control)

#### routing rule

```rust
#[inline]
fn way_of(tls_id: u32, hash: u64, way_mask: usize) -> usize {
    // tls_id を hash と独立な軸として混ぜる。両軸を XOR で重ねるだけだと
    // identity 偏りが起きるので、tls_id 側を golden ratio で multiplicative
    // mix してから XOR。
    let t = (tls_id as u64).wrapping_mul(0x9E37_79B9_7F4A_7C15);
    ((t ^ hash) as usize) & way_mask
}
```

- `way_mask = ways - 1` (power-of-2 ways 前提)
- `set_mask` (shard 内の set 選択) は r1 と同様 `hash` の上位ビットを使う
- thread t の insert は K に依存して全 way に **uniformly に散る** (期待値)
- 同一 K でも thread が変われば way が変わる → duplicate 化が r1 と同等

#### 期待挙動

- **cluster019 で r1@8 +77% → r2h@8 では +0〜30% 程度に縮退** (predicted)
  - writer state line が全 way に分散、bouncing が r1@1 (= c17s) 相当に戻る
  - 残るのは "shard 数が増えた効果" のみ
- **`senba::Cache(shards=512)` (= 64 shards × 8 ways) と Mops で差 ≤5%** (predicted)
  - hash 軸で thread 軸を消した routing は事実上 shard 細分化と等価。lock 粒度は
    senba::Cache の shard-level mutex に対し r2h は shard 内 way-level なので構造的
    に同じ
- **cluster006 / ARC OLTP の HR drop は r1 と同等** (predicted)
  - duplicate 発生機序は thread 軸ではなく "evict → 別 thread で再 insert" の時間
    依存性なので routing rule で大きく変わらない

#### oracle 等価条件

- ways=1 のとき `way_mask=0` で常に way=0、tls_id 依存性が消え c17s と bit-equivalent
- WAYS≥2 では r1 と evicted-key 順が違うので oracle PASS は ways=1 のみ

### 3.2 r2s (soft spillover)

#### 読み出し側の構造制約 (r2s / r2p 共通)

spillover や power-of-2 抽選で **insert 先 way が writer's way 以外になり得る**ため、
read path は set 内 W way 全てを probe する必要がある。これをやらないと「writer の
spill insert が後続 read で見えない (lost insert 病)」が発生し、HR drop 緩和どころか
HR がさらに崩壊する。

- `get` / `contains_key`: 同じ set 内 W shard を順に probe、最初に LIVE が見つかれば
  返す。全 way miss で None
- `insert`: §3.2 / §3.3 の routing rule で target way 選定後、ターゲット shard に insert

reader 側 probe は **shard 単位の lock-free fast path をそのまま W 回呼ぶ**だけなので
mutex 取得は伴わない。L1/L2 line touch が W 倍に増える点が perf cost で、これが r1 vs
r2s/r2p の trade-off の主軸 (affinity 利得 = writer bouncing 解消、対 cost = read 側
multi-probe)。

`r1-design.md` §2.3 で "Soft scheme: read は set 全 probe" と明記されていたが、r1
実装は Hard (single-way probe) で済ませていた。r2s/r2p は r1 と異なり真の Soft scheme
となる。

#### routing rule

```rust
fn insert_target_way(tls_id: u32, ways: usize, lens: &[AtomicUsize], soft_cap: usize)
    -> usize
{
    let writer_way = (tls_id as usize) & (ways - 1);
    if lens[writer_way].load(Relaxed) < soft_cap {
        return writer_way;
    }
    // spill: writer's way が soft_cap 超 → 隣接 way を順に試す
    for step in 1..ways {
        let w = (writer_way + step) & (ways - 1);
        if lens[w].load(Relaxed) < soft_cap {
            return w;
        }
    }
    // 全 way が soft_cap 超 → writer's way に戻す (通常 eviction)
    writer_way
}
```

- `soft_cap = (cap_per_way × α).floor()` で const generic `α ∈ {1.25, 1.5, 2.0}` を 3 値
- `lens` は既存 r1 の per-way `len: AtomicUsize` を流用
- spill 順序は `(writer_way + 1) mod ways` 固定 (random spill にすると r2p に近づくので
  差別化のため固定)

#### 期待挙動

- **affinity 半保持**: α=1.5 なら writer's way が cap × 1.5 まで埋まるまでは r1 hard と
  完全同一動作。普段は bouncing-free を維持
- **cliff 緩和**: cap-fits 帯 (ARC OLTP) で writer's way が早期に soft_cap 到達 →
  spillover で隣接 way に逃がす → 全体的に均等化、HR drop が r1 から半減 (predicted)
- **affinity sacrifice の対価**: spillover が頻発する workload (= cluster006 のように
  全 thread が同じ key 帯を踏む) では結局 ways 横断で writer state line が触られ、
  bouncing が部分復活
- **cluster019 のような r1 sweet spot**: spillover が trigger されない (writer's way
  に余裕) → r1 と同等 Mops 維持 (predicted)

#### const generic α の sweep 設計

```rust
pub struct SieveR2s<K, V, S, const SHARDS: usize, const ALPHA_NUM: u32, const ALPHA_DEN: u32> {
    // ...
}
// soft_cap = cap_per_way * ALPHA_NUM / ALPHA_DEN
// α=1.0 → (1, 1) で r1 hard と等価 (oracle 用)
// α=1.25 → (5, 4)
// α=1.5  → (3, 2)
// α=2.0  → (2, 1)
```

bench_concurrent.rs の dispatch arm で `--variant r2s --alpha 1.5` のような CLI flag を
parse して `(ALPHA_NUM, ALPHA_DEN)` に解決。

#### oracle 等価条件

- ways=1 のとき writer_way=0 固定で spillover 発生せず c17s と bit-equivalent
- WAYS≥2 + α=1.0 でも spillover が常に trigger される (soft_cap=cap_per_way なので
  insert 直後に等しくなる) ので r1 hard とは bit-equivalent にならない可能性あり →
  oracle は ways=1 のみで PASS、α≥1.0001 程度の閾値で r1 と diff することは仕様内

### 3.3 r2p (power-of-2)

#### routing rule

```rust
fn insert_target_way(tls_id: u32, hash: u64, ways: usize, lens: &[AtomicUsize])
    -> usize
{
    let writer_way = (tls_id as usize) & (ways - 1);
    // 2 候補抽選: writer's way と (writer's way XOR (hash mix lsbits))
    let alt = writer_way ^ (((hash >> 32) as usize) & (ways - 1));
    let lw = lens[writer_way].load(Relaxed);
    let la = lens[alt].load(Relaxed);
    if la < lw { alt } else { writer_way }
}
```

- 2 つ目の way 候補は `hash` から派生 (per-key 確定的、cluster006 のような全 thread が
  同じ K を見る workload で抽選結果が一致するよう lock-free)
- `lens[w].load(Relaxed)` で 2 way 分の load = 1 cache line 程度
- 古典 Mitzenmacher 2001 の "power of two choices" を SIEVE insert path に適用

#### 期待挙動

- **load balance**: ways 間の fill レベルが log log n 補正で均等化
- **affinity 半減**: 確率 ~1/2 で alt way へ migration、bouncing は r1 の半分程度に
  残る (predicted)
- **cluster006 で HR drop 緩和**: 全 thread が抽選で alt 側に行くと同一 K の duplicate
  が ways 全体に均等分布 → 重複の絶対数は r1 と同等だが分布が均等化、結果的に hot K
  が evict されにくくなる (predicted)

#### oracle 等価条件

- ways=1 のとき writer_way=0=alt 固定で抽選不発、c17s と bit-equivalent

## 6. 採用基準

`r1-design.md` §6.3 と同じ pareto 構造判定 + r2 固有の追加条件:

### 6.1 単独採用 (r2s / r2p を r1 と並べて lib opt-in 候補にする)

以下を **同時に** 満たす:

- r1 採用領域 (cluster019, MergeP) で r2 が r1 の **80%+ Mops** を維持
- r1 不採用領域 (cluster006, OLTP) で **HR drop が r1 から ≥ 10pp 改善**
- 全 cell で **c17s を Mops で同等以上**

### 6.2 部分採用 (workload hint API として残す)

- 特定 workload class (例: HR-tolerant long-tail のみ) で r1 を pareto dominate
- 他 class で r1 に劣後するが c17s より上

### 6.3 reject

- r2 全 variant が r1 を pareto dominate できず、かつ c17s に劣後する cell が
  全 variant で発生

### 6.4 r2h の特例判定

H1/H2 の検証用 control なので採否対象外。**結論を見て r1 設計の justification を
update** するのが主目的:

- H1 否定 (r2h ≈ r1) なら r1 の thread affinity 仮説が崩れる → 設計再考
- H2 肯定 (r2h ≈ senba(512)) なら r2h は単なる細分化と等価 → r1 系 lib 化判断は
  「ways 構造を残すべき定量根拠が無い」と整理される

## 7. 撤退条件

- H1 否定 + H3 否定 + H7 否定: r1 routing 設計が空転 + r2 spillover も効かず → r1 を
  捨てて senba::Cache(shards 細分) に戻る判断
- H1 肯定 + H3-H7 全否定: r1 は正当だが r2 で弱点が解けない → r1 を opt-in API で
  lib 化、shared-hot / cap-fits 帯はユーザに対し "推奨外" として doc 明記
- H1 肯定 + H3-H4 肯定 + H6-H7 半分肯定: **r2s 採用**、α=1.5 を default に lib opt-in
