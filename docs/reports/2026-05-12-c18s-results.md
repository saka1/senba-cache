## 0. TL;DR

c18s = c17s から `Entry::version` を別配列 `versions: [AtomicU32; 64]` に逃がし、Entry を
Slot16 に戻し、`path_c_epoch` を ShardHot から ReaderState block に移動した variant
(G2-α-2)。**仮説は完全に外れ、reject**。

| Step | 結果 | 判定 |
|---|---|---|
| 1 oracle 一致 (4 (skew, cap) × 200k op) | diff = 0 | **PASS** |
| 2 gate (a) adv-hot read-heavy 16T (skew 1.4) | c18s 133.84 / c16s 125.65 = **+6.5%** (5 trial) | **PASS (≥ +5%)** |
| 3 gate (b) skew=1.0 gim T=4 | c18s 33.15 / c16s 36.22 = **−8.5%** | **FAIL (要 ≥ −2%、c17s −4.7% より悪化)** |
| 4a thread sweep gim T=1〜16 | 全帯で c18s が c16s と c17s 両方を下回る (−1〜−10%) | (b) と整合 |
| 4b thread sweep adv-hot read-heavy T=1〜16 | c18s は c16s には勝つが、**c17s に対し T=4 で −1.5%、T=16 で −14.2%** | **(a) は薄勝、c17s 比で構造的劣化** |

**設計仮説 (entries Slot32 footprint が gate (b) regression の主因) は反証された**。
gate (b) で c17s より更に退行、adv-hot read-heavy では c17s に大きく劣後。entries half 化
の利得 (1024B 削減) を、versions 別配列化と path_c_epoch 分離による reader cache line
touch +2 のコストが上回った構造。

c18s は **採用しない、c17s 据え置き**。本稿は「naive な SoA 化が cache line transfer を
増やすパターン」を改善カタログに artifact として残す目的で記録する。

## 3. Step 2 — Mops AB gate (a): adv-hot read-heavy 16T

```bash
./target/release/bench_concurrent --variant c16s,c17s,c18s --threads 16 --shards 64 \
    --cap 4096 --keys 100000 --skew 1.4 --ops 40000000 --warmup 1000000 --trials 5 \
    --op-mix read-heavy
```

| variant | trial Mops | 平均 Mops | stdev | CV | HR |
|---|---|---:|---:|---:|---:|
| c16s | 128.06, 126.53, 123.32, 125.57, 124.79 | **125.65** | 1.55 | 0.012 | 0.9250 |
| c17s | 153.58, 157.01, 150.48, 160.10, 156.65 | **155.56** | 3.43 | 0.022 | 0.9279 |
| c18s | 126.42, 136.65, 131.13, 137.21, 137.80 | **133.84** | 4.34 | 0.032 | **0.9278** |

**c18s / c16s = 133.84 / 125.65 = 1.065 (+6.5%)** — gate (a) (≥ +5%) PASS。
**c18s / c17s = 133.84 / 155.56 = 0.860 (−14.0%)** — c17s に対し大幅退行。

HR は c18s も c17s と同水準 (0.9278 vs 0.9279) で改善構造は継承できている。

副次観察:
- c18s の adv-hot 利得 (+6.5%) は c17s の +23.8% (本日同条件再測) の 1/4。EMPTY-lane SIMD
  削除 + Path A 由来 false-miss 構造除去の効きはあるが、**versions 別配列の reader cache
  line touch +1 が大半を打ち消している**。
- per-thread CV は 0.022〜0.032 で c17s と同水準、scheduling 偏り無し。

## 5. Step 4 — Thread sweep

### 5.1 skew=1.0 gim (gate (b) 同シナリオの T sweep)

3 trial 平均 Mops:

| threads | c16s | c17s | c18s | c18s/c16s | c17s/c16s | c18s/c17s |
|---:|---:|---:|---:|---:|---:|---:|
| 1  | 11.31 | 11.52 | 11.19 | **−1.1%** | +1.9% | −2.9% |
| 2  | 19.82 | 19.58 | 18.69 | **−5.7%** | −1.2% | −4.5% |
| 4  | 35.37 | 34.08 | 33.80 | **−4.4%** | −3.7% | −0.8% |
| 8  | 52.86 | 49.42 | 47.78 | **−9.6%** | −6.5% | −3.3% |
| 16 | 61.46 | 59.80 | 57.04 | **−7.2%** | −2.7% | −4.6% |

観察:
- **全 T で c18s は c16s/c17s 両方を下回る**。HR 完全一致なので SIEVE 等価。
- T=1 単独で −1.1% (c16s 比) — single-thread でも reader cache line touch +1 cost は
  既に出ている。並列性が無くても α-2 は不利。
- T=8 で −9.6% (c16s 比) が最大、T=16 で僅かに回復 (−7.2%) — c17s と同じ shape (T=8 が
  L1d/L2 圧迫最大帯)、α-2 化はこのパターンを変えていない。

### 5.2 adv-hot read-heavy (gate (a) 同シナリオの T sweep)

3 trial 平均 Mops:

| threads | c16s | c17s | c18s | c18s/c16s | c17s/c16s | c18s/c17s |
|---:|---:|---:|---:|---:|---:|---:|
| 1  | 23.23 | 25.06 | 23.97 | +3.2% | +7.9% | **−4.3%** |
| 2  | 39.12 | 43.75 | 42.09 | +7.6% | +11.8% | **−3.8%** |
| 4  | 69.21 | 78.61 | 77.43 | +11.9% | +13.6% | **−1.5%** |
| 8  | 101.91 | 121.06 | 111.42 | +9.3% | +18.8% | **−8.0%** |
| 16 | 120.82 | 154.05 | 132.16 | +9.4% | +27.5% | **−14.2%** |

観察:
- c18s は c16s には全 T で勝つ (+3〜+12%) が、**c17s には全 T で負ける** (−1.5〜−14.2%)。
- c17s vs c16s の利得 (+7.9% → +27.5%) は T と共に super-linear で伸びる、
  c18s vs c16s の利得 (+3.2% → +9.4%) は **T=4 でピークアウトして以降 飽和**。
  versions cache line touch +1 が高 T 帯で利得を食い潰している構造。
- HR は 3 variant 全て c17s 同水準 (~0.928) を維持、c18s は HR 改善は継承できている
  (versions 別配列でも Path A 由来 false-miss 解消は同型で動く)。

## 7. なぜ仮説が外れたか — root cause 分析

設計時の思考は「entries arena 1024B 削減 (footprint 半減) > versions cache line +1
touch」だったが、実際には逆だった。原因分解:

### 7.1 c17s では version load が **ほぼ無料**だった

c17s の Entry layout:
```text
align(32):
  offset 0:  AtomicU32 version
  offset 4:  pad (4)
  offset 8:  K (u64)
  offset 16: V (u64)
  offset 24: pad (8)
```

reader が `try_candidate` 内で:
1. `entries[id].version.load` (tier 1) — Entry の 0..32 line を L1 に持ってくる
2. `ptr::read entries[id]` — 既に L1 に居るので 0 cost
3. `entries[id].version.load` (v2) — 同 line から 0 cost

→ **version 2 load は entries[id] line を運ぶ単一 cache line transfer に乗っていた**。
別 line ではない。「2 atomic load 追加」は MESI 的にも追加 transfer ゼロ。

c18s では versions[id] が別 line:
1. `reader_state.versions[id].load` — versions line (4 line のうち 1 line) を L1 に持ち込む
2. `ptr::read entries[id]` — entries[id] line (16 line のうち 1 line) を別 fetch
3. `reader_state.versions[id].load` — 同 versions line から 0 cost

→ **per hit cache line fetch が 1 → 2 に増えた**。entries arena half 化 (32 line → 16 line)
の global footprint 削減は、per-hit access pattern では効かない (1 hit = 1 entries
line touch は不変)。

### 7.2 path_c_epoch 移動の効きはゼロ以下

c17s で `path_c_epoch` は ShardHot 同居 → 既に reader が `len.load` で line を L1 に持って
いるので 0 cost。c18s で ReaderState の独立 line に逃がす → reader が path_c_epoch line
を別 fetch する必要が出た (1 line 追加)。Path A の visited fetch_or による invalidate も
gim では Path A 率が低いので元から問題ではなかった (c17s §11.2 の確認と整合)。

→ **エポック分離は純コスト**。adv-hot 高 T で「Path A 並行多」を期待した分離が逆に
T=16 で c17s -14% の主犯になった。

### 7.3 entries footprint 削減 (1024B) の利得は L3 透過率には乗らない

skew=1.0 gim でも L1d/L2 transfer の主役は **per-op の cache line miss 数** であって、
arena footprint 全体ではない。footprint 削減が効くのは「全 entries が L1d に乗るか
否か」の境界帯のみで、本実験 cap=4096 / cores=16 では各 shard = 64 entries × 16 B =
1024B、c17s でも 2048B、いずれも L1d 32KB に余裕で乗る範囲。

→ Slot32 → Slot16 の footprint 削減は **本実験では効かない構造**。
cap が大きくて per-shard が複数 L1d 占有する帯では効くかもしれないが、それは別 sweep
が要る (今回未検証)。

### 7.4 まとめ — naive SoA 化の典型的失敗パターン

「reader が touch する field を分離すれば writer 干渉が減る」という直感は、c17s のように
**writer が同 line を 1/N 確率でしか触らない場合**は false positive。c17s は version を
entries[id] と同居させることで:
- writer (Path A): 1 line W (entries[id])
- reader: 1 line R (entries[id]、version 含む)

を達成しており、これは **MESI 的にも cache transfer 数的にも最適** だった。c18s で
versions を別 line にした瞬間、reader と writer が touch する line 数の合計が増える。

## 8. 採否 — REJECT

### 8.1 採否ロジック

| 軸 | c18s 評価 |
|---|---|
| 正しさ (oracle 一致) | ✓ SIEVE 等価、`sieve_orig` と完全一致 |
| 主目的 (gate (b) 退行解消) | ✗ −8.5% で c17s −4.7% より更に悪化 |
| 副次目的 (gate (a) 維持) | △ +6.5% で gate は通過するが c17s 比 −14.0% |
| 設計の説明性 | ✓ failure mode は cache line transfer 数で完全説明可能 |

「c17s の gate (b) 退行を構造的に解消する」目的は **完全失敗**。lib 採用候補からも
外れる (c17s vs c16s と異なり c18s は c17s に支配される)。

c18s は **research artifact として残置**、コードは `research/src/experimental/sieve_c18s.rs`
に維持 (将来の SoA 関連検証の負例として参照、削除はしない)。

### 8.2 c17s 据え置きの判断

c17s は既に「条件付き採用 (G2-α-1 確定採用、`cseries-string-baseline` で V::Clone
soundness 確認済)」の状態。c18s が c17s を越えなかった以上、c17s が **adv-hot 読み主体
workload の選択肢として確定**、gate (b) 退行は構造として受け入れる (c17s + c16s の
workload 別 use-case map で運用)。

## 9. 後続 design space の再評価

c18s reject により、`improvement-ideas.md` の優先度マップを次のように更新したい
(本稿の §10 で improvement-ideas に反映):

### 9.1 単体シャード read 側で残る筋

1. **`len.load` 削除 (TOCTOU 検証込み)** — Path B の `entries[len]` 書込 → `tags[len]=LIVE`
   順なので、SIMD で LIVE 判定された tag は entry init 済 (TOCTOU セーフ)。`pos < len`
   フィルタが redundant な可能性。削れれば `len.load` 1 回 + per-candidate branch 1 個。
   c18s 失敗の教訓を踏まえ、これは **「touch する line 数を 1 減らす」** 方向で筋が良い
   (versions split は逆方向だった)。
2. **t1 二重 load の削減** — SIMD で一致した tag を再 load せず、`vpcompressd` /
   `vpermd` でレジスタ抽出する。HASH 9 bit で candidate 0〜1 個が大半なので効きは小〜中。
3. **AVX-512 V5** — cap=64 で 32-lane scan 1 chunk 化、scan 回数 4→2。ただし tags が
   L1-hot な現状は scan 自体が支配 cost ではないので効きは小。

### 9.2 単体シャード write 側で残る筋 (天井最大)

4. **D.4 G3-δ install-at-(hand+cap/2) / G3-ε free-list / D.3 writer batching** —
   Path B/C Mutex を構造的に外す。c18s が示した「single-shard read は c17s で天井」
   ならば、次の天井は write/Mutex 側にある。実装大、HR リスク有。

### 9.3 諦める筋

- **G2-α-2 系 SoA 派生** — c18s で示された通り naive な field split は逆効果。SoA する
  なら all-in (G2-β SoA state、tags も含めて全 array 分離 + AVX-512 統合) でない限り
  cache transfer が悪化する。β に行く価値の判断は別 task。
- **path_c_epoch 移動単独** — c18s 失敗の主因の 1 つは epoch 分離による cold line 追加。
  これは将来の variant でも繰り返さない。

## 11. 今回の learning を `improvement-ideas.md` にどう持ち込むか

`§D.1 G2-α` を「α-1 (entry 同居) を確定、α-2 (versions 別配列) は反証済 (`c18s-results`)」
と注記する。`§D.2 G2-β SoA state` の判断材料に「α-2 の失敗から、SoA は naive な部分
split では逆効果。all-in な SoA 化 (tags / versions / visited 全部) でないと意味がない
可能性が高い」を追加。

`§D.4 G3 writer 系` の優先度を上げる (read 側天井が c17s で確定したので、次の天井は
write 側 Mutex)。
