## ERRATUM (sweep 後追記)

§2.3 の reader bounded retry は **"true-miss と false-miss を区別する手段がない
ため上限を設ける"** という前提で書いたが、これは誤り。実装では `try_candidate`
の seqlock failure + scan 中の LIVE=0 lane 観測の 2 点で racing を検出でき、
**racing 観測時のみ retry / それ以外は 1 回 scan で None 確定** という形に
すべきだった。`MAX_READER_RETRY=4` を無条件適用した初版は uniform read-only 16T
で全 query が 4× scan を踏み、c11s 比 -75% の壊滅的 regression を出した。
詳細と修正は `2026-05-08-c14s-sweep.md` §1〜§2 を参照。設計書本文は元の意図を
記録するため当時のまま残す。

## TL;DR

c13s は「SIEVE 等価を保ったまま hot key write contention を部分的に lock-free 化」という
目標を初めて達成した。残課題は 2 つ:

| 課題 | c13s の数値 | 原因 |
|---|---|---|
| uniform read-heavy regression | c11s 比 -67% (16T) | try_path_a の find_lockfree が scalar 64 scan × 毎 insert |
| adversarial-hot HR drop | -0.26 (16T) | reader seqlock の false-miss (VERSION bit flip) |

どちらも構造的問題ではなく **実装 tunable** (c13s-sweep §3)。c14s はこの 2 点を:

1. **MAX_RETRY=1** — Path A を 1 回だけ試みる (CAS 失敗時即 Mutex escalate)
2. **find_lockfree AVX2 化** — scalar 64 scan → SIMD 4 chunk (c11s reader と同形)
3. **reader bounded retry** — seqlock false-miss 検出時に get() 内で最大 N 回再試行

の 3 点で対処する。変更はすべて c13s 内部に閉じており、SIEVE 等価性・API 表面・
shard 構造に影響しない。

**目標性能** (read-heavy 16T):

| workload | c11s | c13s | c14s 目標 |
|---|---:|---:|---:|
| uniform | 19.2 | 6.4 | ≥ 18 (c11s −5%) |
| zipf-1.0 | 18.9 | 27.1 | ≥ 25 (c13s 水準維持) |
| zipf-1.2 | 20.4 | 43.1 | ≥ 40 (c13s 水準維持) |
| adversarial-hot HR | 0.865 | 0.606 | ≥ 0.85 |

## §1 背景 — c13s の問題の構造的整理

### 1.1 uniform regression の根本原因

c13s の `insert(key, value)` は常に `try_path_a` を呼んでから `path_bc` に落ちる。

```
c13s insert:
  1. try_path_a:
       find_lockfree_for_path_a  ← scalar: tags[0..len] を 1 slot ずつ全 scan (64 iterations)
       → None (uniform では key が cache 内にいない) → Err
  2. path_bc (Mutex):
       writer_find                ← EMPTY spin 付きで再 scan
       → evict + install
```

c11s の同経路:

```
c11s insert:
  1. path_bc (Mutex):
       writer_find                ← 1 回だけ
       → evict + install
```

c13s は **write 1 回ごとに try_path_a の scan が純 overhead として乗る**。
scalar 64 iterations は ~100–200 ns 規模で、uniform 1T で既に -64% が出ている
(c13s-sweep §2.6)。MAX_RETRY=4 は CAS 失敗時の再試行数であり、uniform の None ケース
は最初の find で即 Err になるため MAX_RETRY の値は無関係。**ボトルネックは
find_lockfree_for_path_a の scan 本体のコスト**。

また path_bc 内の `writer_find` も c11s より重い (EMPTY spin を含む)。こちらは構造上
避けられないが、Path A との並行期間は短いので実測では小さい。

### 1.2 adversarial-hot HR drop の根本原因

c13s の false-miss メカニズム:

```
reader:
  t1 = tags[pos].load(Acquire)       ← 0x8001 (LIVE, VERSION=0)

  [ここで Path A が完走: EMPTY → store(0xC001, VERSION=1)]

  snapshot = entries[id]             ← entry は一貫 (Path A 完了後)
  t2 = tags[pos].load(Acquire)       ← 0xC001 (LIVE, VERSION=1)
  t1(0x8001) != t2(0xC001) → None   ← false-miss
```

VERSION bit は ABA 防止のために必要 (同一ハッシュ値の 2 回の Path A を reader が見逃すと
torn read になりうる)。したがって **VERSION flip 自体は消せない**。false-miss は
seqlock-via-tag の構造的帰結であり、reader 側でのリトライが唯一のフィールド内対処。

なお、「false-miss は我々が選んだアルゴリズムの問題であって本質的ではない」
という認識が正しい。tag に同期用途とコンテンツ用途を兼ねさせていることが干渉の源泉で、
entry 側 seqlock に分離すれば API 表面から false-miss を消せる (→ §7 c15 方向)。
c14s では構造変更コストを避け bounded retry で対処する。

## §2 変更点

### §2.1 find_lockfree AVX2 化 (主軸・uniform regression 修正)

`find_lockfree_for_path_a` を c11s の `find_get_avx2` と同形の SIMD scan に変える。

```
現状 (scalar):
  for pos in 0..len:
    t1 = tags[pos].load(Acquire)
    if (t1 & SCAN_MASK) != needle: continue
    ...seqlock + K::eq...

変更後 (AVX2):
  needle_v = _mm256_set1_epi16(needle)
  mask_v   = _mm256_set1_epi16(SCAN_MASK)
  for chunk in tags[..tags.len()].chunks(LANE):
    v      = _mm256_loadu_si256(chunk)
    cmp    = _mm256_cmpeq_epi16(_mm256_and_si256(v, mask_v), needle_v)
    bitmask = _mm256_movemask_epi8(cmp)
    for each lane in bitmask:
      pos = chunk_offset + lane
      if pos >= len: break
      ...seqlock + K::eq...  ← 候補のみ scalar 検証
```

64 slot のスキャン: 64 scalar iterations → 4 SIMD chunk (16 lane × 4 = 64)。

**miss ケース (uniform write: key が cache にいない) での効果が最大**:
候補が 0 なら seqlock 検証は一切走らず 4 chunk の SIMD scan のみ。
現状の 64 Acquire load が 4 回の `_mm256_loadu_si256` に置き換わり、
一桁以上高速化する。

hit ケース (zipf hot key update): 候補発見後の seqlock + CAS は scalar のままで、
Path A の lock-free 性能は維持される。

**c11s reader の `find_get_avx2` との差分**:

| | find_get_avx2 (reader) | find_lockfree_for_path_a (c14s) |
|---|---|---|
| SIMD scan ループ | 同形 | 同形 |
| 候補 seqlock | t1 == t2 && LIVE | t1 == t2 && LIVE (同形) |
| hit 後の動作 | V::clone + conditional fetch_or | (pos, t1, id) を返す |
| 目的 | get の reader hot path | Path A の writer find |

scan 本体はほぼそのまま流用できる。

**spin_loop() との関係**:
c13s の考察 (c13s-sweep §3.1 の補足) で述べたとおり、scalar find_lockfree では
64 iterations が自然な backoff として機能していたが、AVX2 化すると scan が
~4 iterations に縮まり backoff が消える。**MAX_RETRY=1 (§2.2) のため c14s では
CAS 失敗後のリトライループ自体が存在せず spin_loop() は不要**。
もし将来 MAX_RETRY を 2 以上に戻す場合は CAS 失敗時に `hint::spin_loop()` を
挿入すること (tight loop 化防止)。

### §2.2 MAX_RETRY=1 (CAS 失敗時の即 escalate)

```rust
// before (c13s)
const MAX_RETRY: usize = 4;

// after (c14s)
const MAX_RETRY: usize = 1;
```

`try_path_a` のループが 1 回で終わる:
- find_lockfree → None → Err: 変化なし (元から即 Err)
- find_lockfree → Some → CAS 成功 → Ok: 変化なし (1 回で終わる)
- find_lockfree → Some → CAS 失敗 → c13s は find からやり直し最大 4 回。
  **c14s はここで即 Err → Mutex escalate**

CAS 失敗 = 別 writer (Path A or Path B/C) が同 slot に競合中。
Mutex escalate 後の `path_bc` 内 `writer_find` は Mutex 配下で確実に一貫した結果を
返すため、fairness は保たれる。

**zipf への影響**: c13s-sweep §2.2 で示したとおり、skewed zipf の
Path A ゲインは「hot key への CAS が初回成功しやすい」ことから来ている。
適度な skew (zipf-1.0/1.2) では 16T でも CAS 初回成功率は高く、
MAX_RETRY=1 による escalate 増加の影響は小さい見込み。
adversarial-hot (= 全 thread が同一 key を連打する極端ケース) では
CAS 競合が激しいが、そのケースでは writer throughput より reader HR の
確保が優先課題であり §2.3 で対処する。

### §2.3 reader bounded retry (adversarial-hot HR 修正)

`get` の実装 (= `find_get` の呼び出し元) に bounded retry を追加する。

```rust
pub fn get(&self, key: &K) -> Option<V> {
    let needle = self.make_needle(key);
    const MAX_READER_RETRY: usize = 4;
    for _ in 0..MAX_READER_RETRY {
        if let Some(v) = self.find_get(key, needle) {
            return Some(v);
        }
        std::hint::spin_loop();
    }
    None
}
```

false-miss 発生時 (`find_get` が None を返した場合) に同じ lookup を最大 4 回再試行する。
Path A は ~10–20 ns で完了するため、adversarial-hot での false-miss は
1–2 retry で解消する蓋然性が高い。

**true-miss との区別不可問題**: false-miss と true-miss を区別する手段がないため
上限を設ける。上限 4 回での latency ペナルティは O(4 × scan_cost)。AVX2 化後の
scan_cost が下がるため (§2.1)、ペナルティも縮小する。

**adversarial-hot 以外への影響**: false-miss 発生頻度が低い workload では
ほぼ 1 回目で hit or true-miss が確定し retry はほぼ空振りしない。
read-only / read-heavy zipf での latency ペナルティは無視できるレベルの見込み。

**false-miss の本質についての補足**: false-miss は「tag に同期用途とコンテンツ用途を
兼ねさせた」設計から来ており、アルゴリズムの選択の問題で本質的ではない。
entry 側に seqlock counter を分離すれば false-miss を API 表面から消せる (→ §7)。
c14s では構造変更コストを避け bounded retry で実装的に対処する。

## §3 SIEVE 等価性への影響

3 点の変更はいずれも SIEVE state machine に触れない。

| 変更 | SIEVE state | visited | tag HASH/ID/LIVE | hand/len |
|---|:---:|:---:|:---:|:---:|
| find_lockfree AVX2 化 | 変化なし | 変化なし | 変化なし | 変化なし |
| MAX_RETRY=1 | 変化なし | 変化なし | 変化なし | 変化なし |
| reader bounded retry | 変化なし | fetch_or (c13s と同様) | 変化なし | 変化なし |

SIEVE 等価性 (= sieve_orig との eviction 順一致) は c13s から継承される。

## §4 期待性能

### 4.1 uniform read-heavy regression

AVX2 化で scan が 64 → 4 iterations に縮まるため、
try_path_a の overhead が c13s 比で大幅に縮小する。

```
c13s: try_path_a = scalar 64 Acquire load × 毎 write ≈ 数百 ns
c14s: try_path_a = 4 × _mm256_loadu_si256 ≈ 数十 ns
```

path_bc の `writer_find` の overhead は残るが相対的に小さい。
**目標: 1T / 16T ともに c11s ± 5%**。

### 4.2 skewed zipf read-heavy

c13s のゲイン (zipf-1.0 +43%、zipf-1.2 +111% vs c11s) は Path A の
lock-free 化から来ており、MAX_RETRY=1 でも初回成功ケースは同じ経路を辿る。
**目標: c13s 比 ≥ 90%** (多少の Mutex escalate 増加を許容)。

### 4.3 adversarial-hot HR

bounded retry MAX=4 で false-miss をほぼ吸収できる見込み。
PATH A は ~10–20 ns で完了し、retry 1–2 回で真値が返ってくる蓋然性が高い。
**目標: HR ≥ 0.85** (c11s の 0.865 と同水準)。

### 4.4 read-only / gim

reader path の変更は `find_get_avx2` (= reader hot path) には影響しない。
bounded retry も false-miss 時のみ発動し、read-only では Path A が走らないため
false-miss は発生しない。**目標: c13s / c11s と ± 5%**。

## §7 将来方向 — c15 の輪郭 (entry-level seqlock)

c14s の bounded retry は false-miss を実装内で隠す対症療法。
**根本解決は「tag の同期用途とコンテンツ用途の分離」**であることが
今回の議論で明確になった。

### c15 の基本設計

```rust
struct Entry<K, V> {
    version: AtomicU32,  // 偶数=stable, 奇数=in_progress
    key: K,
    value: V,
}
```

```
Path A:
  tag は read-only scan のみ (CAS しない、変更しない)
  CAS(entry[id].version, even → odd)   ← 所有権獲得
  entry[id].value を書き換え
  store(entry[id].version, odd+1)       ← 偶数に戻す (= release)

reader:
  tag scan で pos/id を特定 (tag は Path A で変化しない)
  v1 = entry[id].version.load          (奇数なら spin → in_progress 中)
  snapshot key + value
  v2 = entry[id].version.load
  v1 == v2 && even → Some(value)       ← false-miss なし
  v1 != v2 → 実装内 retry             ← API 表面に None は出ない
```

Path A がタグを変更しないため、reader の `t1 == t2` 検査が Path A に反応しなくなる。
false-miss は API 表面から消えて実装内部の retry に化ける。

### c15 のコスト

- entry ごとに +4 byte (version field)。`Entry<u64,u64>` は 16 byte → 24 byte (padding 次第)
- reader hot path に atomic load 2 本追加 (entry と同 cache line なら実質タダに近い)
- tag-based seqlock は Path C (shift-on-evict) のためには依然必要 → 2 段 seqlock になる
- uniform regression は解消しない (find コストは同じ) → AVX2 find と組み合わせが前提

### c14s / c15 の選択軸

c14s (bounded retry) は実装コストが小さく adversarial-hot HR を実用上の閾値まで
改善できる。c15 (entry-level seqlock) は false-miss を構造的に解消するが
entry サイズ増・実装複雑度増のコストがある。

adversarial-hot が実ワークロードで問題になるかどうかが分岐点。
c14s の bench sweep で HR が安定して ≥ 0.85 を示せば c14s で十分。
そうでなければ c15 を検討する。
