# r3 (RwLock baseline) vs c17s vs senba::concurrent::Cache

## Hypothesis

`docs/reports/2026-05-13-senba-concurrent-vs-c17s.md` で観測した lib (Arc<V> + crossbeam-epoch + Mutex<Box<WriterState>>) の median −34% / worst −63% 退行に対する 3 follow-up 案のうち未着手だった **RwLock baseline** を `sieve_r3` として組み、c17s に対して退行ゼロ + senba::concurrent を median +20% 以上で上回るのが採用基準だった (`2026-05-14-r3-rwlock-plan` plan)。

仮説の根拠は atomic 数の見積もり: c17s reader は ~8-10 atomic (path_c_epoch×2 Acquire + Entry::version×2 Acquire + tag Acquire + visited fetch_or + retry budget)、senba::concurrent は 10+ atomic + Arc strong-count RMW × 2 + `epoch::pin`、r3 は ~5 atomic (parking_lot::RwLock read acquire/release + visited fetch_or)。

## Method

`docs/benchmark/r3-vs-c17s/{run.sh, plot.py}`。c17s / r3 / senba_concurrent 同一 axis × 同一 trial 数:

- variant : c17s, r3, senba_concurrent (全部 shards=512 = cap/8 c8x sweet spot)
- cap     : 4096
- skew    : 0.8 / 1.0 / 1.4
- threads : 1 / 4 / 8 / 16
- op_mix  : gim / read-heavy
- value   : u64, ops = 2 M, trials = 3
- machine : 12600K (P-core 6 + E-core 4, SMT 16 threads, WSL2)

72 invocations × 3 trials = 216 rows, crashes 0。

## Result

**仮説は両側とも棄却**。r3 は c17s に届かず (median **−17.1%** / worst **−76.7%**)、しかし senba_concurrent はちゃんと上回る (median **+12.2%** / best **+75.9%**)。

### Scaling structure (r3/c17s)

| axis | low end | high end |
|---|---|---|
| threads | T=1 median −2.2% | T=16 median −43.2% |

| op_mix  | gim median −12.4% | read-heavy median −19.8% |

**T × skew が dominant、read-heavy が worst を引く**。read-heavy z=1.4 T=16 で r3 は c17s の **23%** しか出ない (32 vs 138 Mops)。

### Mechanism: RwLock reader counter contention

r3 の reader hot path は `parking_lot::RwLock::read()` を呼ぶ。uncontended fast path は ~3 atomic (state load + CAS to increment reader count + memory fence) — atomic 数だけ見れば c17s より明らかに少ない。

しかし parking_lot::RwLock の reader counter は **single shared atomic per lock** で、同一 shard の hot key に 16 thread が同時 read-lock を取ると、その atomic に対して 16 core が連続して `fetch_add` / `fetch_sub` する → cache-line ping-pong (MESI Invalid → Shared → Modified) が 1 lock の reader counter で律速になる。これは Arc::increment_strong_count の `senba::concurrent` 退行と同じ root cause の構造で、ただ「Arc → RwLock counter」に局所性が移っただけ。

c17s が逃げているのは **lock-free reader が共有 atomic を一切書かない** から: `path_c_epoch` は Path C 完了時のみ writer が bump、`Entry::version` は per-entry (= shard 内 ID で分散) で書き込みは Path A/C writer のみ、`tags[pos]` も Path B/C writer のみ。reader は全て read-only load なので cache line は MESI Shared のまま 16 core が並列で読める。これが c17s の hot-shard read-heavy 帯の支配的優位源だった。

**RH > gim worse の解釈**: gim では writer 経路 (insert on miss) が write-lock を頻繁に取り、reader counter contention の壁を「writer 排他 → reader counter reset」で消す瞬間がある。read-heavy 95% read では write-lock 取得が薄く、reader counter atomic が常時 hot で thrash し続ける。

### r3 が c17s に勝つ領域 (gim z=0.8 全 T、gim z=1.0 T=1)

低 skew で同一 shard に集中する thread が少ない領域では reader counter contention が解け、~5 atomic (parking_lot fast path) が c17s の ~8-10 atomic を素直に上回る。T=8 z=0.8 gim で +5.7%、T=8 z=0.8 で r3 が senba を +81.1% 上回るのも同じ構造。

## Implications

仮説は両側 (≥ c17s / +20% vs senba) とも plan の採用基準を満たさない。r3 は **publishable surface 候補としては reject**:

1. **採用基準 (退行ゼロ vs c17s) 未達**: median −12.6%、worst −80% で全 cell 退行ではない (8/24 cell で勝つ) が、c17s が支配的優位を持つ hot-shard 帯を r3 で置き換えると lib 主軸 use case (hot key への並列 read) で大きく退行する。
2. **senba::concurrent には勝つ (median +15%、best +81%)** が、c17s に劣るため lib の置換先として r3 を選ぶ理由が薄い。lib 採用するなら c17s 由来の lock-free reader を保ったまま Arc<V> を別軸で剥がす設計 (`V: Copy` specialization 案 ①) のほうが筋がいい。

c17s の lock-free seqlock design は単に「atomic 数が少ない実装」ではなく **shared atomic を書かないことで MESI ping-pong を構造的に回避する** クリーンな並行設計だった。RwLock baseline (= 教科書解) はこの構造を持たないため、atomic 数が少なくても hot-shard で律速する。

## Follow-ups

- ❌ **r3 を lib に上げる**: reject。
- ⏳ **`V: Copy` specialization で senba::concurrent から Arc<V> + epoch を剥がす** (follow-up 案 ①): c17s 由来の lock-free reader を保ったまま Arc 撤廃すれば、u64/u32 value workload で c17s 同等まで戻せる可能性。これが残された主 path。
- ⏳ **concurrent perf-gate の設置** (follow-up 案 ③): 本 sweep の hotspot (T=16 z=1.4 RH と T=16 z=0.8 gim) を perf-gate 化して `src/concurrent/` の今後の変更を監視。
- 📝 **r3 は research artifact として保持**: parking_lot::RwLock の reader counter limit を定量的に示すデータポイントとして残す価値あり (将来「なぜ RwLock baseline を選ばないか」の引き合いに使える)。bench_concurrent には arm を残す。
