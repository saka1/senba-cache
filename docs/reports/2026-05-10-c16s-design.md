## 0. TL;DR

c14s のヒトコブ写真 (writer state cluster 0.416s + visited 0.276s + Mutex word 0.251s = 計 0.94s mem stall, L3 Bound 21.9%) を直撃する 1-day variant。**c14s から差し替えるのは per-shard struct layout の 1 点のみ**、ロジックは `vbit` の word-index 縮退を除いて変えない。

```rust
#[repr(C, align(64))]
struct ShardHot {
    writer: Mutex<WriterState>,  // raw word @ offset 0
    visited: AtomicU64,          // 1 shard 全 visited (per-shard cap ≤ 64)
    len: AtomicUsize,
}
```

合格 gate は (a) bench_concurrent T=4 skew=1.0 で c14s 比 +5% Mops、(b) VTune memory-access で 3 hot line のうち 2 つ以上が abs mem stall -30%、の同時成立。両方未達なら reject、片方のみなら "explainability gap" として report 残し。

## 1. 動機 — c14s 報告書 §8.1 の直接実装

`c14s-vtune-write-contention.md` §5.3 で得られた 3-line picture:

| 順位 | hot line cluster | abs mem stall | 主体 |
|---:|---|---:|---|
| 1 | writer state cluster (Mutex 配下で書く tags / hand / len / entries) | 0.416s | writer-writer |
| 2 | per-shard `AtomicU64` visited word (1 shard = 1 line 内 8 byte) | 0.276s | reader-writer + writer-writer |
| 3 | per-shard `parking_lot::Mutex` word | 0.251s | writer-writer |

§8.1 の論旨を再掲:

> 別 core が shard ownership を奪うときに 1 transfer で全部 dirty を持っていける。

writer は 1 op で `Mutex word → hand → visited → len` の 4 line を順に取りに行く。各 line で coherence transfer (Modified→Modified / Shared→Modified) が走る。これらを **1 cache line に集約**すれば、別 core が shard を奪う時の transfer は 1 回で済む。tags は 128 byte / 2 line を占めるので別 line に残す (§8.1 mock の通り)。

c15s (visited 単独 sloppy 化) が reject されたのは vbit 0.276s だけ削っても残り 0.667s が残ったから (§7 of c14s report)。c16s は 3 line を一括で殴る方針で、量的根拠を持つ。

## 3. Per-shard layout

### 3.1 ShardHot 仕様

```rust
use parking_lot::Mutex;
use std::sync::atomic::{AtomicU64, AtomicUsize};

/// writer が 1 op で取る hot field を 1 cache line に co-locate。
/// per-shard cap ≤ 64 (6-bit ID 上限) なので visited は AtomicU64 1 つで全 pos を表現。
#[repr(C, align(64))]
struct ShardHot {
    /// parking_lot::Mutex の内部 raw word は構造の先頭 (lock_api::Mutex は #[repr(C)])。
    /// したがって ShardHot が align(64) のもとで Mutex word は cache line 内の offset 0。
    writer: Mutex<WriterState>,
    /// reader fetch_or / writer fetch_and 両方が触る。c14s では別 Box<[AtomicU64]> だった。
    visited: AtomicU64,
    /// scan 上限。reader Acquire load / writer Path B store。
    len: AtomicUsize,
}

struct WriterState {
    hand: usize,
}

pub struct Shard<K, V> {
    capacity: usize,
    /// 別 line 群、~128 byte 占有 (per-shard cap=64 × 2B)。
    tags: Box<[AtomicU16]>,
    /// 別 line 群、Entry size に依存。
    entries: EntriesArena<K, V>,
    /// 1 cache line に集約された writer hot state。
    hot: ShardHot,
}
```

### 3.2 サイズ計算 (x86_64, parking_lot 0.12 系)

| field | size | offset |
|---|---:|---:|
| `Mutex<WriterState>` (raw `AtomicU8` + 7 byte pad + WriterState{hand:usize}) | 16 byte | 0 |
| `AtomicU64 visited` | 8 byte | 16 |
| `AtomicUsize len` | 8 byte | 24 |
| **小計** | **32 byte** | |
| trailing pad (align(64) 強制) | 32 byte | 32–63 |

→ ShardHot は 64 byte 占有、全 hot field が **1 cache line 内**。

### 3.3 静的検査

```rust
const _: () = {
    assert!(std::mem::size_of::<ShardHot>() == 64);
    assert!(std::mem::align_of::<ShardHot>() == 64);
};
```

`parking_lot::Mutex<T>` の repr は `#[repr(C)] lock_api::Mutex { raw: R, data: UnsafeCell<T> }` で raw が offset 0 — version 依存だが現行 0.12 系で確実。万一 layout が崩れた場合は const assertion で compile-time に検出される。

### 3.4 vbit の縮退

c14s では visited が `Box<[AtomicU64]>` だったため `vbit(pos) → (word_idx, mask)` の 2 値返しだった。c16s では word_idx は常に 0 なので `vbit_mask(pos) → mask: u64` の 1 値返しに縮退。

```rust
#[inline]
fn vbit_mask(pos: usize) -> u64 {
    debug_assert!(pos < 64);
    1u64 << pos  // pos は per-shard cap ≤ 64 なので & 63 不要、debug でガード
}
```

## 4. コード変更スコープ

### 4.1 機械的置換 (c14s 比)

| c14s | c16s |
|---|---|
| `self.len.load(...)` | `self.hot.len.load(...)` |
| `self.len.store(...)` | `self.hot.len.store(...)` |
| `self.visited[w].fetch_or(b, ...)` | `self.hot.visited.fetch_or(mask, ...)` |
| `self.visited[w].fetch_and(!b, ...)` | `self.hot.visited.fetch_and(!mask, ...)` |
| `self.visited[w].load(...)` | `self.hot.visited.load(...)` |
| `self.writer.lock()` | `self.hot.writer.lock()` |
| `let (w, b) = Self::vbit(pos);` | `let mask = Self::vbit_mask(pos);` |

### 4.2 ロジック変更 (機械的でない箇所)

1. **`Shard::new`** — `Box<[AtomicU64]>` 確保コードを削除。`hot: ShardHot { writer: Mutex::new(WriterState{hand:0}), visited: AtomicU64::new(0), len: AtomicUsize::new(0) }` で初期化。`visited_words` ループは消える。
2. **`scan_evict` の visited 走査** — c14s は word 単位で進めて bit を順に剥がしていた。c16s は word 1 つ固定で、index 計算が消えるだけで loop 構造は同じ。c14s の `sw, sb / dw, db` の同 word ケース最適化分岐は `s_word == d_word`（常に true）に縮退、整理しなおせる。
3. **テスト** — c14s のユニットテストの大半は `Shard` 経由 API を呼ぶので流用可。`vbit` を直接呼ぶ箇所があれば `vbit_mask` 用に書き直す。

### 4.3 変わらない部分 (重要)

- `tags: Box<[AtomicU16]>` の構造と layout
- `entries: UnsafeCell<Box<[MaybeUninit<Entry<K,V>>]>>`
- Path A (lock-free CAS update) のロジック全体
- Path B/C escalate の trigger 条件 (MAX_RETRY=1)
- AVX2 `find_get_avx2` の reader 実装
- seqlock-via-tag retry idiom
- SHARDS=64、6-bit ID schema、tag bit 配置 (HASH/ID/LIVE/VERSION)
- `unsafe impl Send/Sync` の SAFETY 注釈 (c14s からそのまま流用、entries 所有権の議論は変わらない)

## 6. 検証 step (順序固定)

### Step 1: 正しさ — oracle 一致

```bash
cargo test -p senba-research sieve_c16s
```

- c14s から流用したユニットテストが pass
- `research/tests/oracle.rs` に c16s arm を追加 — トレース replay で `sieve_orig` と evicted-key 列が一致

不合格なら以降の bench/VTune 計測には進まない。

### Step 2: 単発 Mops AB (gate (a))

```
bench_concurrent --variant c14s,c16s --threads 4 \
    --cap 4096 --keys 100000 --skew 1.0 --ops 80000000 --warmup 1000000 --trials 5
```

**Gate (a)**: `c16s_mops / c14s_mops >= 1.05` (5 trial 平均、CV < 0.05)。

不合格なら以降の VTune 計測前に design 再考。reject 路線に振る。

### Step 3: thread sweep

```
bench_concurrent --variant c9,c14s,c16s --threads 2,4,8,16 \
    --cap 4096 --keys 100000 --skew 1.0 --ops 80000000 --warmup 1000000 --trials 5
```

Mops と thread CV を表形式で取得。**T=2 で大幅 regression / T=16 で逆転**などの異常パターンを検出。
c9 を含めるのは「Mutex-only → Path A lock-free → + co-locate」の進化軸を 1 表で見るため。

### Step 4: VTune memory-access (gate (b))

T=4 / skew=1.0 で 1 回採取 (報告書 §3 と同条件)。

```
vtune -collect memory-access -knob analyze-mem-objects=true \
      -start-paused -- bench_vtune_concurrent.exe --variant c16s --threads 4 \
      --cap 4096 --keys 100000 --skew 1.0 --ops 80000000 --warmup 1000000
```

**Gate (b)**: 以下のうち **2 つ以上**:
- `writer_evict_and_install` の abs mem stall ≤ 0.291s (= 0.416s × 0.7)
- `vbit_mask` (相当の関数) の abs mem stall ≤ 0.193s (= 0.276s × 0.7)
- `parking_lot::lock + unlock` の abs mem stall 合算 ≤ 0.176s (= 0.251s × 0.7)

### Step 5: VTune uarch-exploration (参考)

同条件で `uarch-exploration` 採取。L3 Bound% が **21.9% → 15% 以下**に下がるか確認 (gate ではなく観測指標)。

### 合否判定

| Gate (a) | Gate (b) | 判断 |
|---|---|---|
| ✓ | ✓ | **採用**。c16s をレポートに残し index 更新、後続 design space 探索の起点 |
| ✓ | ✗ | "Mops は出たが説明が弱い"。report 残し、追加 VTune 角度から再検証 |
| ✗ | ✓ | "計測上は説明できたが wall-clock に出ない"。次に何を狙うかが見える結果 |
| ✗ | ✗ | reject。c15s 同様 "rejected variant" として理由を残し closure |

## 7. 反証リスト (実装前に立てた仮説)

- 仮説 1 「writer hot 4 line を 1 line co-locate で transfer cost が落ちる」 — Step 4 で直接検証
- 仮説 2 「reader bouncing は writer 側改善より小さく net 改善」 — Step 2/4 で間接検証 (`find_get_avx2` Memory Bound% 監視、悪化なら仮説否定)
- 仮説 3 「parking_lot::Mutex の repr で raw word が ShardHot offset 0」 — `const _: () = ...` static assertion で compile-time に検査
- 仮説 4 「visited が 1 word に縮退しても scan_evict のロジックが破れない」 — Step 1 oracle 一致で検証

## 8. 不採択時の next move 候補

c16s が reject された場合の判断材料を先に並べる:

1. **A2 (writer / reader 分離 layout) を c17s として試す** — Mutex word を独立 line に隔離、visited+len を別 line にまとめる。reader bouncing 仮説 (仮説 2) が崩れたケースで根拠が立つ。
2. **§8.2 (writer batching) に軸を移す** — eviction 順序保存制約の design doc 先行。1-day では収まらない。
3. **shard-affinity (per-shard worker)** — moderate Zipf 帯の保険札としてのみ。本稿の hot-key workload では効かない既知。

## 9. スコープ外 (今回はやらない)

- **§8.2 writer batching** — 同一 op 内で複数 op を amortize する設計。Q1 で「1-day 単発検証」を選択したため別変種で扱う。
- **A2 layout (writer / reader 分離)** — 設計時に「A1 採用、A2 は c17s 候補」と決定 (reader bouncing が顕在化した場合の次の手)。
- **A3 layout (entries pointer も co-locate)** — ROI 低い (entries 本体は別 line) と判定、保留。
- **senba lib への取り込み** — c16s は experimental。lib 化は採用が確定し perf-gate / oracle が通った後。
- **moderate skew (skew=0.7) / uniform workload** — Q5 で thread sweep T=2,4,8,16 を選択。skew sweep は別タスク。
- **Loom 等の formal concurrency 検証** — 構造変更が layout のみで Path A/B/C のロジックが c14s と同型なので、Loom 範囲は c14s と同等。優先度低。
