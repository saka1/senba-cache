# 2026-05-11 — c17s 計測結果: entry-level seqlock (adv-hot 16T +28.1%、skew=1.0 mixed −4.2%)

- 種別: **計測結果 + 採否 (条件付き採用)**。Step 1 (oracle) / Step 2 (Mops AB gate (a))
  / Step 3 (regression check gate (b)) / Step 4 (thread sweep) を WSL2 で完走。
- 関連:
  - `2026-05-11-c17s-design.md` — 設計仕様 (本稿はこれに対する計測追試)
  - `2026-05-10-c16s-results.md` — 直前 baseline (c16s 採用報告)
  - `2026-05-08-c14s-sweep.md` §4.2 — c17s が解こうとした構造的退行の同定

## 0. TL;DR

c17s = c16s の同期通知を tag → entry に逃がした variant (G2-α-1)。`Entry` に
`AtomicU32 version` を追加、tag VERSION bit を削除 (HASH 8→9 bit)、Path A は tag を
一切触らずに entry version 偶奇 flip だけで reader を seqlock。`find_get` の EMPTY-lane
SIMD 検出を削除し、Path C false-miss は新設の `path_c_epoch` で coarse 補償。

| Step | 結果 | 判定 |
|---|---|---|
| 1 oracle 一致 (4 (skew, cap) × 200k op) | diff = 0 | **PASS** |
| 2 gate (a) adv-hot read-heavy 16T (skew 1.4) | c17s 150.87 / c16s 124.42 = **+21.3%** (5 trial) | **PASS (≥ +5%)** |
| 3 gate (b) skew=1.0 gim T=4 | c17s 33.67 / c16s 35.26 = **−4.5%** | **FAIL (要 ≥ −2%)** |
| 4a thread sweep skew=1.0 gim T=1〜16 | 全帯で c17s が c16s を 1〜5% 下回る | (b) と整合 |
| 4b thread sweep adv-hot read-heavy T=1〜16 | T=1 で −0.9%、T=2 で +8.9%、T=4 で +13.9%、T=8 で +18.5%、**T=16 で +28.1%** | (a) を強化、scaling 形状 super-linear |

設計 §5 合否表の **(a) ✓ / (b) ✗** 位置 (= "adv-hot は伸びたが skew=1.0 で退行")。
**条件付き採用**: c17s は **adv-hot read-heavy / 高並列 / hot-key 集中** workload では
構造的に c16s を支配し、HR も +0.3pp 構造改善する。一方で **skew=1.0 mixed (gim、
eviction-heavy)** では Slot32 化 (Entry 16B → 32B) 由来の footprint cost が 2-tier
seqlock の利得を上回り、3〜5% regress する。次の手は §6 で議論する。

## 2. Step 1 — oracle 一致 (correctness gate)

```bash
cargo test -p senba-research sieve_c17s
cargo test -p senba-research --test oracle c17s
```

`research/tests/oracle.rs::c17s_1shard_matches_orig_on_synthetic_zipf` を新規追加。
構成は c16s 版と同型: `(skew, cap) ∈ {(1.05, 16), (1.1, 32), (1.2, 64), (1.5, 64)}`、
各 200,000 op を `sieve_orig` と c17s で並走。**4 構成全てで diff = 0**。

`cargo test --workspace` も全 pass (合計 535 tests + 1 ignored、c17s unit テスト 12 件
を含む)。c17s 専用 unit テストの主なもの:

- `update_via_path_a_preserves_id_and_tag`: Path A update 後 tag が完全不変 (c17s 固有、
  c14s/c16s は VERSION bit が flip)
- `path_a_increments_entry_version_by_two`: 同 key 連続 update で entry version が 2, 4
  と steady increment (偶数で着地)
- `evict_reuses_id_at_tail_position_and_bumps_epoch`: Path C で id 再利用 + path_c_epoch bump
- `bit_layout_exclusivity_u64_u64`: HASH 9 bit (`HASH_MASK.count_ones() == 9`)、Entry sizeof=32
- `shard_hot_layout_contract`: ShardHot sizeof=64, align=64 維持 (path_c_epoch 追加でも 64B)
- `concurrent_invariants_under_zipf`: 4 thread × 50k op の zipf workload で id 重複 / len
  整合 / value 破壊なし

## 3. Step 2 — Mops AB gate (a): adv-hot read-heavy 16T

```bash
./target/release/bench_concurrent --variant c16s,c17s --threads 16 --shards 64 \
    --cap 4096 --keys 100000 --skew 1.4 --ops 40000000 --warmup 1000000 --trials 5 \
    --op-mix read-heavy
```

c14s-sweep §4.2 の T1 シナリオに対応 (adv-hot ≈ skew 1.4、read-heavy = read 95% /
write 5%)。

| variant | trial Mops | 平均 Mops | stdev | CV | HR |
|---|---|---:|---:|---:|---:|
| c16s | 123.62, 125.21, 126.50, 125.40, 121.37 | **124.42** | 1.97 | 0.016 | 0.9248 |
| c17s | 149.20, 145.02, 154.49, 153.32, 152.34 | **150.87** | 3.62 | 0.024 | **0.9280** |

**c17s / c16s = 150.87 / 124.42 = 1.213 (+21.3%)**。CV は両方 < 0.05。
gate (a) = ≥ +5% を **PASS** (期待値の 4 倍超の効きで着地)。

副次観察:
- **HR が +0.32pp 向上** (c16s 0.9248 → c17s 0.9280)。adv-hot 16T で c14s/c16s は
  「reader が Path A の EMPTY transient を観測したまま racing を消化しきれず None を
  返す race」由来の false-miss が ~0.3pp あった、という c14s-sweep §4.2 の予想に整合。
  c17s では Path A が tag を EMPTY 化しないため、この false-miss は構造除去された。
- **per-thread CV (c17s) = 0.022〜0.047** (gate (b) `T1 thread CV < 0.05` を満たす)、
  scheduling 偏りなし。

## 4. Step 3 — Regression check gate (b): skew=1.0 gim T=4

```bash
./target/release/bench_concurrent --variant c16s,c17s --threads 4 --shards 64 \
    --cap 4096 --keys 100000 --skew 1.0 --ops 40000000 --warmup 1000000 --trials 5
```

c14s/c16s の sweet spot シナリオ (skew=1.0 mixed Gim、insert/get half-and-half)。

| variant | trial Mops | 平均 Mops | stdev | CV | HR |
|---|---|---:|---:|---:|---:|
| c16s | 35.37, 35.50, 35.33, 35.26, 34.85 | **35.26** | 0.245 | 0.007 | 0.7195 |
| c17s | 33.80, 33.84, 33.43, 33.53, 33.76 | **33.67** | 0.183 | 0.005 | 0.7195 |

**c17s / c16s = 33.67 / 35.26 = 0.955 (−4.5%)**。HR は完全一致 (0.7195)。
gate (b) = ≥ −2% を **FAIL** (regression が gate を 2.5pp 超過)。

退行の主因候補 (定量検証は Step 4 / VTune follow-up 待ち):
1. **Slot32 化 (Entry 16B → 32B)** — entries arena が per-shard 1024B → 2048B (16 → 32
   cache line)。Gim mixed は eviction/insert が op 半分を占めるので、entries 列の L1d/L2
   transfer が支配。skew=1.0 では hot key 集中が薄く、entries 全域が touch される。
2. **2-tier seqlock の固定 cost** — try_candidate で entry version load が 2 回 (v1, v2)
   追加。skew=1.0 で hit/miss 半々の場合、hit 時の cost が直接乗る。
3. **path_c_epoch の reader 側 load** — get_by_hash の miss 経路で 2 回 atomic load。
   miss rate 28% に比例して overhead が出る。

## 5. Step 4 — Thread sweep

### 5.1 skew=1.0 gim (gate (b) 同シナリオの T sweep)

```bash
for t in 1 2 4 8 16; do
  ./target/release/bench_concurrent --variant c16s,c17s --threads $t --shards 64 \
      --cap 4096 --keys 100000 --skew 1.0 --ops 40000000 --warmup 1000000 --trials 3
done
```

3 trial 平均 Mops:

| threads | c16s | c17s | c17s/c16s | HR c16s | HR c17s |
|---:|---:|---:|---:|---:|---:|
| 1  | 11.21 | 11.12 | **−0.8%**  | 0.7197 | 0.7197 |
| 2  | 19.69 | 19.07 | **−3.1%**  | 0.7196 | 0.7196 |
| 4  | 34.81 | 33.33 | **−4.2%**  | 0.7195 | 0.7194 |
| 8  | 50.25 | 47.86 | **−4.7%**  | 0.7196 | 0.7194 |
| 16 | 62.28 | 60.02 | **−3.6%**  | 0.7196 | 0.7196 |

観察:
- 全帯で 1〜5% regression、HR 完全一致。**SIEVE 等価性は thread 数によらず保持**。
- T=8 で regression 最大 (−4.7%)。T=16 で僅かに回復 (−3.6%) — Slot32 化の L1d 圧迫が
  T=8 thread 同居 cache shared 帯で最も痛い、と思われる。T=16 では既に各 core が
  saturate しているので相対比が縮む。
- Step 3 の T=4 単独 (−4.5%) と T=4 sweep (−4.2%) の差は trial 数 (5 vs 3) のばらつき
  範囲内。CV < 0.01 で安定。

### 5.2 adv-hot read-heavy (gate (a) 同シナリオの T sweep)

```bash
for t in 1 2 4 8 16; do
  ./target/release/bench_concurrent --variant c16s,c17s --threads $t --shards 64 \
      --cap 4096 --keys 100000 --skew 1.4 --ops 40000000 --warmup 1000000 --trials 3 \
      --op-mix read-heavy
done
```

3 trial 平均 Mops:

| threads | c16s | c17s | c17s/c16s | HR c16s | HR c17s |
|---:|---:|---:|---:|---:|---:|
| 1  | 23.47 | 23.25 | **−0.9%**  | 0.9282 | 0.9282 |
| 2  | 38.65 | 42.09 | **+8.9%**  | 0.9281 | 0.9281 |
| 4  | 67.53 | 76.92 | **+13.9%** | 0.9279 | 0.9282 |
| 8  | 96.85 | 114.80 | **+18.5%** | 0.9273 | 0.9281 |
| 16 | 123.67 | 158.34 | **+28.1%** | 0.9249-0.9261 | **0.9280** |

観察:
- **scaling 形状 super-linear**: T=1 で −0.9% (2-tier seqlock cost) → T=16 で +28.1%。
  c17s の利得は thread 数に対して単調増加、c14s-sweep §4.2 で同定された adv-hot 16T
  退行の構造的原因 (find_get の EMPTY-lane SIMD overhead) を構造除去できたことが
  thread スケール側からも裏付けられる。
- **HR の構造改善が高 T で顕在化**: c16s は T=8 で 0.9273、T=16 で 0.9249-0.9261
  と HR が thread 数に対して劣化 (false-miss race)。c17s は **T=16 でも 0.9280 を
  保持** (T=1 と同水準) — Path A 由来の false-miss が消えたための構造的安定性。
- **path_c_epoch は false-miss を起こしていない**: Path C false-miss 補償 (epoch ベース
  の coarse retry) は HR を c16s 同水準以上で維持できており、設計仮説が量的に整合。

| # | 仮説 | 結果 |

| 2 | EMPTY-lane SIMD 削除で adv-hot read-heavy +5% 以上 | **大幅 PASS**: T=16 +28.1%、T=8 +18.5%、T=4 +13.9% で連鎖的に効いている |

## 7. 採否 — 条件付き採用

### 7.1 採択ロジック

| 軸 | c17s 評価 |
|---|---|
| 正しさ (oracle 一致) | ✓ SIEVE 等価、`sieve_orig` と完全一致 |
| 主目的 (adv-hot read-heavy 構造除去) | ✓ T=16 +28.1% / HR +0.32pp の二重勝ち |
| 副次目的 (skew=1.0 gim regression ≤ 2%) | ✗ −4.2〜−4.7% で gate 越え |
| 設計の説明性 | ✓ scaling 形状 / HR 改善 / 退行原因が一貫して仮説と整合 |

「adv-hot read-heavy で c14s/c16s が落ちる帯を構造的に救う」という当初目標は **完全達成**。
ただし「c14s/c16s sweet spot で regress しない」という併存条件は満たせていないので、
**lib 公開は保留、experimental 採用 (= "次の設計の base")** として位置付ける。

### 7.2 c14s/c16s/c17s の使い分けマップ (現時点の理解)

| workload class | 最適 variant | 根拠 |
|---|---|---|
| adv-hot read-heavy / 高 T (T ≥ 4) | **c17s** | Step 5.2 で 14〜28% 勝ち、HR 構造改善 |
| moderate skew / mixed Gim / T ≤ 8 | **c16s** | Step 5.1 で 3〜5% 勝ち、Slot16 entries が L1d 友好 |
| evict-heavy / churn-dominated | **c16s** (現状) | Slot32 化のコストが見える、c17s は α-2 化が要 |

## 8. 後続 design space (採用後の next move)

### 8.1 c17s の cost 側を削る方向

1. **G2-α-2 (versions 別配列化)** — Entry sizeof を 16B (Slot16) に戻し、versions[] を
   別 cache line に追い出す。reader が cache line 1 個追加 touch する代わりに、
   entries 列の footprint が c16s 同水準に戻る。skew=1.0 gim regression が解消する見込み。
   実装規模は中 (c17s-α1 から layout 1 点だけ差し替え)。
2. **path_c_epoch の場所最適化** — 現在 ShardHot 内で writer が触る hot 4 line に同居。
   reader-only line (例: tags 直前) に逃がせば writer transfer から外れる。
3. **2-tier seqlock の tier 2 を skip** — tier 2 (tag re-load) は Path C 検出専用なので、
   `path_c_epoch` snapshot で代替できる。get_by_hash の epoch コードを `try_candidate`
   内に下ろせば try_candidate から `t1 == t2` が削れる。実装規模小。

### 8.2 c17s の win 側を伸ばす方向

4. **G2-β SoA state** — α-1 の long-term 上位互換、`tags / slots(version+lock+epoch) /
   visited / entries` の 4 配列分離。AVX-512 V5 の前段、α-1 の Slot32 footprint も
   解消。実装規模大。
5. **Loom / formal verify の追加** — Path A version CAS / 2-tier seqlock は新規構造
   なので、長期化に向けた formal モデル化。

### 8.3 c17s で確認できなかった軸 (今回 scope 外)

- **VTune memory-access** — Windows native 経由で entries 列の MESI transfer / L3 pressure を
  直接観測。Step 4 の skew=1.0 gim regression 主因 (Slot32 化 vs 2-tier seqlock) を分解できる。
- **read-heavy 比 sweep (98/2, 99/1)** — adv-hot の利得が write 比でどう変わるか。
- **skew sweep (0.6, 0.8, 1.2, 1.4) で c16s/c17s クロスオーバー曲面** — どの skew 帯から
  c17s が c16s を抜くかを地図化。

## 9. 既知の race / clean-up

- **clone-mid-flight UB (V::Clone soundness)** — c14s/c16s と同じ既知問題 (V = u64 では sound、
  heap-owning V には Arc<V> / Epoch GC が要)。c17s でも継承、module doc に記載。
- **Path A vs `writer_update_in_place` race** — c14s/c16s と同様、entries[id] 2 重書き race
  は id 重複を引き起こさないので既存テストでは検出不可。c17s では entry version CAS で
  排他されるため理論的には消えているはず (= writer_update_in_place の version CAS と
  Path A の version CAS が同じ word を奪い合う)、実機検証は別 test 設計要。
- **path_c_epoch wraparound** — `AtomicU64` で 64 bit、毎秒 1G eviction でも 600 年不変、
  実用上は不要。

## 11. Tuning pass: tier 2 削除 + epoch hot-path optimization (try / revert)

§7.1 採択ロジックの (b) regression を縮める方向で 2 つの実装 tuning を試した結果。

### 11.1 (b) tier 2 (tag re-load) 削除 — 採用

`try_candidate` / `try_path_a_candidate` 末尾の `t2 = tags[pos].load + (t1 != t2 || (t2 & LIVE) == 0) → Racing` を削除。tier 1 (entry version) + `path_c_epoch` で覆える理由は §8.1 #3 で議論済み。Path C shift で tag が古い id_X を指していても (a) entries[X] が consistent なら buf.key 比較で正しく判定、(b) entries[X] が evict 対象なら version flip が tier 1 を fire させる。残る "Path C 中 chunk SIMD で candidate 不発" は path_c_epoch で coarse 検出。

per-candidate atomic load 数: c16s 2 (t1+t2) → c17s v1 3 (t1+v1+v2+t2) → **c17s v2 3 (t1+v1+v2)** (tier 2 削除で c16s と並ぶ ID 抽出 t1 + tier 1 で v1, v2)。

実装は `try_candidate` で `t2` load + `t1 != t2` check の 4 行削除のみ。

### 11.2 (c) get_by_hash hit hot-path で epoch_before を skip — **却下、revert**

仮説: hit 経路で `epoch_before` を load しないことで hit op あたり 1 atomic load 削減。実装は post-scan epoch 取得 + 連続 2 scan で true-miss 判定。

```rust
// 試した実装 (revert 済み)
let (v, racing) = find_get(...);  // Round 1, no epoch
if let Some(v) = v { return Some(v); }
let mut last_epoch = self.hot.path_c_epoch.load(Acquire);  // baseline post-scan
for _ in 1..MAX_RETRY {
    spin_loop();
    let (v, r) = find_get(...);  // Round 2 強制
    if let Some(v) = v { return Some(v); }
    let new_epoch = ...;
    if !last_racing && !r && new_epoch == last_epoch { return None; }
    ...
}
```

**問題**: true-miss path で **find_get が 2 回必要** になる (baseline epoch と検証 scan で対比する必要)。原 c17s は 1 find_get + 2 epoch、新版は 2 find_get + 2 epoch。skew=1.0 gim T=4 で miss rate 28%、`find_get` の SIMD scan (16 chunk × ~5 instr) cost が epoch load 1 回分を遥かに超える。

#### 計測結果 (5 trial 平均、原 c17s と比較)

| シナリオ | 原 c17s (epoch_before あり) | tuning v1 (epoch skip + tier 2 削除) | Δ |
|---|---:|---:|---:|
| Gate (a) adv-hot 16T | +21.3% | +24.5% | +3.2pp (tier 2 効果) |
| Gate (b) gim T=4 | **−4.5%** | **−14.3%** | **−9.8pp (epoch skip 失敗)** |

#### 失敗の root cause

- **epoch_before は ShardHot 内 64B line 同居**: `find_get` 冒頭の `len.load` で line を L1 に呼び込むので、epoch_before load 自体は L1-hot で ~0.5ns。**節約効果はほぼゼロ**。
- **find_get の 2 回 cost** が支配: 28% miss × ~50ns extra find_get = ~14ns / op の overhead が直接に乗る。
- 原 c17s の `if let Some(v) = v { return Some(v); }` 早期 return が既に epoch_after を skip しているので、hit hot path は 1 epoch_before のみ — **epoch placement は元から最適だった**。

教訓: 「同 cache line 内 atomic load の削除」は L1-hot 前提下では ~0ns 効果。footprint 削減や transfer 削減でないと意味がない。実装難易度の低い optimization でも cost-benefit が逆転しうる典型例として artifact (本 §11.2) に残す。

### 11.3 計測結果 — c17s v2 (tier 2 削除のみ)

#### Gate AB (5 trial 平均)

| Gate | c16s | c17s v1 (原) | c17s v2 (tier 2 削除) | v2/c16s | v2 vs v1 |
|---|---:|---:|---:|---:|---:|
| (a) adv-hot read-heavy 16T | 123.89 | 150.87 | **157.87 Mops** | **+27.4%** | +6.1pp |
| (b) skew=1.0 gim T=4 | 34.90 | 33.67 | **33.62 Mops** | **−3.7%** | +0.8pp |

**Gate (a) は更に伸びて +27.4% / Gate (b) は −3.7% で gate −2% を 1.7pp 越え (依然 fail)**。
