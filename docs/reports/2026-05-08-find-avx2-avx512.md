# 2026-05-08 — `find_avx2` AVX-512 経路: server share を踏まえた opt-in 設計案

- 親:

  - `2026-05-08-find-avx2-pext.md` (PEXT/PDEP 系最適化との相互作用 — 結論的に AVX-512 は P1/P2 を吸収)

## 0. TL;DR

サーバ向け CPU では AVX-512 がほぼ ubiquitous (Intel Xeon Skylake-X / Cascade Lake /
Ice Lake / Sapphire Rapids / Emerald Rapids / Granite Rapids、AMD EPYC Zen 4 / Zen 5) で、
client 側の AVX-512 撤去とは事情が異なる。`senba::Cache` を server で運用する利用者向けに
**opt-in の AVX-512 経路を用意する**価値がある。具体的な勝ち手は次の 3 つ:

- **V1 (AVX-512 VL + kmask, 256-bit width)**: `vpcmpeqw_mask` で `vpmovmskb` (5 cy latency)
  を消し、kmask が最初から 1-bit / 1-lane の形式なので **BLSR ×1 / 1 lane** で済む
  (現状の BLSR ×2 ペア廃止)。`mask.trailing_zeros()` が直接 lane index となり
  `lane = bit >> 1` も不要。**downclock リスクなし** (256-bit 幅維持)。
- **V2 (zmm, 512-bit width)**: per_shard=64 が 4 chunks → 2 chunks。outer ループ半減。
  Sapphire Rapids+ / Zen 4+ では downclock 実質ゼロ、Skylake-X / Cascade Lake では
  懸念あり → cargo feature opt-in が筋。
- **V5 (V2 + B1 SoA tag split)**: 8-bit hash byte 配列を `vpcmpeqb_mask` で **64 lane
  同時比較**。per_shard=64 が **1 chunk shot** で済み outer ループ自体が消える。
  AVX-512 と B1 は強い相乗。

PEXT 系最適化 (`find-avx2-pext` の P1/P2) は AVX-512 では **kmask が初めから lane-mask
形式なので不要**。**P3 (PDEP `needle_from_hash`) は orthogonal で温存**。

実装は **cargo feature `avx512-server` + runtime feature detection** の二層で、crates.io
配布の portability は維持。本稿は実装方針と効果見積もりまで、実測は別 PR。

## 1. AVX-512 が変える前提

### 1.1 命令レベルでの変化

`find_avx2` の inner loop で AVX2 使用部分は次の 3 軸:

```
A: 32B load + AND + cmpeq            : vpand (mem), ymm, ymm → vpcmpeqw ymm, ymm, ymm
B: 結果を mask register に            : vpmovmskb r32, ymm   ← 5 cy latency
C: lane index を取り、tag を再 load : tzcnt → movzwl + and  → cmp [entries+id*S]
```

AVX-512 BW + VL を許すと B が消える: `vpcmpeqw_mask` が `__m256i × __m256i → mmask16`
を 1 uop で返す (= `_mm256_cmpeq_epi16_mask`)。AVX2 比で:

| 軸 | AVX2 (現状) | AVX-512 VL (256-bit) | AVX-512 zmm (512-bit) |
|---|---|---|---|
| 結果取得 | `vpcmpeqw + vpmovmskb` (lat 1+5 cy) | `vpcmpeqw_mask` → kmask (lat ~3 cy) | 同左 (kmask が 32-bit) |
| mask 形式 | **1 lane = 2 bit pair** (vpmovmskb_epi8 の副作用) | **1 lane = 1 bit** (kmask 直結) | 同左 |
| BLSR | ×2 (pair clear) | **×1** | ×1 |
| lane = bit ? | `lane = bit >> 1` 必要 | **lane = bit** 直接 | 同左 |
| chunk あたり lane 数 (u16) | 16 | 16 | **32** |
| per_shard=64 で chunk 数 | 4 | 4 | **2** |

`vpmovmskb` は port 0 のみ、`kmovd` (k → r32) は port 5。pipeline 上の port 分散も改善
方向。movemask + GPR test の chain がそのまま `kortestz_mask16` (k 同士の OR-test) で
GPR を経由せず分岐に流せる。

### 1.2 server share

このレポートの動機: client (Alder Lake 以降の Intel desktop) は AVX-512 を E-core 都合で
無効化したが、**server 用 SKU は全て AVX-512 を保持**している。2026 年現在のおよその分布:

| 区分 | 世代 | AVX-512 | downclock 規模 |
|---|---|---|---|
| Intel Xeon Skylake-X / Cascade Lake | 2017〜 | あり (F/CD/BW/DQ/VL) | 大 (heavy turbo 帯で −10〜30%) |
| Intel Xeon Ice Lake-SP | 2021 | あり (+VBMI/VBMI2/VPOPCNTDQ/...) | 中 |
| Intel Xeon Sapphire Rapids | 2023 | あり (+VNNI/BF16/AMX) | **小〜ほぼゼロ** |
| Intel Xeon Emerald / Granite Rapids | 2024〜 | あり | ほぼゼロ |
| AMD EPYC Zen 4 (Genoa) | 2023 | あり (双倍 256-bit FPU) | ほぼゼロ |
| AMD EPYC Zen 5 (Turin) | 2024 | あり (native 512-bit FPU) | ゼロ |
| Intel client (Skylake〜Rocket Lake) | 2017〜2021 | 一部あり | (server 同様) |
| Intel client (Alder Lake〜) | 2021〜 | **なし** (E-core 都合で無効化) | — |

主要クラウド providers の最新世代 instance (AWS c7i/c7a、GCP c3/c3d、Azure HBv4/HX)
はすべて AVX-512 を持つ。**senba を server LRU として組み込むユーザは恩恵が直接的**。

### 1.3 downclock の現実的影響

`find_avx2` は **連続して回る hot path** (per request 1〜数十回 invocation)。downclock は
"重い AVX-512 を使う期間中は frequency が落ちて、それ以外の scalar code は遅くなる"
という現象。本ライブラリの利用パターンは:

- (a) cache lookup が application の主要 hot path → downclock してもライブラリ自身は
  AVX-512 が速いので net で勝つことが多い
- (b) cache lookup は薄く、scalar 側の処理が重い → downclock で scalar が遅くなり net で
  負けうる

(a) の場合 downclock は問題視しなくてよい。(b) を考慮すると **256-bit 幅 (V1) は安全、
512-bit 幅 (V2) は opt-in**、という分け方が筋。client AVX-512 (Skylake-X / Cascade Lake)
で 512-bit を踏むと heavy turbo 帯で −10〜30% の周波数下落があり、用途次第。
Sapphire Rapids+ と Zen 4+ ではこの問題は実質消えている。

## 2. 勝ち手候補 (V1〜V6)

### V1. AVX-512 VL + kmask (256-bit 幅維持、no downclock)

最小侵襲案。LANE = 16 のまま、`_mm256_cmpeq_epi16_mask` を使うだけで Path B が短縮される。

```rust
#[target_feature(enable = "avx512f,avx512bw,avx512vl,bmi1")]
unsafe fn find_avx512vl<Q>(&self, key: &Q, needle: u16) -> Option<(usize, u16)>
where K: Borrow<Q>, Q: Eq + ?Sized,
{
    use std::arch::x86_64::*;
    let limit = (self.len + LANE - 1) & !(LANE - 1);
    let tags_ptr = self.tags.as_ptr();
    let entries_byte_ptr = self.entries.as_ptr() as *const u8;
    let id_mask_u32 = Self::ID_MASK as u32;

    let needle_v = _mm256_set1_epi16(needle as i16);
    let mask_v   = _mm256_set1_epi16(Self::SCAN_MASK as i16);

    let mut i = 0usize;
    while i < limit {
        let v       = _mm256_load_si256(tags_ptr.add(i) as *const __m256i);
        let masked  = _mm256_and_si256(v, mask_v);
        let k: __mmask16 = _mm256_cmpeq_epi16_mask(masked, needle_v);  // 1 uop, lat ~3
        let mut mask = k as u32;                                       // kmovw / kmovd

        let chunk_byte_ptr = (tags_ptr as *const u8).add(i * 2);

        while mask != 0 {
            let lane = mask.trailing_zeros() as usize;                 // 直接 lane
            let tag  = *(chunk_byte_ptr.add(lane * 2) as *const u16) as u32;
            let off  = (tag & id_mask_u32) as usize;
            let e    = &*(entries_byte_ptr.add(off) as *const Entry<K, V>);
            if e.key.borrow() == key {
                return Some((i + lane, tag as u16));                    // lane >> 1 不要
            }
            mask = _blsr_u32(mask);                                     // BLSR ×1
        }
        i += LANE;
    }
    None
}
```

#### 効果見積もり (per_shard=16, N_cand≈0.69, N_false≈0.06)

| 区分 | AVX2 (現状) | V1 (AVX-512 VL) | Δ |
|---|---:|---:|---:|
| Path B (1 false-match) | 2 cy (BLSR ×2) | **1 cy** (BLSR ×1) | −1 cy/false-match |
| outer test path | `vpcmpeqw → vpmovmskb → testl → je` lat ≈ 1+5+1 = 7 cy | `vpcmpeqw_mask → kortestz → je` lat ≈ 3+1 = 4 cy | **−3 cy/chunk** |
| success path | `lane = bit >> 1; pos = i + lane` 2 ops | `pos = i + lane` 1 op | −1 op/hit |

per_shard=16 (1 chunk) で:
```
−3 cy/chunk × 1 + −1 cy/false × 0.06 + −1 op/hit × 0.6 ≈ −3.3 cy/scan ≈ −0.83 ns/op @4 GHz
```

per_shard=64 (4 chunks) で:
```
−3 cy/chunk × 4 + −1 cy/false × 0.25 ≈ −12.3 cy/scan ≈ −3.1 ns/op @4 GHz
```

V1 単独で `find-avx2-pext` の P1+P2 と概ね同等の効果が、**PEXT を使わずに**得られる。
PEXT を温存した上で AVX-512 を opt-in する人にとっては P1/P2 は不要になる。

#### 必要な CPU 機能

`avx512f` + `avx512bw` (epi16 系の kmask 命令) + `avx512vl` (256-bit 上で k 命令を使える)
+ `bmi1` (BLSR 単独使用)。これは Skylake-X 以降の Intel server 全てと Zen 4+ に揃う。

### V2. 512-bit zmm 幅で chunk 数を半減

LANE = 32 (zmm) に拡張。per_shard ≤ 32 なら 1 chunk、per_shard ≤ 64 なら 2 chunks。

```rust
#[target_feature(enable = "avx512f,avx512bw,bmi1")]
unsafe fn find_avx512zmm<Q>(&self, key: &Q, needle: u16) -> Option<(usize, u16)>
{
    const LANE_512: usize = 32;
    let limit = (self.len + LANE_512 - 1) & !(LANE_512 - 1);
    // ...
    let needle_v = _mm512_set1_epi16(needle as i16);
    let mask_v   = _mm512_set1_epi16(Self::SCAN_MASK as i16);

    let mut i = 0usize;
    while i < limit {
        let v      = _mm512_load_si512(tags_ptr.add(i) as *const __m512i);
        let masked = _mm512_and_si512(v, mask_v);
        let k: __mmask32 = _mm512_cmpeq_epi16_mask(masked, needle_v);
        let mut mask = k;
        // (V1 と同形の inner loop)
        i += LANE_512;
    }
    None
}
```

#### 効果見積もり (per_shard=64, N_cand≈0.88, N_false≈0.25)

| 区分 | V1 (256-bit, 4 chunks) | V2 (512-bit, 2 chunks) | Δ |
|---|---:|---:|---:|
| outer per chunk | ~4 cy (load+and+cmp+test) × 4 = 16 | ~5 cy × 2 = 10 | −6 cy/scan |
| inner candidate | (load+and+cmp 6 cy) × 0.88 | 同左 (per scan は同じ) | 0 |
| inter-chunk branch | 3 個 × ~1 cy = 3 | 1 個 × ~1 cy = 1 | −2 cy/scan |

合計 ≈ **−8 cy/scan ≈ −2 ns/op @4 GHz**、per_shard=64 帯で。per_shard ≤ 32 なら 1 chunk
で完結 (outer ループ自体が 1 iter) で更に強い。

#### Tags 配列の alignment 要件

`vmovdqa64 zmm, [mem]` は **64-byte 揃え**を要求する (= 1 cache line)。現状の
`AlignedTags` は 32-byte 揃えなので、V2 を入れるなら `repr(C, align(64))` への変更が要る。
ただし aligned-tags-load レポートで観測した cache-line split 解消の方向性とも整合
(64-byte align は 64-byte cache line と完全一致)。

なお `vmovdqu64` (unaligned 512-bit load) でも動くが、現代の Intel/AMD 実装では aligned
load との性能差はほぼ無いと言われる (memory operand fold される文脈なら無差別) — ここは
実測で決める軸。

#### Downclock 注意点

Skylake-X / Cascade Lake で 512-bit op が連続すると AVX-512 turbo bin に落ちる。Cache
hot path 専従なら net で勝つことが多いが、application 側との相互作用で scalar 側に
回帰が出る可能性。`cargo feature avx512-zmm` (default off) で opt-in する形を推奨:

```toml
[features]
default = []
avx512-vl = []        # V1 経路 (256-bit + kmask) を有効化、downclock なし
avx512-zmm = ["avx512-vl"]  # V2 経路 (512-bit zmm) を更に有効化、downclock 注意
```

### V3. AVX-512 vpgather による candidate batched key fetch

`find-avx2-pext` の P4 (vpgather AVX2) を AVX-512 上で再評価。

inner candidate loop の Path A (`tag load → cmp+load`) は dependent load 2 段で 14 cy。
candidate が 1 個ずつ直列に走るのを **chunk 内の全 candidate を一度に gather** で並列化:

```rust
// k1 = matched lanes (kmask)
let ids   = _mm256_and_si256(v, _mm256_set1_epi16(ID_MASK));   // tag & ID_MASK = byte offset
// 16 lane の u16 を 8 lane の u32 に拡張 (low 16 / high 16)
let ids_lo = _mm256_cvtepu16_epi32(_mm256_extracti128_si256(ids, 0));   // 8x u32
let ids_hi = _mm256_cvtepu16_epi32(_mm256_extracti128_si256(ids, 1));
// マスク付き gather: 該当 lane だけ entries[id].key を読む
let keys_lo = _mm256_mmask_i32gather_epi32(zero, k1_lo, ids_lo, entries_ptr, 1);
let keys_hi = _mm256_mmask_i32gather_epi32(zero, k1_hi, ids_hi, entries_ptr, 1);
// SIMD で needle_key と全 lane 比較
let cmp_lo  = _mm256_cmpeq_epi32_mask(keys_lo, needle_key_v);
let cmp_hi  = _mm256_cmpeq_epi32_mask(keys_hi, needle_key_v);
// 一致した最初の lane が答え
let final_mask = (cmp_hi as u32) << 8 | cmp_lo as u32;
if final_mask != 0 {
    let lane = final_mask.trailing_zeros() as usize;
    return Some((i + lane, tag));
}
```

#### 適用条件

- key が `u32` (Slot16): vpgatherdd で 8 lane 同時 fetch
- key が `u64` (Slot32): vpgatherdq で 4 lane 同時 fetch (chunk を 2 分割して 2 回 gather)
- key が `String` その他可変長: 適用不能 (gather は固定 stride 前提)

#### 効果と懸念

- gather 自体の latency: Skylake-X 12 cy / Ice Lake 10 cy / Sapphire Rapids 8 cy + per-element 1 cy
- 16 lane の inner loop が **直列 14 cy × 0.88 candidate ≈ 12 cy/scan** から **gather 一発 ~14 cy** に。candidate 0〜1 個帯では現状と互角、candidate 2+ で gather 勝ち。**per_shard=64 / 高 false-match 帯で本領発揮**。
- false match 確認は SIMD 比較で 0 cy/lane。

`find-avx2-pext` の P4 を AVX-512 で書き直すと **同じ仕事を 1/2 の latency** で済む
(gather 性能向上分)。本命は B3 + V2 の合成で per_shard=64 / 高 candidate 帯を狙う。

### V4. `vpcompressw` で matched candidate を front-pack

AVX-512 VBMI2 の `_mm256_mask_compress_epi16` を使うと、kmask に従って matched lane を
レジスタの低位に詰めて並べられる。inner loop が:

```rust
let ids = _mm256_and_si256(v, _mm256_set1_epi16(ID_MASK));
let compressed: __m256i = _mm256_maskz_compress_epi16(k1, ids);
// compressed の低位 popcnt(k1) lane に matched id がそのまま並ぶ
let n_match = _mm_popcnt_u32(k1 as u32);
for j in 0..n_match {
    let off = (_mm256_extract_epi16(compressed, j) as u32 & ID_MASK as u32) as usize;
    let e   = &*(entries_byte_ptr.add(off) as *const Entry<K, V>);
    if e.key.borrow() == key { return Some(...); }
}
```

利点: tzcnt + BLSR の bit walk が popcnt + 連続抽出に置き換わる。スカラ命令数が減る方向。
ただし `_mm256_extract_epi16` は SIMD-to-GPR の path でやや高コストなので、**candidate
数が確定して loop 回数が決まる**点が主な勝ち。

#### 適用条件

`avx512vbmi2` 必要。Ice Lake (server) / Tiger Lake (client) 以降の Intel と Zen 4+。
Skylake-X / Cascade Lake は VBMI2 無し。

### V5. V2 + B1 (SoA tag split) — per_shard=64 を 1 chunk shot で

`find-avx2-frontier` の **B1 (SoA tag split: 8-bit hash 配列 + 8-bit id|visited 配列)**
は AVX-512 と最も強い相乗を示す:

- B1 単独: 32 u8 lane / ymm = chunk あたり 32 lane、per_shard=64 で 2 chunks
- **B1 + V2 (zmm)**: **64 u8 lane / zmm** = chunk 数 1 で per_shard=64 が完結

つまり per_shard=64 の `find` が **outer ループ無し / 1 zmm shot** で済む。

```rust
#[target_feature(enable = "avx512f,avx512bw,bmi1")]
unsafe fn find_avx512_b1<Q>(&self, key: &Q, needle_byte: u8) -> Option<(usize, u8)>
{
    let n_lanes = self.len.next_multiple_of(64).min(64);   // 1 chunk only (cap ≤ 64)
    let tags_scan_v = _mm512_load_si512(self.tags_scan.as_ptr() as *const __m512i);
    let needle_v    = _mm512_set1_epi8(needle_byte as i8);
    let k: __mmask64 = _mm512_cmpeq_epi8_mask(tags_scan_v, needle_v);

    // len で右側を切り捨て
    let live_mask = (1u64 << n_lanes).wrapping_sub(1);
    let mut mask  = (k as u64) & live_mask;

    while mask != 0 {
        let lane = mask.trailing_zeros() as usize;
        let id   = self.tags_meta[lane] & ID_MASK_BYTE;     // 6-bit id
        let e    = &*(self.entries_byte_ptr.add(id as usize * S::SIZE) as *const Entry<K, V>);
        if e.key.borrow() == key {
            return Some((lane, ...));
        }
        mask = _blsr_u64(mask);
    }
    None
}
```

#### 効果

per_shard=64 / N_cand≈0.88 で:

- outer: 1 chunk = `vpand なし (cmp 直接) + vpcmpeqb_mask + kortestz` ≈ 4 cy
- inner: candidate 1 個あたり `tags_meta[lane] load + entries[id].key cmp+load` ≈ 11 cy
- 直接比較: 現状 (V1) の 4 chunks × 4 cy + N_cand × 6 cy = 16 + 5.3 ≈ **21 cy/scan**
- V5: 4 + 0.88 × 11 = **13.7 cy/scan**

**−7 cy/scan ≈ −1.8 ns/op @4 GHz**。

ただし B1 由来の **false-match 倍増 (1/256 → 1/128)** は inner candidate を増やす方向で、
N_cand が 0.88 → ~1.0 に近づく。Path A 短縮効果が打ち消される可能性は要実測。

#### 必要な改造

- `tags` を `tags_scan: Vec<u8>` (LIVE | 7-bit hash) と `tags_meta: Vec<u8>` (6-bit id +
  visited) に分離
- 全 tag layout 操作 (`needle_from_hash`、tag construction in insert、VISITED bit set、
  evict 後の id rewrite、retain の bitmap remap、Drop iteration) を 2 配列対応に
- AlignedTags は scan 配列 64-byte align、meta 配列 32-byte align (=ymm load 用) など
- oracle テスト (`oracle_cache_match.rs`) を pass 必須

`find-avx2-frontier` の B1 と同じ大改修。AVX-512 を前提にするとペイオフが大きくなる、と
いう点が本稿の追加情報。

### V6. predicated tail load (`vmovdqu16_mask`)

partial chunk (last chunk で len が LANE 単位に揃わない場合) は現状 EMPTY pad で
カバーしている。AVX-512 では:

```rust
let tail_mask = (1u32 << (self.len - i)).wrapping_sub(1) as __mmask32;
let v = _mm512_maskz_loadu_epi16(tail_mask, tags_ptr.add(i));
```

で正確に `len - i` lane だけ load できる。利得は限定的 (EMPTY pad scan は確率的に
false-match を引かないので): per_shard=63 のような半端な capacity で chunk あたり
1 lane の余計な scan が消える程度。優先度低。

## 3. Downclock 懸念と二段 dispatch 設計

### 3.1 関連する Intel 文書 / 観測

- Intel Optimization Reference Manual §15 "AVX-512 Programming"
- Travis Downs "Gathering Intel on Intel AVX-512 Transitions" (lemire blog)
- Cloudflare engineering "On the dangers of Intel's frequency scaling" (2017、Skylake-X 評価)

要旨は「heavy AVX-512 (= 512-bit 幅 + heavy ALU) を hot path で連続発火すると frequency
が L1 turbo に落ちる、復帰に数百 µs」。Skylake-X / Cascade Lake で顕著、Ice Lake 以降は
緩和、Sapphire Rapids 以降はほぼ非問題。AMD Zen 4/5 はそもそも周波数フックなし。

### 3.2 dispatch 三段案

```
Tier 0 (default, portable):
  scalar fallback  →  AVX2 + BMI1 (現状の find_avx2)

Tier 1 (cargo feature `avx512-vl`, 既存 Cache::new で runtime detect):
  scalar  →  AVX2  →  AVX-512 VL (V1, 256-bit, kmask)

Tier 2 (cargo feature `avx512-zmm`, 起動時 bench で downclock cost を計測 or static):
  scalar  →  AVX2  →  AVX-512 VL  →  AVX-512 zmm (V2)
```

`avx512-vl` を default 有効にしない理由は:

- Skylake / Skylake-X 中盤以前で AVX-512 サブセット差 (`bw` / `vl` の有無) があり、
  `avx512f` だけある CPU で kmask op が落ちる可能性
- compile target に AVX-512 命令が混入することを意識的に opt-in させる方が library
  ユーザに親切

代替案: feature flag を付けず Cache::new で全部 runtime detect、対応 instruction が
揃った CPU で自動的に高速 path に乗る。bin size が増える代償あり (Tier 0/1/2 の三系統が
全部 link される)。

### 3.3 Cache 構造体の dispatch flag 拡張

```rust
pub struct Cache<K, V, S, ...> {
    // 既存
    has_avx2_bmi1: bool,
    // 新規
    has_avx512_vl: bool,
    has_avx512_zmm: bool,   // cargo feature `avx512-zmm` でのみ true 化
    // ...
}

impl<K, V, S, ...> Cache<K, V, S, ...> {
    fn detect() -> (bool, bool, bool) {
        let avx2_bmi1 = is_x86_feature_detected!("avx2") && is_x86_feature_detected!("bmi1");
        #[cfg(feature = "avx512-vl")]
        let avx512_vl = avx2_bmi1
            && is_x86_feature_detected!("avx512f")
            && is_x86_feature_detected!("avx512bw")
            && is_x86_feature_detected!("avx512vl");
        #[cfg(not(feature = "avx512-vl"))]
        let avx512_vl = false;

        #[cfg(feature = "avx512-zmm")]
        let avx512_zmm = avx512_vl;
        #[cfg(not(feature = "avx512-zmm"))]
        let avx512_zmm = false;

        (avx2_bmi1, avx512_vl, avx512_zmm)
    }
}
```

`find` 側 dispatch:

```rust
fn find<Q>(&self, key: &Q, needle: u16) -> Option<(usize, u16)> {
    #[cfg(target_arch = "x86_64")]
    {
        if self.has_avx512_zmm {
            return unsafe { self.find_avx512_zmm(key, needle) };
        }
        if self.has_avx512_vl {
            return unsafe { self.find_avx512_vl(key, needle) };
        }
        if self.has_avx2_bmi1 {
            return unsafe { self.find_avx2(key, needle) };
        }
    }
    self.find_scalar(key, needle)
}
```

3 段の if は branch predictor が瞬時に学習するので overhead 無視。

## 4. 既存提案 (Tier-S/A/B/P) との相互作用

`find-avx2-frontier` + `find-avx2-pext` 提案項目との関係:

| 項目 | AVX-512 VL (V1) | AVX-512 zmm (V2) | 備考 |
|---|---|---|---|
| S1 (find returns tag) | **直交、両方必要** | 直交 | tag を return に乗せる軸は SIMD 種別と無関係 |
| S2 (entry_ptr raw arith) | 直交 | 直交 | bounds check 撤廃も SIMD 非依存 |
| S3 (Shard field 並び替え) | 直交 | 直交 | cache-line layout は普遍 |
| S4 (Slot16 broadcast hoist) | **要再観察** | 要再観察 | 新 codegen で reg-alloc が変わる、現象が消える / 別所に出る可能性 |
| A1 (inner unroll ×2) | **依然有効** (Path A は AVX-512 でも変わらず) | 同左 | Path A の load 並列化軸は SIMD 種別と独立 |
| A2 (4-chunk specialize) | 部分有効 (4 chunks のまま) | **2 chunks specialize に縮退**、利得減 | V2 で chunk 数が半分になるため A2 の旨味が薄い |
| A3 (tag-direct entry helper) | 直交 | 直交 | |
| A4 (VISITED in find_avx2) | kmask + masked store で別実装案あり | 同左 | mask register でフィルタした上で `vpor + vmovdqu_mask` の選択肢 |
| B1 (SoA tag split) | per_shard=64 が 2 chunks | **per_shard=64 が 1 chunk shot** (= V5) | **強い相乗**、AVX-512 採用時の最有力構造変更 |
| B2 (32-bit tag) | epi32 cmp に対応 | 同左 | hash 16-bit 化で false match ほぼゼロ |
| B3 (vpgather batch) | AVX2 比で gather 高速化 (= V3) | 同左 | per_shard=64 高 candidate 帯の本命 |
| B4 (tags+entries co-locate) | 直交 | 直交 | |
| **P1 (mask 圧縮単独)** | **不要** (kmask が初めから lane-mask) | **不要** | AVX-512 が P1 を構造的に吸収 |
| **P2 (PEXT + unroll ×2)** | **PEXT 不要、unroll ×2 部分のみ A1 として残る** | 同左 | P2 は AVX-512 経路では A1 に等価 |
| P3 (PDEP needle) | **直交、温存** | 直交 | needle 構築は SIMD 種別非依存 |
| P4 (vpgather AVX2) | **superseded by V3** | 同左 | AVX-512 gather が AVX2 の上位互換 |

要点:

- **AVX-512 経路を入れると PEXT 系 (P1/P2) は要らなくなる** — kmask が PEXT で得たかった
  形を最初から提供する。P3 (PDEP needle) だけは独立で残す。
- **B1 SoA split は AVX-512 と最も相乗が強い**。逆に言うと B1 の判断は AVX-512 採否と
  bundle するのが筋。
- S1/S2/S3 は **どの SIMD path にも効く前提条件**。先に消化する順序は変わらない。

## 5. 推奨着手順

`find-avx2-pext` §6 の推奨順を AVX-512 を踏まえて update:

| 優先度 | 項目 | 期待効果 (机上) | リスク |
|---|---|---|---|
| 1 | **S1** `find -> Option<(pos, tag)>` | hit path tag reload 解消、−5 cy | 低 |
| 2 | **S2** `entry_ptr` raw arith | shift round-trip 解消、−2〜3 cy | 低 |
| 3 | **S3** Shard フィールド並び替え | cold-self で 1 line/call 削減 | ほぼゼロ |
| 4 | **P3** PDEP `needle_from_hash` | call ごと −2〜3 cy、CPUID dispatch | 低 |
| 5 | **V1** AVX-512 VL + kmask (256-bit) | per_shard=16 で −0.8 ns、64 で −3 ns | 中 (cargo feature 設計、target_feature 追加) |
| 6 | A2 4-chunk specialization | branch 削減 ≈ −3〜−6 cy | 低 (V1 後でも有効) |
| 7 | **V2** AVX-512 zmm (512-bit) | per_shard=64 で更に −2 ns | 中〜高 (downclock 注意、opt-in feature) |
| 8 | A1 inner unroll ×2 | per_shard=64 高 N_cand 帯で大 | 中 |
| 9 | **V5** B1 (SoA split) + V2 | per_shard=64 を 1 chunk shot、−7 cy/scan | 高 (大改修、false-match 倍増との相殺は要実測) |
| 10 | V3/B3 vpgather batch | per_shard=64 高 candidate 帯、要実機計測 | 高 |

V1 は **S1/S2/S3 + P3 commit 後の baseline で計測**するのが順序的に正しい。AVX-512 の
有無で codegen 全体が動くので、Tier-S 適用前に V1 を測ると Tier-S の効果と AVX-512 の
効果が混ざって判断つかない。

V2 は cargo feature 切って opt-in 配布。downclock リスク評価は実機 (Skylake-X / Sapphire
Rapids / Zen 4) でそれぞれ取りたい — 持ってる SKU 次第で実験計画が立つ。

V5 は S2/S3 と V2 が両方 commit された後、prototype 1 monomorph (Slot32/u64) で cell 数
小さく試すのが現実的。`find-avx2-pext` の P2 vs B1 比較と同じ要領で、V5 を **AVX-512
専用の B1 実装**として比較する。

## 6. 配布形態の整理

`senba` crate のリリース工程に与える影響:

### 6.1 cargo features

```toml
[features]
default = []
avx512-vl   = []                # V1 経路 (256-bit + kmask、no downclock)
avx512-zmm  = ["avx512-vl"]     # V2 経路 (512-bit zmm、要 downclock 検討)
```

ユーザは:
- `senba = "0.1"` — portable、AVX2 + BMI1 自動 detect、AVX-512 は使わない
- `senba = { version = "0.1", features = ["avx512-vl"] }` — V1 を有効化、server で全自動
  fast path
- `senba = { version = "0.1", features = ["avx512-zmm"] }` — Sapphire Rapids+ / Zen 4+
  の server で最速、Skylake-X / Cascade Lake では downclock 注意

### 6.2 perf-gate との関係

`research/benches/sieve_cache_perf.rs` は senba::Cache の安定契約。AVX-512 を入れると:

- default feature では codegen 不変 → perf-gate baseline 維持
- `--features avx512-vl` で baseline を **別系統** として save (例: `before-avx512vl`)。
  V1 の効果を見る対比は同 feature 上で AVX2 path を強制 (debug helper 追加) して取る、
  または **V1 適用前後の commit 間で比較**する。
- perf-gate 自体は portable に保つ (CI で AVX-512 検出を強制しない)。

### 6.3 ドキュメント

`README.md` / `docs/api-comparison.md` に:

- AVX-512 への opt-in は cargo feature 経由
- Skylake-X 系で `avx512-zmm` を有効にすると downclock の可能性、Sapphire Rapids 以降
  および Zen 4 以降では非問題
- AVX-512 が無い CPU では feature を有効にしてもランタイム detect で AVX2 path に
  自動 fallback、エラーや panic は出ない

の 3 点を明記。

## 7. オープン課題

| # | 課題 |
|---|---|
| OQ-V1 | V1 の `vpcmpeqw_mask` が実機で本当に latency 3 cy / 1 uop に乗るか (uarch ごとの
        port assignment 差)。Skylake-X / Sapphire Rapids / Zen 4 で `cargo asm` + perf
        counter (uops_dispatched_port) で確認。 |
| OQ-V2 | V2 の downclock 影響を定量。`find_avx512_zmm` を 1M 回回した直後に scalar
        bench を走らせて frequency の dip と回復時間を測る。Skylake-X / Cascade Lake で
        どれだけ問題か、Sapphire Rapids でほぼ消えるかの裏取り。 |
| OQ-V3 | Tags 配列を 64-byte align に上げた時に他 path (insert / evict / retain) で
        悪影響が無いか。`AlignedTags` の `align(32)` → `align(64)` 変更コスト評価。 |
| OQ-V4 | V5 (B1 + V2) の **false-match 倍増 vs chunk 数 1 化** の損益が prototype 実装で
        どう出るか。Slot32/u64 の 1 monomorph で実装して per_shard=64 / Twitter
        cluster018 で 5 trials 計測すれば決着。 |
| OQ-V5 | AVX10 (Granite Rapids 以降の Intel) との将来互換。AVX10/256 が AVX-512 VL の
        後継として位置付けられるなら V1 path がそのまま乗る、AVX10/512 で V2 が動く、
        という対応の確認。仕様確定段階で再評価。 |

## 8. まとめ

`find-avx2-frontier` Tier-C で「non-portable」と棚上げした AVX-512 は、**server 利用に
限れば実態として ubiquitous** で、`senba::Cache` を server LRU として使うユーザに恩恵を
直接届けられる。具体的な勝ち手は:

- **V1 (AVX-512 VL + kmask, 256-bit)**: kmask の 1-bit-per-lane 形式で `vpmovmskb`
  (5 cy) と BLSR pair (2 cy) を一掃、per_shard=64 で −3 ns/op。**downclock 無し**。
- **V2 (zmm 512-bit)**: chunk 数半減で更に −2 ns/op、ただし downclock 注意で opt-in。
- **V5 (V2 + B1 SoA split)**: per_shard=64 を 1 chunk shot へ縮め、AVX-512 と B1 が
  最も強く相乗。

PEXT 系 (P1/P2) は AVX-512 経路では構造的に吸収されて不要。P3 (PDEP needle) と
Tier-S (S1/S2/S3) は SIMD path 非依存で温存。

実装は cargo feature で opt-in、runtime detect で AVX-512 が無い CPU は AVX2 path に
fallback。本稿は方針提案まで、prototype と実測は別 PR。

着手順は **S1/S2/S3 + P3 で baseline を整える → V1 を入れる → A2 (chunk specialize) →
V2 と V5 の二択を prototype で詰める** が現実解。
