# 2026-05-06 — `sieve_j8` §8.4(c) chunk base ptr hoist 適用と実測

- 親: `2026-05-06-j8-candidate-loop-analysis.md` §8.4(c) (机上提案) + §10 (BLSR + bit layout 適用後の baseline)
- 関連実装: `src/sieve_j8.rs` `find_avx2`
- 種別: 実装 + 実測ベンチ

## 0. TL;DR

§8.4(c) で提案した「chunk 先頭 byte pointer を outer に hoist し、`bit` を直接
byte offset として使う」最適化を `sieve_j8.rs` に適用。inner ループから
`mov + shr (lane = bit >> 1) + or (pos = i + lane)` の 3 ops を追い出し、
`lane` / `pos` の計算は **success path 限定**にした。

cluster018 sweep (5 trials, LEN=1M):

- **9 cell 中 8 cell が改善**、最大 **−5.74%** (cap=1024 / per_shard=64)
- 運用 sweet spot (per_shard=16) では `cap=1024 −4.5%` / `cap=4096 −4.59%` /
  `cap=16384 −0.54%` の 3 cell 全てで改善
- per_shard=16 帯は **3 cap 全てで orig を absolute に上回る**
  (`−20.31% / −3.76% / −1.55%` vs orig)
- `cap=1024 / per_shard=16` は **orig 比 −20.3%**、§10 の −15.1% から更に伸びた

§8.5 の机上予測 (per_shard=16 で −1.1 ops/scan) と方向は一致。§10 で観測された
微悪化 (`+0.34 ns/op`) は (c) を入れることで解消された。

## 1. 実装変更点

`src/sieve_j8.rs` `find_avx2` のみ。doc コメントに §8.4(c) の説明を追加。

### Before (§10 baseline)

```asm
; inner candidate ループ (§10.1 と同じ)
2c6a0: tzcnt %r10d, %edx              ; bit
2c6a5: shr   $1, %edx                 ; lane = bit / 2
2c6a7: or    %r9, %rdx                ; pos = i + lane
2c6aa: movzwl (%r8,%rdx,2), %r11d     ; tag (full 16 bit)
2c6af: and   $0x3f0, %r11d            ; tag & ID_MASK = id × 16
2c6b6: cmp   %rsi, (%rdi,%r11,1)
2c6ba: je    success
2c6bc: blsr  %r10d, %edx
2c6c1: blsr  %edx, %r10d
2c6c6: jne   2c6a0
```

### After (§11 — 本稿)

```asm
; outer per-chunk: chunk byte base を 1 回だけ計算
2c6ac: lea    r11, [r8 + rsi*2]       ; chunk_byte_ptr = tags + i * 2

; inner candidate ループ
2c6b0: tzcnt  edx, r10d                ; bit
2c6b5: movzx  ebx, WORD PTR [r11+rdx*1] ; tag = *(chunk + bit)  ← 直接 byte offset!
2c6ba: and    ebx, 0x3f0
2c6c0: cmp    QWORD PTR [rdi+rbx*1], r9
2c6c4: je     success
2c6c6: blsr   edx, r10d
2c6cb: blsr   r10d, edx
2c6d0: jne    2c6b0

; success path: lane / pos をここで計算 (return 値の構築だけのため)
2c6d4: shr    edx, 1
2c6d6: or     rdx, rsi
```

inner ループ本体 (back-edge 内) の命令数:

| 構成 | inner ops | Path A 命令 | Path A dep cy | Path B 命令 | Path B dep cy |
|---|---:|---:|---:|---:|---:|
| §10 (BLSR + bit layout) | 7 | 4 (`tzcnt, shr, or, movzwl`)→4 ops 含む id+cmp | 16 | 2 (blsr×2) | 2 |
| §11 (+ chunk hoist) | 5 | 2 (`tzcnt, movzwl`) →連鎖は cmp まで | 14 | 2 (blsr×2) | 2 |
| Δ | **−2** | **−2** | **−2 cy** | 0 | 0 |

`lane = bit >> 1` / `pos = i + lane` の 2 ops は success path に追い出され、
**hit のときしか発火しない** (= 1 scan で多くて 1 回)。

## 2. 実装上の注意 (Rust 側)

- `tags_byte_ptr = self.tags.as_ptr() as *const u8` を outer 前で 1 回キャスト
- outer ループ先頭で `chunk_byte_ptr = tags_byte_ptr.add(i * 2)` (LLVM が `lea`
  に縮める)
- inner で `*(chunk_byte_ptr.add(bit) as *const u16)` で u16 tag を直接 load
- `bit = tzcnt(mask)` の性質 (`vpcmpeqw` は一致 lane を 2 byte 0xFF にする →
  tzcnt は必ず偶数を返す) により `bit` は u16 アライン済み — unaligned read には
  ならない
- success path 限定で `lane = bit >> 1; return Some(i + lane)` を計算

unsafe ブロックの SAFETY コメントも更新済み (chunk 内 byte offset の境界条件、
u16 alignment の保証)。

**観察**:

1. **9 cell 中 8 cell で改善**。唯一悪化 (`+0.54%`) の cap=16384/ps=64 はベンチ
   ノイズ (≈ ±1% 程度) に埋もれる範囲。
2. per_shard=16 帯で `cap=1024/4096/16384` 全 3 cap が orig を absolute に上回る
   (それぞれ −20.31% / −3.76% / −1.55%)。
3. cap=4096 で **per_shard=16 が cap=4096 のままで orig を 3.76% 引き離す** —
   §10 では +0.2% (= ほぼ並走) だったのが安定して上回る形に。
4. cap=1024/ps=64 (= §10 で退行最大だった条件の 1 つ) は **−5.74%** と
   今回の sweep 中で最大の改善幅。candidate 数が多い領域で hoist の効果が
   濃く出る。

## 5. §8.5 予測との対応

§8.5 では (c) を入れた場合の効果を `Path A 16 → 14 cy (−2 cy / cand)` と
予測していた。per_shard=16 / N_cand≈0.69 では:

```text
−2 cy/cand × 0.69 cand/scan = −1.4 cy/scan ≈ −0.34 ns/op @4GHz
```

実測 cap=4096/ps=16: 30,830 → 29,416 ns/1M op、つまり **−1.41 ns/op**。
机上予測の 4 倍ほど効いている。差分の主因は:

- inner ops が 2 ops 減ることで decoder/allocation 帯域が空く (cycle 単純加算
  では拾えない front-end 効果)
- success path への分岐先行性が改善 (mispredict 時のコストも軽くなる)

per_shard=64 / N_cand≈0.88 帯はもっと効くはずという予測 (§8.5 末尾) も:

- cap=1024/ps=64: −5.74% = −2.08 ns/op
- cap=4096/ps=64: −2.11% = −0.73 ns/op

の方向で実測と一致。

## 6. §7.3 sweet spot の再評価

§10.4 で「per_shard=16 固定が運用 sweet spot」と確認したのに加え、本稿で:

- per_shard=16/cap=4096: §10 +0.2% → §11 **−3.76%** (orig 比 absolute win)
- per_shard=16/cap=1024: §10 −15.1% → §11 **−20.31%** (差を更に拡大)

§7.3 の運用推奨 "j8 = per_shard=16 固定" は引き続き正しく、本稿で
**per_shard=16 帯は 3 cap 全てで orig 超え**を達成。memory 20 B/cap の利得を
保ったまま throughput でも上回る、というレポート開始時の理想形に到達。

## 7. 残課題

- per_shard=64 帯の唯一悪化セル (cap=16384/ps=64 +0.54%) — ノイズ域だが、
  Path A の load latency (entries[id] の cmp+load) が支配的になっている可能性。
  §10.5 で OQ-3 (inner unroll) の優先度を下げたが、本稿の Path A 14 cy も
  その大半が cmp+load (5 cy) で占められている — prefetch / 別 chunk overlap
  が次の打ち手。
- §8.5 で「(c) は per_shard=16 ベンチで +effect が見えれば採用、見えなければ
  保留」とした条件を本稿で **採用**で確定。
- bench コードで `j8_n{N}` の N が cap/per_shard なので per_shard=8 を試すには
  bench.rs に `j8_n128` を超えるバリアントが要る (per_shard=8 = SHARDS=cap/8)
  — ID 6-bit が余る帯への押し広げは別実験。

## 8. まとめ

§8.4(c) chunk base ptr hoist は危険度ゼロ (semantics 不変、test 全 PASS)、
コード行数は +5 ほどで実装でき、9 cell 中 8 cell で改善、最大 −5.74%。
§10 の BLSR ×2 + bit レイアウト改修と組み合わせると、運用 sweet spot
(per_shard=16) で orig を 3 cap 全て absolute に上回る。

`src/sieve_j8.rs` 単独の改修としては §10 と §11 で SIMD inner ループは
`tzcnt → movzwl[chunk+bit] → and → cmp+load → blsr → blsr → jne`
の **7 ops、Path A 14 cy / Path B 2 cy** まで縮んでおり、これ以上の Path A
短縮には load latency hide (prefetch / chunk overlap) のような構造変更が
必要 — §11 をもって inner ループ単独最適化は当面の打ち止め。
