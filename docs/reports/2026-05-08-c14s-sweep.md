# 2026-05-08 c14s sweep — 設計書 §2.3 のバグ修正後

## TL;DR

c14s (= c13s + AVX2 find_lockfree + Path A `MAX_RETRY=1` + reader bounded retry)
の single-shard sweep。設計書 §2.3 の reader bounded retry に **真因が
"true-miss と false-miss を区別する手段がない" という前提誤り** に基づく構造的
バグがあり、初版 c14s は uniform read-only 16T で c11s 比 -75% の壊滅的 regression
を出した。バグ修正後の sweep がこの稿の本論。

修正内容: `find_get` を `(Option<V>, racing: bool)` 返却に変え、`try_candidate`
の seqlock failure と "scan 中に LIVE=0 lane を観測" の二点を racing シグナルとして
caller に伝達。 `get_by_hash` は **racing が立った時のみ** 最大 4 回まで retry し、
立っていなければ true-miss 確定として 1 回 scan で None を返す。設計書 §2.3 の
"4× 無条件 retry" が出していた miss-path のスループット 1/4 化を解消した。

§4 採否判定 (16T):

- **T1 uniform read-heavy ±5% (vs c11s)** — c14s/c11s = **0.323 ❌ fail**。これは
  c14s の責任ではなく Path B/C (writer Mutex) の構造的競合。c13s も同様 (0.345)、
  c11s 自身が Mutex を持つ baseline 同士でも uniform write 16T では Mutex が
  bottleneck になる。c14s の retry バグ修正だけでは解消できない。
- **T2 zipf-1.0 read-heavy ≥ c13s × 0.9** — **0.935 ✅**
- **T3 zipf-1.2 read-heavy ≥ c13s × 0.9** — **0.979 ✅**
- **T4 adversarial-hot HR ≥ 0.85** — **HR 0.920 ✅** (c13s は 0.571、これが c14s の
  存在意義の中核)

採否判定: T1 を fail のまま採用するか別変種に切り出すかは保留。**c13s との比較
で c14s が確実に勝っているのは "adversarial-hot HR の構造的修復"** であり、
スループットでは概ね c13s 同等 (skewed read-heavy で −1〜−7%、adv-hot 16T で −22%)
の trade-off になっている。

## §1 修正前: 設計書 §2.3 のバグの観測と原因同定

### 1.1 初版 sweep の数字

修正前の c14s で sweep を回した時の §4 acceptance:

| Target | 値 | 判定 |
|---|---|---|
| T1 uniform read-heavy 16T (c11s ±5%) | c14s/c11s = 0.245 | ❌ |
| T2 zipf-1.0 read-heavy 16T (≥ c13s ×0.9) | 0.965 | ✅ |
| T3 zipf-1.2 read-heavy 16T (≥ c13s ×0.9) | 0.912 | ✅ |
| T4 adversarial-hot HR ≥ 0.85 | 0.938 | ✅ |

T1 を満たさないどころか、read-only 系すべてで c11s/c13s 比 0.15〜0.62 の深刻な
regression が出ていた:

| read-only 16T | HR | c14s/c13s |
|---|---|---|
| adversarial-hot | 1.000 | 0.892 |
| zipf-1.2 | 0.659 | 0.640 |
| zipf-1.0 | 0.377 | 0.492 |
| zipf-0.7 | 0.074 | 0.416 |
| uniform | 0.001 | **0.145** |

HR が下がるほど c14s/c13s が線形に下がる、という極めて目立つパターン。

### 1.2 samply で同定したホットスポット

read-only uniform 16T (= 全 miss) を samply (4 kHz × 2.2 s × 16 thread, ~130k 検体)
で記録、addr2line で逆引きしたところ:

```
0x31bf5  61.7% self  Shard::get_by_hash → range::next  (sieve_c14s.rs:389)
0x31c93   3.8% self  find_get_avx2                   (sieve_c14s.rs:310)
0x31bf8   3.0% self  get_by_hash → cmp                (sieve_c14s.rs:389)
0x12728d  2.7% self  (libc / spin_loop 関連)
0x863c7   1.6% self  xxh3_stateful_update
...
```

leaf の 62% が `get_by_hash` の **`for _ in 0..MAX_READER_RETRY` ループの iterator
advance** に集中。`find_get_avx2` 本体 (内側 SIMD scan) は 4〜8% にしか出ていない。
つまり全実行時間の大半は、`find_get` を呼び出すための外側ループにあった。

### 1.3 真因

`get_by_hash` の旧実装:

```rust
for _ in 0..MAX_READER_RETRY {  // = 4
    if let Some(v) = self.find_get(key, needle) { return Some(v); }
    hint::spin_loop();
}
None
```

設計書 §2.3 は "false-miss と true-miss を区別する手段がないため上限を設ける" と
している。ところがこれは **true-miss でも 4 回 scan + 4 回 spin_loop が走る** こと
を意味する。read-only uniform は warmup 後 hit ratio ≈ 0 のワークロードで、全 query
が 4 回 scan を踏む経路に流れていた。1 回当たりの scan は SIMD で速くなっても、
4× 倍の壁が乗る。

§4.4 の予測 ("read-only / gim ではほぼ 1 回目で hit or true-miss が確定し retry は
ほぼ空振りしない") は false-miss 起因の retry には正しいが、 **true-miss 起因の
retry は救えていない**。コードは "racing を観測したから retry" ではなく "結果が
None だから retry" になっていた。

### 1.4 racing と true-miss を区別する仕組みは存在する

- **try_candidate の seqlock failure** (`t1 != t2`): ある slot で tag が一致したが
  K::eq 直前に再 load した tag が変化した場合 — Path A の cycle を踏んだ徴候。
- **scan 中の LIVE=0 lane**: `[0, len)` の範囲に LIVE=0 が見えれば Path A が
  CAS-EMPTY 状態 (cycle の中途) にあるか、Path C/B が稼働中の徴候。

どちらも reader scan のついでに観測でき、両方とも観測されない場合は **scan は
settled な tag 列を見た = retry しても結果は変わらない** と保証される。

## §2 修正

### 2.1 `Probe<V>` の導入

`try_candidate` の戻り値を `Option<V>` から enum `Probe<V> { Found(V), Miss, Racing }`
に変えた。Miss は "tag が一致しないか、一致したが key が異なる" の settled な不一致、
Racing は "tag が一致したが seqlock validate に落ちた"。

### 2.2 `find_get` の戻り値拡張

`find_get` / `find_get_scalar` / `find_get_avx2` を `(Option<V>, bool)` 返却に変更。
2 つ目の bool は "scan 中に何らかの racing を観測したか"。

AVX2 path は 1 chunk あたり追加で `_mm256_cmpeq_epi16(masked, zero_v)` + movemask
+ `[0, len)` への trim を行い、live 範囲内の LIVE=0 lane を racing として拾う。

### 2.3 `get_by_hash` を racing 条件付き retry に変更

```rust
pub fn get_by_hash(&self, key: &K, hash: u64) -> Option<V> {
    const MAX_READER_RETRY: usize = 4;
    let needle = Self::needle_from_hash(hash);
    let (v, mut racing) = self.find_get(key, needle);
    if let Some(v) = v { return Some(v); }
    if !racing { return None; }
    for _ in 1..MAX_READER_RETRY {
        hint::spin_loop();
        let (v, r) = self.find_get(key, needle);
        if let Some(v) = v { return Some(v); }
        racing = r;
        if !racing { return None; }
    }
    None
}
```

settled な true-miss は 1 回 scan で確定し、retry を踏まない。

## §4 残課題と評価

### 4.1 T1 (uniform read-heavy 16T) の解釈

c14s/c11s = 0.323、c13s/c11s = 0.345 で **c11s も c13s も c14s も同じスケール感で
落ちている**。原因は c14s 固有の問題ではなく、uniform write 16T で writer 同士が
Path B/C の Mutex を奪い合う構造的な競合。c14s の Path A は uniform write には
ほぼ効かない (新規 key 同士は find_lockfree が miss するため Path B/C に escalate
する) ので c11s の制約をそのまま継承する。

設計書 §4.1 の uniform regression 解消の見立ては、Path A 自身の retry 空振りまで
は解消したが、**writer-vs-writer Mutex 競合は c14s スコープ外**。これを救うには
c15 (entry-level seqlock) または shard 数を増やす方向 (= per-shard sub-sharding)
の構造変更が必要で、c14s の責任ではない。

### 4.2 read-only adv-hot 16T の -22%

c14s/c13s = 0.783 で **修正後の唯一の意外な退行**。HR=1 のシナリオなので racing は
本来観測されず、retry の有無は数値に効かないはず。残る差分は新規 SIMD オーバヘッド
(`_mm256_cmpeq_epi16(masked, zero_v)` + movemask + `live_lanes` mask 計算が chunk
ごとに固定で乗る) で、490 Mops/16T = 31 Mops/thread = ~32 ns/op の per-op コスト
にこの数命令が見えている形。EMPTY-lane 検出を AVX2 path から外し scalar fallback
に限定する、あるいは検出を粗く済ませる (e.g. 全 chunk の OR で代用) ことで取り戻せる
余地がある。本稿ではこれを後続課題として残す。

### 4.3 c13s との位置づけ

c13s は uniform read-heavy で c11s 比 -67% / adv-hot HR -0.26 の 2 点で不採用、
c14s は前者を解消できず (構造原因)、後者を完治させた。c14s は **c13s を adv-hot
で完全に置き換える** 価値があり、skewed read-heavy では c13s 同等。research
artifact としては両方残す価値があるが、**lock-free Path A の代表格は c14s に更新**
してよい。

### 4.4 design doc §2.3 の erratum

設計書 §2.3 の "false-miss と true-miss を区別する手段がない" は誤り。**try_candidate
の seqlock failure と LIVE=0 lane の観測** の 2 点で区別できる。設計書時点では
"4× 一律 retry" を許容していたが、実装では racing 観測時のみ retry する形に変更。
レポート §2 が公式の修正版実装。設計書側にも postscript 入れて erratum を残した。

## §5 評価と今後

### 5.1 c14s の採否

- **採用条件**: SIEVE state machine 等価性 + c13s 同等以上の skewed 性能 + adv-hot
  HR の structural fix が必要 → 採用。
- **不採用条件**: T1 (uniform read-heavy) を必須要件にする → 不採用、c15 または
  per-shard sub-shard を待つ。

c14s の存在意義は **adv-hot HR の structural fix**。c13s の "Path A だけ lock-free"
の方向性は正しく、reader 側の seqlock 検出が貧弱だった点だけが c13s の問題。c14s は
そこを racing シグナル + bounded retry で塞いだ。今後 lock-free Path A を継承する
変種は c14s を base にすべき。

### 5.2 残作業

1. **read-only adv-hot 16T の -22%**: AVX2 EMPTY-lane 検出のオーバヘッド削減。
2. **uniform read-heavy 16T の T1 fail**: c14s では救えない (Path B/C Mutex 競合)。
   c15 (entry-level seqlock 案、設計書 §7) または per-shard sub-sharding に持ち越し。
3. **適切な workload での Twitter trace cross-check**: 単一 shard sweep だけでは
   実 trace との挙動差が見えにくい。 c14s vs c11s の Twitter trace cross-check を
   別途実施してから library 採用判定する。

c14s は `senba_research::experimental::sieve_c14s` に永続化、`docs/reports/data/
2026-05-08-c14s-sweep.csv` に sweep 結果を保存。
